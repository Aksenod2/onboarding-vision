# Проект: Сбер — Онбординг юрлиц (1DAY Initiative) · прототип

Этот репозиторий — кликабельный прототип общего вижна онбординга юрлиц (открытие расчётного счёта, рынок **Индия**) на mock-данных. Боевой React-фронт, который позже забирают разработчики. Вынесен из Obsidian-vault `Хранилище/` 2026-06-01: спеки + код в одном месте.

## Кто работает

- **Денис Бугров** — продуктовый дизайнер, лидирует проект. ADHD — отвечать тезисно, TL;DR в начале, один шаг → пауза → следующий, не вываливать длинные простыни (детали ниже).
- **Claude** — агент-архитектор фундамента. На фазе наполнения экранов подключаются роли-субагенты (BA / SA / дизайн / прототайпер) + verifier.

## Что за процесс

Перевод корпоративного онбординга с ручного (5–7 дней) на цифровой (1 рабочий день): автозаполнение из реестров, авто-скрининг (санкции/KYC/кредит), видеоидентификация (VCIP). Где автоматика не проходит — гибрид/офлайн с ручной проверкой (DVU). Контекст индийский: PAN, CKYC, CRILC, Probe42, суммы в Cr (крорах).

## Ключевые архитектурные решения (не пересматривать без обсуждения с Денисом)

- **Две роли = две UI-библиотеки (асимметрия):**
  - **Клиент** (заполняет онбординг) → **SDDS Serv** (`@salutejs/sdds-serv`) — боевой опенсорс, MIT, React+TS. Палитра ~73 компонента.
  - **RM** (менеджер банка) → целевая либа **Caldera** (внутренняя СберДС). Денис подтвердил 2026-06-25: **Caldera построена на чистом `@mui/material@5` + оранжевая тема**. К SDDS отношения не имеет.
  - Общий только инженерный приём: React + слой компонентов + темизация токенами. Визуально это две разные системы.
  - ✅ **Решение Дениса 2026-06-25:** новый **CRM/RM делаем сразу на MUI (`@mui/material@5`)** — это целевой стек Caldera, код не выкидной. Каркас поднят: `app/src/rm/` (`theme.ts` оранжевый `#F5811F` из `ui/rmTheme.ts`, `RmThemeProvider` через `StyledEngineProvider injectFirst`), ветка `/rm/*` в `App.tsx` обёрнута в провайдер; emotion (MUI) и styled-components 5.3.1 (SDDS) изолированы по роутам. Smoke: `/rm/crm`.
  - ⚠️ **Устарело (было 2026-06-01):** пилот RM на «SDDS Serv с оранжевым акцентом» — временная мера, чтобы не блокироваться на Caldera. Блокер снят. Старые 5 экранов `screens/RM0*.tsx` (SDDS-оранжевый) пока остаются, мигрируют на MUI позже. На стыке ролей — транзитный экран (TR-01).
- **RM — одной ролью на старте**, позже под-роли (юр.проверка, KYC, операционка) через RBAC поверх единого Caldera-интерфейса. DVU-функции — под-роли внутри RM.
- **Прототип боевой, не выкидной.** Креативность ограничена палитрой: компонент вне whitelist → ошибка, а не «творчество».
- **Мультиагентка — на фазе наполнения, не фундамента.** Скелет (карта экранов, mock-модель, контракт компонентов) собираем вручную, один владелец.
- **Стек `app/`:** Vite + React + TS + styled-components **5.3.1** (именно эта версия — требование SDDS) + тема `sdds_serv__light` + CDN-шрифты SB Sans.

## SDDS Serv через MCP

MCP-сервер `sdds-serv` подключён в `.mcp.json`. Инструменты: `list_components`, `get_component`, `get_component_props`, `get_component_examples`, `get_tokens`, `get_installation_guide`, `get_form_guide`, `get_design_system_configuration`. **Перед использованием компонента — свериться с MCP** (боевые имена и props), а не угадывать. Боевые имена: `Input`→**TextField**, `PhoneInput`→**TextField+Mask**, `OTP`→**CodeField**, плюс **Button**, **Notification**, **Select**, **Steps** и др.

## Provenance (трассировка источников)

Схемы аналитиков обновляются. Каждый экран/сущность знает первоисточник: колонка `source` = `схема-версия : node-id`. Обновление схемы → diff-агент по node-id → пересобираем только затронутое. Старые версии схем не перезатираем.
