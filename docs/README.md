## TL;DR

Проект — перевод корпоративного онбординга (открытие расчётного счёта для юрлиц) с ручного процесса на **полностью цифровой**. Цель: сократить срок открытия счёта (TAT) с **5–7 дней до 1 рабочего дня** за счёт автозаполнения данных из реестров, авто-скрининга (санкции, KYC, кредитная нагрузка) и видеоидентификации (VCIP). Где автоматика не проходит — гибридный или офлайн-сценарий с ручной проверкой (DVU).

## Прототип — вижн и процесс работы

> Денису отдан проект: **залидировать как продуктовый дизайнер и собрать кликабельный прототип общего вижна** по всему процессу онбординга на mock-данных. В перспективе — боевой фронт на React, который забирают разработчики.

### Два интерфейса = две роли = две библиотеки (асимметрия!)

| Роль | Кто | UI-библиотека | Откуда берём код |
|---|---|---|---|
| **Клиент** (проспект → клиент банка) | заполняет онбординг | **SDDS Serv** (Salute Design System, B2B) | **боевой опенсорс** (npm `@salutejs/sdds-serv`) |
| **RM** (Relationship Manager) | менеджер банка, бэк-офис | **Caldera** (внутренняя СберДС) | **кода нет → свой набор-имитация** по образцу Caldera |

**Две части идут разными путями:**
- **Клиент** — на **боевом SDDS** сразу. Палитра готова (81 компонент), визуал диктует дизайн-система.
- **RM** — на **самодельном наборе под Caldera**. К SDDS отношения не имеет. Воссоздаём компоненты по принципу, как устроена Caldera.
- **Общий только инженерный приём** (не визуал): обе части — React + слой компонентов + темизация токенами (так устроен и сам SDDS/Plasma). Визуально это две разные системы.

- **RM ведём как одну роль на старте.** Позже декомпозируем на под-роли (юридическая проверка, KYC, операционка) через слой доступов (RBAC) поверх единого Caldera-интерфейса — не плодим интерфейсы, плодим права на функции.
- **DVU-функции** (ручная верификация) = под-роли внутри RM/Caldera.

### Архитектурные решения

- **Прототип боевой, не выкидной.** React. Клиентская часть — прямо на SDDS-компонентах. RM-часть — на своём наборе-имитации Caldera с тем же инженерным подходом.
- **Креативность ограничена палитрой.** Клиент: whitelist = 81 компонент SDDS (задокументированы, прототайпер не вылезет). RM: whitelist = наш Caldera-набор. Компонент вне палитры → ошибка, а не «творчество».
- **Мультиагентка — на фазе наполнения, не фундамента.** Скелет (карта экранов, модель данных, контракт компонентов) собираем вручную, один владелец. Роли-субагенты (BA / SA / дизайн / прототайпер) включаются на параллельном наполнении экранов по готовому скелету. Verifier-агент сверяет результат с карточкой экрана.

### Ресурсы: SDDS (Salute Design System)

- **Что это:** опенсорс-дизайн-система Сбера (SberDevices) поверх ядра **Plasma**. **React + TypeScript, лицензия MIT** — можно использовать свободно.
- **npm:** `@salutejs/sdds-serv` (B2B-сервисы) · `@salutejs/sdds-cs` (клиентский сервис) · `@salutejs/sdds-themes` (токены/темы). Движок стилей: styled-components v5 **или** Emotion.
- **Доки / Storybook:** [plasma.sberdevices.ru](https://plasma.sberdevices.ru/) · **GitHub:** [salute-developers/plasma](https://github.com/salute-developers/plasma)
- **Состав:** ~81 компонент (TextField, Select, Combobox, Steps, Table, Modal, Drawer, DatePicker, Tabs, Toast…).
- **Темизация:** одно ядро `plasma-new-hope` + темы через CSS-токены (`sdds_serv__light/dark`). Единый API компонентов, подменяется визуал — удобно для прототипа.
- ✅ **Решено:** под клиента-юрлицо берём **`@salutejs/sdds-serv`** (B2B-сервисы) — онбординг юрлица это корпоративный B2B-сценарий.
- **Caldera** — внутренняя СберДС, **в открытом доступе кода нет**; воссоздаём свой набор по её принципам.

### Артефакты-фундамент (порядок сборки)

1. **Карта экранов** — реестр экранов на каждую роль, в три среза: процесс / данные / экран. ← *следующий шаг*
2. **Mock-модель данных** — единые сущности (prospect, company, signatory, document, KYC-result…).
3. **Контракт компонентов** — UI-примитивы и их маппинг на SDDS / Caldera.

### Кто ведёт

Фундамент — **Денис (продакт) + агент-архитектор (Claude)**, без параллельных рук. Наполнение — роли-субагенты под готовый скелет.

## Работа с источниками: версионирование, provenance, обновление

> Схемы аналитиков **будут обновляться** (новые версии drawio из Confluence). Чтобы обновление не превращалось в переделку всего — разделяем «источник» и «производное» и связываем их трассировкой.

### Принцип

- **Источник** (схемы, фото) — приходит извне, меняется. Лежит в `_исходники/`.
- **Производное** (карта экранов, модель данных, прототип) — порождаем из источника.
- Связь **прослеживаемая**: каждый экран/сущность знает, из какого узла какой схемы он родился.

### Версионирование схем

- Новые схемы кладём в `_исходники/схемы/` как `ГГГГ-ММ-ДД_<имя>.drawio.xml`.
- **Старые версии не перезатираем** — храним историю.
- Текущая базовая схема: `_исходники/схемы/2026-06-01_obo_full_process.drawio.xml`.

### Provenance (трассировка)

- У узлов drawio есть стабильные `id` (`mxCell id="…"`). При **редактировании** того же файла id сохраняются → надёжный diff по версиям.
- В карте экранов — обязательная колонка **`source`** = `схема-версия : node-id`. Экран всегда знает первоисточник.
- ⚠️ **Договорённость с аналитиками:** редактировать существующий файл схемы, **не пересоздавать с нуля** — иначе id меняются и трассировка грубеет (diff пойдёт по подписям блоков).

### Процедура обновления (update pipeline)

Пришла новая версия схемы →
1. **Diff-агент** сравнивает старую/новую по node-id → changelog: `добавлено / удалено / изменено`.
2. По колонке `source` находим затронутые экраны и сущности.
3. Точечно пересобираем **только затронутое** теми же ролями-агентами. Остальное не трогаем.

Вход поменялся → прогнали дельту → обновили затронутое. Никакого «переписать всё».

## Ключевые сущности

- **DVU** (Data Verification Unit) — подразделение ручной верификации; на него уходят все исключения.
- **CRM / RM** — менеджер по работе с клиентом, ведёт офлайн-кейсы.
- **VCIP** (Video Customer Identification Process) — видеоидентификация подписантов.
- **STP** (Straight Through Processing) — сквозная автоматическая обработка без ручного вмешательства.
- **CKYC** — Central KYC Registry (центральный реестр KYC).
- **BOV** — Business Object Verification / физическая проверка бизнеса.
- **PAN** — Permanent Account Number (индийский налоговый идентификатор).

> ⚠️ **Контекст: это индийский банковский онбординг** (PAN, NSDL, CKYC, CRILC, Probe42, суммы в Cr — крорах). Документ — спецификация процесса, вероятно для платформы Сбера на индийском рынке (ICBSBASE).
