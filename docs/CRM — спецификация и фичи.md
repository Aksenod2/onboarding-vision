## 0. Замысел

**CRM одного окна** для клиентского менеджера и операторов колл-центра. Материнская сущность — **Company Profile** — агрегатор, вокруг которого: воронка (Lead→Offer→Deal→Application), коммуникации/история, AI-сводка, таски. Заменяет разрозненность Caldera (Customers / Online Onboarding / Sales). Зеркалит клиентский онбординг («зелёную зону»).

## 2. Акторы

| Актор | Роль |
|---|---|
| Клиентский менеджер | одно окно; ведёт профиль; создаёт лид/оффер/сделку/звонок/встречу/ссылку; таски; трекает воронку |
| Оператор колл-центра | входящий звонок; поиск по PAN; история/AI summary; заводит профиль (телефон/Probe42) или направляет в self-reg; возобновление брошенной заявки |
| DVU (под-роль) | [OQ-9, не подтверждён] предположительно работа с брошенными/офлайн-заявками |

## 3. Сущности и атрибуты

- **CompanyProfile:** entityType, legalName, **pan (обяз.)**, cin?, registeredAddress, source (call/potential/self-registration/onboarding), подписанты, продукты, aiSummary?, createdAt.
- **Lead:** product, createdAt, note?
- **Offer:** **product (обяз.)**, expectedProfit?, commission?, source, tags[], description?, onboardingLink? (ссылка рождается из оффера — Хафизова).
- **Deal:** kind (deposit/current-account/credit…), channel (online/offline), stage, createdAt.
- **Application:** type (online/offline-mode), stage, status (active/abandoned/rejected), source (self-reg/онлайн-банк), updatedAt.

## 4. Фич-лист (24 фичи, 10 групп)

**A. Company Profile:** материнская сущность-агрегатор (A1); primary-атрибуты для ЕПК (A2); передача в ЕПК/ПАУС (A3, [OQ-7]); унифицированная карточка + дропдаун «откуда» (A4); сводка подписанты/продукты/цифровой след (A5); готовые продукты клиента (A6).
**B. Поиск/одно окно:** одно окно вместо прыжков по разделам (B1); **поиск по PAN** (B2); список клиентов + столбец Application (B3).
**C. Воронка:** лиды по продукту (C1); создать оффер из профиля (C2); создать сделку из оффера (C3); статусы/стадии всех сущностей (C4); Application online/offline-mode (C5); Application из 2 источников (C6); возобновление брошенной заявки (C7).
**D. Коммуникации:** история коммуникаций (D1); цифровой след/точки касания (D2); отказы в профиле (D3).
**E. AI Summary:** рекомендательное (E1); сигналы ВЭД + задолженность (E2).
**F. Таски/календарь:** таски менеджеру (F1); создать встречу/звонок/ссылку (F2); таски+календарь «аля Битрикс» (F3).
**G. Создание профиля:** по телефону (G1); авто из Probe42 (G2); подсказка self-reg (G3); дропдаун источника (G4).
**H. Зеркальность:** зелёная зона→CRM (H1); авто-оффер по завершении онбординга (H2); сигнал проверки→оффер (H3); авто-profile при self-reg (H4).
**I. Ссылка-из-оффера:** ссылка на онбординг привязана к офферу, не к профилю (I1).
**J. Sales (НЕ трогаем):** Sales-воронка живёт своей логикой (J1); статусы Sales линкуем, не строим (J2).

## 5. Юзкейсы

- **UC-1.** Входящий звонок, клиент найден по PAN → история (лиды/оффер/брошенная заявка) + AI summary → предложить возобновить/оффер.
- **UC-2.** Входящий звонок, НЕ найден → (а) направить в self-reg; (б) завести профиль → Probe42 авто-заполнение → дропдаун источника.
- **UC-3.** Оффер рождается авто по завершении онбординга (current account по умолчанию / другой при ограничениях) → сигнал в CRM.
- **UC-4.** Брошенная/отказная заявка → видна стадия/тип/коммуникации → таск «завис» → возобновить; отказы в истории.
- **UC-5.** (произв.) Аппликейшн от существующего клиента из онлайн-банка → новый application к тому же профилю.
- **UC-6.** (произв.) Self-registration без оператора → авто-profile + авто-оффер.

## 6. Архитектура изоляции (микросервис)

Модуль `app/src/crm/` — самодостаточный: `types/` (domain, events, **локальные** primitives PAN/CIN/Address), `mock/` (seed, crmApi, reset), `adapter/` (ingest + fixtures), `components/`, `screens/`, единый barrel `index.ts` (export `CrmRoutes`). Монтируется под `/rm/crm/*` внутри `RmThemeProvider` (тема MUI общая с RM).

**Границы импорта (двусторонний барьер):**
- `crm/**` НЕ импортит `screens/v2*`, `company*`, `mock/v2*`, `ui/v2*`, онбординг-контексты. Разрешено: `@mui/*`, `react`, `react-router-dom`, `../rm/{theme,RmThemeProvider}`.
- Снаружи в CRM — только `crm/index.ts` (barrel), прямые заходы в `crm/mock|screens|types|adapter` запрещены.
- Гард: `scripts/check-crm-isolation.mjs` (grep-проверка обеих сторон, ненулевой код при нарушении), подключить в `build`/CI.

**Контракт-адаптер (одностороннее зеркало, только В CRM):**
- `ingest(event)` — единственный вход внешних данных. События: `onboarding.completed` (→ найти/создать profile + создать Offer с onboardingLink + Application), `application.stage-changed` (→ статус заявки), `dvu.result` (→ отметка + история).
- DTO событий дублируют primary-поля (сериализуемый JSON), НЕ ссылаются на чужой домен. В реальности → consumer очереди/webhook.
- `adapter/fixtures.ts` + кнопка «Симулировать завершение онбординга» — для демо без реального флоу.

**Маппинг на реальность:** crmApi→REST/GraphQL CRM-сервиса; state→своя БД; events→event-контракт на шине; CRM↔Онбординг асинхронно односторонне; CRM↔Probe42 синхронное чтение; CRM↔ДВУ через `dvu.result`. CRM→ЕПК — отдельный исходящий адаптер позже ([OQ-7]).
