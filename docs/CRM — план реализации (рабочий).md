# CRM «одного окна» — план реализации (рабочий, автономный прогон)

Статус: рабочий план оркестрации. Денис смотрит **финал в рабочем виде** (кликабельный CRM на MUI), промежуточно не дёргаем.
Источники: `docs/Разбор встречи — CRM одного окна 2026-06-25.md`, `docs/_исходники/2026-06-25_СРМ-01.txt`.

---

## 0. Цель и ограничения

**Цель:** накидать кликабельный CRM-прототип на MUI от концепции (без скринов от Дениса) — Company Profile одного окна + воронка + сценарий входящего звонка.

**Жёсткие ограничения:**
1. **Микросервис-изоляция.** Вся CRM в `app/src/crm/`. НЕ импортит из `screens/v2*`, `screens/v2/company`, `mock/v2/company*`; и наоборот они не импортят из `crm/`. Изменения CRM не текут в онбординг/ДВУ.
2. **UI = MUI** (`@mui/material@5`, каркас готов: `app/src/rm/theme.ts`, `RmThemeProvider`). НЕ SDDS.
3. **Свой mock-слой** `app/src/crm/mock/` — отдельные сущности, не общий с онбордингом.
4. **Связь с онбордингом/Probe/ДВУ — только через адаптер** (read-only события в CRM), без обратной связности.
5. Прото боевой, не выкидной. Сборка всегда зелёная. Клиентский флоу не ломаем.

## 1. Фазы

### Фаза A — Спека (агенты-аналитики, параллельно)
- **A1 · Костя (tech-lead):** архитектура CRM-микросервиса — структура `crm/`, mock-слой, контракт-адаптер, правила изоляции (ESLint-guard), маппинг на реальный микросервис, оценка.
- **A2 · Боря (BA):** фич-лист (всё в кучу, без деления MVP) + юзкейсы (входящий звонок найден/не найден, авто-оффер, брошенная заявка) + сущности/атрибуты + OQ к Марго + акторы.
- **Гейт A:** оба вернулись → я свожу в `docs/CRM — спецификация и фичи.md`.

### Фаза B — Дизайн-спека (после A)
- **B1 · Ульяна (UX):** продуктовая логика + визуал ключевых экранов на MUI-паттернах:
  - CRM-вход «одно окно» (поиск по PAN + список клиентов).
  - Company Profile (агрегатор): шапка-атрибуты, AI summary, воронка, история, действия.
  - Сценарий входящего звонка (найден / не найден → создание профиля).
  - Дизайн-бриф: иерархия, паттерны, состояния (empty/loading/error), микротексты RU/EN.
- **Гейт B:** дизайн-бриф готов → дополняю спеку.

### Фаза C — Реализация (Федя, по спеке)
Изолированный модуль `app/src/crm/`:
- **C1 · Каркас модуля:** структура папок, роутинг под `/rm/crm/*`, ESLint-guard на границу.
- **C2 · Mock-слой** `crm/mock/`: типы (CompanyProfile, Lead, Offer, Deal, Application) + seed + API (стиль `companyApi.ts`, но свой) + адаптер.
- **C3 · Экраны на MUI:**
  1. **CrmSearch** — одно окно: поиск по PAN + список клиентов (имитация All Clients) + столбец Application.
  2. **CompanyProfile** — агрегатор: primary-атрибуты (PAN обяз./CIN условно), AI summary (заглушка-блок), воронка Lead→Offer→Deal→Application со статусами, история коммуникаций, действия (создать оффер/звонок/ссылку).
  3. **CreateProfile** — сценарий «не найдено»: форма по телефону / авто-заполнение из Probe42 (mock) → дропдаун «откуда пришёл».
- **C4 · Навигация:** вход в CRM из DemoNav (и/или RM-хаба), переходы между экранами.

### Фаза D — QA + пуш
- **D1 · Вера (QA):** прогон CRM-флоу как пользователь (битые переходы, мёртвые кнопки, RU/EN, состояния).
- **D2 · Сборка** `npm run build` зелёная; проверка изоляции (нет cross-import).
- **D3 · Пуш** в `origin/main` → Render.
- **D4 · Журнал/доки:** запись о CRM, обновление статуса.

## 2. Объём экранов v1 (что строим)

| Экран | Роут | Суть |
|---|---|---|
| CRM Search | `/rm/crm` | одно окно: поиск по PAN, список клиентов, статусы Application |
| Company Profile | `/rm/crm/profile/:id` | агрегатор: атрибуты + AI summary + воронка + история + действия |
| Create Profile | `/rm/crm/new` | не найдено → создание (телефон / Probe42 auto) |

Заглушки (блоки, не отдельные экраны в v1): AI summary, таски/календарь, авто-оффер-из-онбординга (показать как событие в истории).

## 3. Изоляция — контракт (проверяемо)

- `app/src/crm/**` — единственное место CRM.
- Запрет импортов: `crm/` ↔ `screens/v2`, `company`, `mock/v2`. Гард: ESLint `no-restricted-imports` (Костя финализирует паттерны).
- Адаптер `crm/adapter/` — единственная точка входа внешних данных (mock-события: onboarding-completed→offer, application-stage-changed, dvu-result). Только В CRM.
- Своя тема (MUI), свой роутинг-неймспейс `/rm/crm/*`.

## 5. Порядок исполнения (для меня)

1. Перезапустить A1 (Костя) + A2 (Боря) параллельно.
2. По возврату обоих → свести спеку (Гейт A).
3. Запустить B1 (Ульяна) → дополнить спеку (Гейт B).
4. Запустить C (Федя) поэтапно C1→C2→C3→C4 (одним заданием с подэтапами, изоляция).
5. D1 (Вера) → правки если надо → D2 сборка → D3 пуш → D4 журнал.
6. Финальная сводка Денису.

**Принцип:** не дёргать Дениса промежуточно. Гейты прохожу сам. Если упрусь в продуктовую развилку без дефолта — зафиксирую допущение в спеке и продолжу, помечу для финального ревью.
