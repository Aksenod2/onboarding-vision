# CRM-трек — правила работы команды (гейты)

Зачем: чтобы фичи не проектировались «из головы», а проходили через **обязательные гейты** (research → обоснование → архитектура → дизайн → реализация → QA → лог). Гейт нельзя пропускать; пропуск = откат к пропущенному шагу. Решение не считается принятым, пока не зафиксировано в документе.

Повод: критика Дениса 26.06 — «нужны лучшие практики, логическое обоснование и приземлённость на наш процесс заведения клиентов».

---

## Конвейер гейтов

| Гейт | Кто | Критерий прохода (Definition of Done) | Артефакт | Нельзя дальше, пока… |
|---|---|---|---|---|
| **G0 · Research** | Марк | паттерн/фича подкреплён бенчмарком лидеров **со ссылками**; помечено, что применимо к нам | запись в `docs/CRM — research рынка.md` | нет источников → фича не проектируется |
| **G1 · Обоснование** | Соня | у фичи есть: job-to-be-done · ценность · референс рынка · **приземление на наш процесс заведения клиентов** · вес | `docs/CRM — фич-каталог.md` | фича «из головы», без обоснования → не принимается |
| **G2 · Архитектура** | Лев (домен) + Костя (тех) | доменная логика, состояния сущностей, границы/зависимости; реализуемость и **изоляция** (онбординг ↔ CRM+DVU) | архитектурный док + карта стыковок | связность/границы не определены |
| **G3 · Дизайн** | Ульяна | дизайн-бриф: задача экрана, иерархия, состояния (empty/loading/error), микротексты **RU+EN**, a11y | `docs/ds/*.md` | нет брифа → Федя не кодит |
| **G4 · Реализация** | Федя | код по утверждённым спеке+брифу; `npm run build` зелёная; **гард изоляции** `check:crm` проходит; клиентский флоу не сломан | модуль `app/src/crm/*` (`dvu/`) | сборка/гард красные |
| **G5 · QA** | Вера / Рома | прогон **реальным браузером**; **правильный вьюпорт** (RM/CRM = десктоп; клиент = мобайл); 0 битых переходов/console-ошибок; на проде после деплоя | отчёт QA + скрины | не проверено на нужном вьюпорте |
| **G6 · Лог + Решение** | оркестратор | запись в `docs/CRM — лог работы команды.md`; спорные решения — отдельный документ-кандидат с **мнениями ключевых ролей**; OQ к Марго/Сергею помечены | лог + документ решения | работа не залогирована / решение не зафиксировано |

## Сквозные правила (на всех гейтах)

1. **Никаких фич «из головы».** Только research (G0) + обоснование (G1) + приземление на наш процесс. Это прямое требование Дениса.
2. **Мнения ключевых ролей — в документ ДО решения.** На спорных архитектурных/продуктовых развилках — собрать панель (Костя/Соня/Боря/Лев + при нужде Марк), свести в документ, и только потом фиксировать. Для оперирования на встречах.
3. **Решение не принято, пока не записано.** Устные договорённости не считаются (урок встреч с Марго: «надо всё писать»).
4. **Изоляция — закон.** CRM/DVU не течёт в онбординг; гард-скрипт `check:crm` — обязательный гейт G4. Граница: онбординг (клиент) | CRM+DVU (банк).
5. **Вьюпорт по роли.** RM/CRM — внутренний инструмент, тестируем на **десктопе**; клиентский онбординг — на **мобайле**. Не путать (урок: десктоп-баг профиля проскочил из-за мобильного QA).
6. **Открытые вопросы — явно.** Всё недосказанное → помечать OQ с источником, не домысливать. Блокеры выносить на встречу с Марго/Сергеем.
7. **Гейт нельзя перескочить.** Если шаг пропущен (напр. кодим без дизайн-брифа) — откат к нему, а не «потом дочиним».
8. **Каждая фича логируется (G6).** Кто из ролей что проработал — в лог работы команды.
9. **Контракт входа (anti-«собрал не то»).** Прежде чем исполнитель начнёт основную работу на своём гейте (особенно **G4 реализация**), он возвращает **краткую сверку входа**: что понял из спеки, как маппит её на результат, открытые вопросы. Владелец гейта/оркестратор сверяет — и только потом исполнитель идёт вглубь. Ловит расхождение «спека ↔ исполнение» ДО траты, а не после. Спека на входе = 4 дока (архитектура + UX-концепты) + зафиксированные решения; исполнитель обязан их прочитать и подтвердить понимание.

|---|---|---|
| Марк (research) | G0 | бенчмарк рынка с источниками |
| Соня (продакт) | G1 | обоснование фич, фич-каталог |

| Ульяна (UX) | G3 | дизайн-бриф |

| Вера / Рома (QA) | G5 | прогон, вьюпорт, прод-гейт |
| Боря (BA) | сквозной | сверка с BRD/процессом, OQ к банку |
| Оркестратор (сессия) | G6 | лог, свод решений, фиксация |

---

## Как это применяется на практике
- Новая фича/тема CRM проходит G0→G6 по порядку. Оркестратор раздаёт гейты ролям, собирает артефакты, не пускает дальше без прохода.
- Параллелить можно независимые гейты (напр. G0 research и G2 разбор нашего процесса), но синтез — на гейте-барьере.
- Спорное решение = панель мнений (правило 2) → документ → утверждение Денисом → лог.
