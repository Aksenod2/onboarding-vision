# Дизайн-бриф — левая навигация-хаб (Компания)

> Автор: Ульяна (Design Lead, клиент). Сценарий: **Компания**. Статус: **на утверждение Денису ДО реализации** (не Феде сразу).
> Рамка: боевой прототип на палитре SDDS Serv, зелёный акцент клиента, тема `sdds_serv__light`, без кастома вне whitelist.
> Соблюдает `docs/UX-гайд.md`. Кода не содержит — это бриф.

---

## TL;DR

Навигация Компании сейчас спрятана «под кнопкой» (демо-DemoNav / четыре кубика) — это инструмент тестеров, не продуктовая навигация. Выношу **всю продуктовую навигацию заявки в постоянную левую панель-хаб**: дашборд перестаёт быть отдельным экраном «куда надо доехать» и становится **постоянно видимым каркасом** — слева список разделов заявки со статус-бейджами, справа контент текущего раздела. Клиент в любой момент видит «что с моей заявкой», проваливается в раздел кликом по пункту слева и возвращается тем же кликом. **Левая панель замещает верхний StepProgress** в сценарии Компания (не плодим две навигации). На мобайле (80–90% трафика!) панель — это **выезжающий drawer слева** + всегда видимая узкая статус-полоса; контент во всю ширину. Необратимые этапы (VKYC + подписание — фаза B) в панели **заблокированы** (read-only, не кликабельны). Финал (счёт открыт) — панель растворяется вместе с онбординг-хромом.

## 1. Пользовательский путь по навигации

### Базовая модель
Дашборд-хаб больше не «экран №8 в списке DemoNav», а **оболочка (layout)**, внутри которой живут все экраны заявки Компании. Левая панель = постоянный хаб (это и есть «дом заявки»). Контент справа = текущий раздел. Переход между разделами = клик по пункту слева ИЛИ кнопка «Продолжить» внутри раздела (линейный проход вперёд сохраняется).

### Две зоны навигации (по обратимости — из ЗАМЫСЛА Бори)

**Зона A — фаза заполнителя (Customer Representative), ОБРАТИМАЯ.**
Разделы `COMPANY_STEPS_A`: Анкета → Подтверждение данных компании → Подписанты и BR → Приглашение подписантов. Все четыре **навигируемы**: клиент свободно ходит вперёд/назад, правит, возвращается. В панели — кликабельны (как «прыжок на раздел», эквивалент клика по сегменту прогресса сегодня).

**Зона B — мониторинг + необратимое (фаза подписантов), READ-ONLY.**
После рассылки приглашений заполнитель попадает в режим **мониторинга**: левая панель показывает блоки заявки как статус-индикаторы (что проверяется банком, кто из подписантов на каком под-этапе). Эти пункты **не кликабельны как «правка»** — VKYC и подписание необратимы (UX-гайд: необратимые шаги дизейблятся, не предупреждают). Клик по блоку-мониторингу = раскрытие деталей (drill-down в участников), НЕ возврат на форму.

### Три стадии — как ведёт себя путь

| Стадия | Что в фокусе | Левая панель | Контент справа |
|---|---|---|---|
| **A. Заполнение (заполнитель)** | пройти 4 раздела фазы A | разделы A кликабельны, активный подсвечен, пройденные = ✓; разделы мониторинга (фаза B) видны, но приглушены/заблокированы (ещё не начаты) | форма текущего раздела (анкета / подтверждение / подписанты-BR / рассылка) |
| **B. Мониторинг (после рассылки)** | следить за подписантами + проверкой банка | разделы A = ✓ done (доступны read, «посмотреть»); блок «Идентификация и подписание участников» активен — статус сводный; клик = drill-down в участников | дашборд-обзор: блоки заявки со статусами + аккордеон участников (текущий `CompanyDashboard`) |
| **C. Финал (счёт открыт)** | переход в интернет-банк | **панель растворяется** — онбординг-хром исчезает (UX-гайд: финал = смена среды, не поздравление) | блок «Счёт открыт · ожидает активации» → кнопка в интернет-банк |

### «Проваливание» и «возврат»
- **Провал в раздел:** клик по пункту слева → контент справа меняется на этот раздел, пункт становится «активным». Никаких отдельных экранов-переходов.
- **Возврат с обновлённым статусом:** завершил раздел → «Продолжить» → следующий раздел; пройденный получает ✓ в панели мгновенно. «Что с заявкой» виден всегда — статусы в панели живые.
- **Кнопка «← Обзор заявки» из шапки больше не нужна** в Компании: обзор и есть постоянная панель. (BackLink в `StepProgress` остаётся только для Sole Proprietor, который мы здесь не трогаем.)

### Приглашённый подписант (отдельная точка входа по invite)
Подписант входит по своей ссылке и видит **только свою мини-сессию** (`SIGNATORY_STEPS`: Согласия → Aadhaar → Подписание → VKYC), а не хаб заполнителя. Левую панель-хаб заявки ему **не показываем** (у него нет доступа к дашборду инициатора — ср. `hideBack` #41). Статусы он видит как read внутри своей сессии (мини-прогресс), не как навигацию по всей заявке. → Левая панель-хаб = инструмент **заполнителя**, не подписанта.

## 2. Форма левой панели-хаба

### Структура сверху вниз
```
┌──────────────────────────┐
│  [Шапка заявки]           │  ← название компании + № заявки + общий статус-бейдж
│  Acme Pvt Ltd            │     (то, что сейчас в PageMeta/AppId дашборда)
│  Заявка #IN-… · В обработке│
├──────────────────────────┤
│  РАЗДЕЛЫ ЗАЯВКИ           │  ← eyebrow-лейбл секции
│                          │
│  ① Анкета            ✓   │  ← пункт = иконка-порядок + название + статус-бейдж/иконка
│  ② Подтверждение …   ●   │     активный — подсвечен (зелёная полоса слева / фон)
│  ③ Подписанты и BR   ○   │
│  ④ Приглашение       ○   │
│  ─────────────────────   │  ← разделитель: фаза A ↔ мониторинг
│  ⑤ Идентификация …  ⟳   │  ← блок-мониторинг (read), сине-серый «Проверяем»
│     участников           │
│  ⑥ Бизнес-профиль … ⟳   │
│  ⑦ VKYC              ⟳   │
├──────────────────────────┤
│  [Помощь / язык?]        │  ← опционально: тихий футер панели (не конкурирует)
└──────────────────────────┘
```

### Что показывает каждый пункт
- **Номер/иконка порядка** (как `StepOrderBadge` дашборда) — для фазы A; для блоков-мониторинга номер можно опустить (это не шаги клиента, а состояния).
- **Название раздела** — из конфиг-массива (плейсхолдеры, см. §5).
- **Статус** — компактный бейдж ИЛИ иконка-оверлей справа: done ✓ / current ● / pending ○ / verifying ⟳ / action-required ⚠. Цвета — те же токены, что в дашборде (done зелёный, current синий, verifying сине-серый, action-required оранжевый). **Verifying ≠ warning** (сине-серый, не оранжевый) — сквозное правило гайда.

### Состояния пункта
| Состояние | Вид | Кликабельность |
|---|---|---|
| **active** (текущий раздел) | зелёная вертикальная полоса слева + лёгкий фон-подсветка, текст `textPrimary` bold | — (уже здесь) |
| **done** (пройден, обратимый) | ✓ зелёный, текст обычный | кликабелен → возврат/правка |
| **pending** (будущий, обратимый) | ○ серый, текст `textSecondary` приглушён | кликабелен, если предыдущие позволяют (или дизейбл до достижения — см. вопрос §5) |
| **verifying / monitoring** (фаза B, read) | ⟳ сине-серый, подпись «Проверяем» | клик = drill-down (раскрыть детали), НЕ правка |
| **locked / irreversible** (VKYC, подписание) | приглушён, курсор default, `aria-disabled` | не кликабелен (UX-гайд: дизейбл, не предупреждение) |

### Связь с контентом справа
Один источник истины — конфиг-массив разделов (по образцу `COMPANY_STEPS_A` + блоки `getApplicationBlocks`). Активный пункт ↔ контент строго синхронны (по роуту/id). Контент справа сохраняет текущую ширину контентной области (см. §3), панель — отдельная фиксированная колонка слева.

### SDDS-компоненты для пунктов
Сами «строки-разделы» можно собрать на **`Cell`** (SDDS Serv) — штатный список-элемент с левым слотом (иконка статуса), title/subtitle и правым слотом (бейдж). Это согласуется с принятым в дашборде паттерном (`Cell` + статус-бейдж, см. UX-гайд «двухуровневый дашборд»). Drill-down блока-мониторинга — **`Accordion` / `AccordionItem`** (уже используется в `CompanyDashboard`). Бейдж статуса — собственный styled-бейдж на токенах (как сейчас `BlockBadge`), штатного Badge в whitelist не подтверждено — сверить по MCP (см. §6).

## 3. Двухколоночный layout + мобильное поведение

### Десктоп
- `ScreenV2` получает второй режим: **сетка из двух колонок** — фиксированная левая панель (ориентир **280–300px**) + контент. Контент сохраняет читаемую ширину (≈ нынешние 880px как max, но теперь max ВНУТРИ правой колонки, не центрируется на всю страницу). Шапка (лого + язык) остаётся сверху на всю ширину.
- Панель — `position: sticky; top: …` (прокручивается контент, панель стоит). Высокий список разделов внутри панели скроллится самостоятельно при нехватке высоты.
- **Важно для Феди:** это структурное изменение `ScreenV2` (сейчас одноколоночный). Двухколоночный режим включается только для роутов Компании-заполнителя; Sole Proprietor и экраны подписанта остаются одноколоночными. Реализацию (проп/обёртка) определяет Tech Lead — НЕ в этом брифе.

### Мобайл (80–90% трафика — приоритет!)
- **Панель уезжает в drawer слева** (свайп-драйвер — упоминалось на встрече 17.06). По умолчанию **скрыта**, контент во всю ширину.
- **Всегда видимый якорь:** узкая верхняя **статус-полоса** под шапкой — кнопка-гамбургер «☰ Обзор заявки» + текущий раздел + общий статус-бейдж. Это и вызов drawer, и постоянный ответ на «где я / что с заявкой» без открытия панели.
- Тап по «☰ Обзор заявки» / свайп от левого края → drawer выезжает поверх контента (overlay-затемнение, `Drawer` SDDS — сверить по MCP), показывает тот же список разделов. Выбор раздела → drawer закрывается, контент меняется.
- Закрытие drawer: тап по затемнению / свайп влево / выбор пункта.
- **Overlay-правила едины** (UX-гайд: модалка/лайтбокс/drawer — единые правила) — drawer подчиняется тем же токенам затемнения/анимации, что лайтбоксы прототипа.
- Брейкпоинт: ниже **768px** (текущий media-порог `ScreenV2`) — двухколоночность сворачивается в drawer-режим.

## 4. Решение по StepProgress

**Левая панель ЗАМЕЩАЕТ верхний `StepProgress` в сценарии Компания (фаза A).** Не сосуществуют.

**Обоснование:**
- Две навигации одного смысла (верхняя полоса-сегменты + левый список разделов) = ровно та «размытая иерархия / навигация под кнопкой», которую Денис просит убрать. Один модуль навигации = один список слева.
- Левая панель **строго мощнее** верхней полосы: даёт и порядок, и статусы, и названия, и постоянную видимость «что с заявкой» — то, ради чего StepProgress и делался. Сегменты-полоски без подписей проигрывают именованному списку со статус-бейджами.
- Принятые правила StepProgress **переносятся в панель** без потери: кликабельность обратимых разделов (= прыжок по сегменту), дизейбл необратимых (VKYC/подписание), «← Обзор заявки» становится не нужен (панель и есть обзор).
- **StepProgress остаётся как есть для Sole Proprietor** (одноколоночный линейный поток без левой панели) и для **мини-сессии подписанта** (фаза B по invite — там короткий линейный прогресс `SIGNATORY_STEPS` уместнее полноразмерной панели-хаба; подписанту хаб заявки не показываем — §1). То есть StepProgress не удаляем, а перестаём применять в хабе заполнителя Компании.

**Итого:** заполнитель Компании → левая панель (без StepProgress). Подписант Компании → мини-прогресс сессии (как сейчас). Sole Proprietor → StepProgress (без изменений).

## 6. SDDS-компоненты (сверить по MCP `sdds-serv` перед кодом)

| Назначение | Компонент SDDS | Статус |
|---|---|---|
| Строка-раздел в панели (иконка + название + статус) | **`Cell`** | в whitelist (используется в дашборде) — сверить слоты (left/right content) по MCP |
| Drill-down блока-мониторинга | **`Accordion` / `AccordionItem`** | подтверждён (в `CompanyDashboard`) |
| Кнопки действий (Продолжить / Войти как / Напомнить) | **`Button`** (view accent/secondary/clear) | подтверждён |
| Мобильный выезжающий drawer панели | **`Drawer`** (если есть в SDDS Serv) | **СВЕРИТЬ по MCP** — если нет, собираем overlay-панель на токенах по единым overlay-правилам гайда |
| Статус-бейдж раздела | штатный **`Badge`/`Indicator`** если в whitelist; иначе styled-бейдж на токенах (как `BlockBadge`) | **СВЕРИТЬ по MCP** |
| Чипы ролей участника (в drill-down) | как сейчас (styled chip) | без изменений |

MCP-инструменты для сверки: `get_component`, `get_component_props`, `get_component_examples` для `Cell`, `Drawer`, `Badge`. Боевые имена не угадывать.

## 7. Что осталось плейсхолдером / вопросом к Марго и Денису

**К Марго (нейминг/состав — не дизайн):**
- Финальные названия разделов: «company details» vs «business profile» — границы и подписи (см. §5, `[PLACEHOLDER-МАРГО]`).
- Раскрывать ли «BR» как «Board Resolution» в подписи раздела.

**К Денису (UX-решения на «да» до кода):**
1. **Замена StepProgress левой панелью в Компании** (§4) — ключевое решение, нужно прямое «да». StepProgress сохраняется для Sole Proprietor и подписанта.
2. **Ширина левой панели ≈ 280–300px** и то, что контент справа сохраняет ≈880px max ВНУТРИ правой колонки (а не центрируется на весь экран как сейчас). Согласуется ли с правилом «единая ширина 880» — или 880 теперь относится к правой колонке?
3. **Мобайл: drawer + узкая статус-полоса** как постоянный якорь (§3) — подтвердить паттерн (свайп слева + «☰ Обзор заявки»).
4. **Pending-разделы фазы A:** кликабельны сразу (свободный прыжок вперёд) ИЛИ заблокированы до достижения по порядку? По духу «обратимая фаза A» — склоняюсь к **свободно кликабельны** (как сегменты прогресса сейчас). Нужно подтверждение.
5. **DemoNav (четыре кубика) НЕ трогаем** как инструмент тестеров — он остаётся отдельно от продуктовой панели. Подтвердить, что так и оставляем.

**Не входит в бриф (передаётся Tech Lead / Феде):** как технически включать двухколоночный режим `ScreenV2` только для роутов заполнителя Компании; реализация sticky-панели и drawer; роутинг. Реализацию кодом здесь не предлагаю.
