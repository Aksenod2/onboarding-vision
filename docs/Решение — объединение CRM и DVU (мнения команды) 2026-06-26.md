# Решение-кандидат: объединение CRM и DVU — мнения команды

Дата: 2026-06-26. Статус: **на обсуждение** (не утверждено; материал для встреч с банком/Марго/Сергеем).
Вопрос Дениса: объединить CRM и DVU в один продукт «отдел работы с клиентами» (разные роли), не разделять на отдельные микросервисы. Просьба — собрать конструктивную критику ключевых участников от наших документов и процессов.

---

## TL;DR — консенсус команды

**ДА, объединять в один продукт/окно. НЕ дробить на микросервисы.** Это подтверждает уже принятый курс (README/CLAUDE.md: «DVU — под-роли внутри RM, RBAC поверх единого Caldera»), а не меняет его.

**Уточнение (все 4 роли):** «один продукт» ≠ «один экран-комбайн». Реализовать как **2 домена (CRM + DVU) под единым UI + RBAC**, с разными рабочими столами по ролям:
- общее ядро — Company Profile / client-360;
- Sales-workspace (менеджер + оператор колл-центра, воронка-центричный);
- DVU-workspace (очередь алертов, task-центричный);
- RBAC переключает не только видимость кнопок, а **рабочий стол**.

**Единственное серьёзное возражение (у всех):** segregation of duties / four-eyes — RBAC обязан жёстко гарантировать разделение обязанностей (KYC/AML-регуляторика). Один продукт — да; одна роль с полным доступом — нет.

## Мнения по ролям

### Костя — техлид
Вердикт: **за объединение в один продукт/сервис; не микросервисы.**
- «Один сервис» ≠ «один плоский модуль»: зонтичный модуль с подмодулями `profile/` (read-mostly) + `dvu/` (write-heavy очередь задач).
- Граница изоляции остаётся одна: **онбординг ↔ (CRM+DVU)**, не CRM↔DVU. `check-crm-isolation.mjs` менять почти не надо.
- DVU сейчас раздроблен по коду в 3 местах (RM01–05 на SDDS; `triggered:'DVU'`/`uploadDvuDocument` в онбординге; `dvu.result` в CRM-адаптере) — настоящий тех-долг «DVU нигде не живёт целиком».
- Риск разделения на микросервисы: дубль primary-полей, две БД, тройная хореография, ломается UX «одного окна».
- Условия: DVU держит собственную сущность Task (владелец-офицер, лог решений); RBAC-границы (Compliance ≠ DVU ≠ менеджер) явно.
- Закрывает OQ-9: `dvu.result` становится внутренним переходом.
- Развилка (нейминг): переименовать ли зонтичный `crm/` → `workspace/`.

### Соня — CRM-продакт
Вердикт: **за объединение платформы/ядра; НЕ «один экран с RBAC-тумблером».**
- Объединяем платформу (client-360, данные, история) — разделяем рабочие столы.
- Job-ы ролей: менеджер (pull, воронка/портфель), оператор (reactive, входящие), DVU (push, очередь алертов). Менеджер+оператор близки → один Sales-workspace; DVU — отдельный alert-queue стол.
- Рынок (Марк): Fenergo/nCino — единые CLM-платформы, но KYC-разбор = отдельный case-management workflow; Salesforce FSC/Dynamics — Sales Cloud vs Service Console (разные консоли над общим Account 360).
- Минусы наивного слияния в один экран: разные ментальные модели (продажа vs проверка), pull vs push, **segregation of duties**, несравнимые KPI.
- 3 развилки: A) гранулярность окна (рек. — ядро + workspace-пресеты); B) менеджер+оператор вместе (рек. — да); C) DVU — отдельный рабочий стол того же продукта (рек.).

### Боря — бизнес-аналитик
Вердикт: **за объединение в один продукт под RBAC; наши документы это прямо поддерживают.**
- BRD: DVU = AOD — организационная служба («part of Account Opening Division»). Наша архитектура (README:40–41) уже трактует как под-роль в RM/Caldera через RBAC. → «служба vs роль» снимается: организационно служба, в продукте — роль.
- Акторов ≥7: RM/CRM Team, Call Centre Operator, DVU/AOD officer, DVU Manager, Compliance (AOD), Concurrent Audit, Management.
- Возражение (SoD из BRD): «система исключает офицера, проводившего VKYC, из её валидации» (four-eyes, `3-DVU #8`); Compliance — отдельный контур; Concurrent Audit — независимая третья рука. Регуляторных причин держать DVU отдельным **продуктом** — нет; есть причины держать отдельные **роли с разными правами**.
- Аргумент ЗА: BRD требует «complete digital audit trail» — единый продукт = единый сквозной след.

### Лев — solution-архитектор
Вердикт: **за объединение как ОДНО окно (UI+RBAC) над ДВУМЯ доменами, не один bounded context.**
- По карте зависимостей CRM↔DVU связаны слабо (1 событие `dvu.result` + обратный запрос документа); DVU плотнее связан с **онбордингом** (7 триггеров). «Есть событие» сам по себе слияние не обосновывает.
- DDD: CRM (продажи) и DVU (комплаенс/верификация) — разные ubiquitous languages → разные bounded contexts. Слить в один домен = смешать коммерческую и регуляторную логику (антипаттерн).
- Граница изоляции к онбордингу не ломается; `dvu.result` становится внутренним контрактом (DTO, не общий стейт) — аудит-граница сохраняется.
- Структурная рекомендация: DVU — **параллельный модуль** `app/src/dvu/` (не подпапка CRM), оба под `/rm/*` в общей теме.
- v1: DVU-очередь пока заглушка (fixture), модуль поднимать отдельной фазой.

### Марк — research рынка (опорные данные)
- Fenergo, nCino — единые client-lifecycle (CLM) платформы: CRM + онбординг + KYC/KYB вместе, single client view.
- Но даже у них KYC-разбор — отдельный case-management/review workflow с очередью и SLA, не та же канва, где продажник ведёт сделку.
- Salesforce FSC / Dynamics — Sales Cloud (менеджер) vs Service/Onboarding Console (оператор/бэк) над общим Account 360.
- Маппинг: «один продукт» = да (как Fenergo/FSC); «одна роль-комбайн» = нет даже у лидеров.

## Согласованная формулировка решения (проект, на утверждение)

> **Единое рабочее место «отдел работы с клиентами» = один продукт/сервис.** Внутри — **2 домена (CRM + DVU)** под общим UI и матрицей RBAC-ролей (RM/Sales · Call Center · DVU-операционка · Compliance · Concurrent Audit). Граница изоляции: **онбординг (клиентский флоу) | CRM+DVU**. Обязательное правило **four-eyes** (исполнитель ≠ финальный аппрувер по одной заявке). НЕ дробить на отдельные микросервисы. НЕ сваливать роли в один экран.
