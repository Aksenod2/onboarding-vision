# Дизайн-принципы фронтенда — v2 (для прототайпера)

> Обязательный чек-лист для любого, кто верстает/правит экраны v2. **Use a frontend design**: каждый экран проходит этот фильтр перед сдачей. Прикладывается к брифам субагентов.
>
> **Сначала — Anthropic frontend-design skill.** Перед версткой/правкой UI проверь доступные навыки (Skill / ToolSearch) на skill фронтенд-дизайна от Anthropic (имена: `frontend-design`, `webdesign`, `web-design`, `artifacts`-related) и применяй его. Принципы ниже — поверх него, как проектные ограничения (SDDS, наша иерархия, наши отступы).

## 1. Иерархия — один главный акцент
- На экране один главный визуальный акцент (CTA / главный индикатор). Всё остальное — подчинено.
- **Не дублировать индикаторы одного веса.** Если есть два прогресса (шаги + под-шаги) — верхний главный, вложенный явно тише: мельче, серее, тоньше, другой паттерн. Они не должны выглядеть как два равных бара.

## 2. Отступы — base-8 и «внешние > внутренние»
- Сетка кратно 8px (8 / 16 / 24 / 32).
- Элементы одной группы — близко (8px); разные уровни — дальше (16px); группа от соседнего блока — ещё дальше (24–32px). Близость = принадлежность.

## 3. Типографика — правильный токен под размер
- Текст — токены `body*` (bodyL/M/S). Заголовки секций — `h4…h6`.
- **`dsplL/M/S`, `h1…h3` тащат крупный line-height** — НЕЛЬЗЯ применять к мелкому тексту (раздувает высоту блока). Проверено на StepName (был 560px высоты из dsplLBold).
- Не злоупотреблять CAPS. Два CAPS-блока рядом — конфликт. Капс — максимум один акцент-лейбл в зоне.

## 4. Не светить внутреннюю терминологию
- Технические имена атрибутов/кодов из источников (Confluence, БД) — НЕ показывать пользователю. Пример анти-паттерна: «BUSINESS INDUSTRY / SEGMENT» как заголовок вопроса. Пользователь видит нормальный человеческий текст вопроса, а не имя поля.
- Риск-скор, внутренние флаги, id — клиенту не показываем.

## 5. Консистентность
- Один визуальный язык на одинаковые сущности (все прогрессы — в одной системе; все карточки — одинаковые скругления/тени из `designSystem.ts`).
- Компоненты — только SDDS Serv (whitelist), props сверять с MCP `sdds-serv`, не угадывать.

## 6. Состояния и выравнивание
- Каждый интерактив имеет hover/active/disabled (где уместно) — но в режиме свободной проверки кнопки не блокируем.
- Выравнивание по сетке контента, ничего не «висит» произвольно.

## Чек перед сдачей экрана
- [ ] Один главный акцент, вложенное — подчинено?
- [ ] Отступы по base-8, внешние > внутренние?
- [ ] Токен типографики под размер (не dspl/h1-3 для мелкого)?
- [ ] Нет технической терминологии в UI?
- [ ] Консистентно с соседними экранами?
- [ ] SDDS-компоненты, props по MCP?
