## Как читать карту

Каждый экран = карточка из **трёх срезов** + происхождение + навигация:

| Элемент | Что значит |
|---|---|
| **ID** | Код экрана: `CL-xx` (Клиент) / `RM-xx` (менеджер) |
| **Роль · Либа** | Кто пользуется и на какой UI-библиотеке (SDDS / Caldera) |
| **`source`** | Происхождение: `схема-версия : код-шага`. По нему диф при обновлении схемы |
| **🔄 Процесс** *(бизнес)* | Шаг процесса, что происходит, действие пользователя |
| **🗄 Данные** *(система)* | Сущности и поля (вход/выход), интеграции, состояния |
| **🖼 Экран** *(дизайн)* | UI-блоки, состояния экрана, компоненты библиотеки |
| **➡ Навигация** | Откуда пришёл / куда ведёт |

Три среза — это «гнёзда»: бизнес-аналитик дополняет 🔄, системный — 🗄, дизайн — 🖼. Структуру не трогают.

## Карточки экранов

### CL-01 — Приглашение / старт по ссылке
- **Роль · Либа:** Клиент · SDDS · **`source`** `2026-06-01_obo : 001`
- **🔄 Процесс:** Проспект приходит на страницу регистрации по ссылке-приглашению (из CRM/сайта). Точка входа в онбординг. Действие: открыть ссылку, начать регистрацию.
- **🗄 Данные:** Вход — токен/параметры из ссылки (entry data from the link). Выход — старт сессии онбординга. Интеграции: PIF.
- **🖼 Экран:** Приветственный лендинг: что за процесс, сколько времени займёт, CTA «Начать». Состояния: **валидная ссылка** / **ссылка истекла или недействительна**.
- **➡ Навигация:** Вход извне (ссылка) → **CL-02**.

### CL-02 — Создание логина и вход
- **Роль · Либа:** Клиент · SDDS · **`source`** `2026-06-01_obo : 002`
- **🔄 Процесс:** Шаг 002. Проспект создаёт учётную запись и входит. Действие: ввод email + телефона, подтверждение кодом.
- **🗄 Данные:** Вход — `email`, `phone`. Создаётся `person` в MR. Выход — учётные данные, аутентифицированная сессия. Интеграции: **MR, CORE, PIF**.
- **🖼 Экран:** Форма (поля email, телефон) → экран ввода OTP-кода. Состояния: **пустой** / **валидация полей** / **код отправлен** / **ошибка кода** / **успех**. Компоненты SDDS: Input, PhoneInput, Button, OTP-input, Notification.
- **➡ Навигация:** ← **CL-01** → **CL-03**. *Побочно:* бездействие (N времени) → событие-нотификация в CRM (это не экран, а триггер коммуникации).

### CL-03 — Анкета компании (KMP / Board Resolution)
- **Роль · Либа:** Клиент · SDDS · **`source`** `2026-06-01_obo : 004`
- **🔄 Процесс:** Шаг 004 «Fill in KMP». Проспект заполняет данные компании и ключевых лиц (Board Resolution — состав, подписанты, операционные инструкции).
- **🗄 Данные:** Вход — PAN компании (ручной ввод), реквизиты, подписанты (PAN/имя/должность), email и телефон подписанта, операционные инструкции. Часть полей **авто-подтягивается из реестра** (Probe42/PAN). Выход — `prospect`, `person`, `prospect_person` в DATA. Состояния полей: ручной ввод / авто-заполнено / требует правки.
- **🖼 Экран:** Многосекционная форма: «Данные компании» · «Подписанты» · «Операционные инструкции». Маркировка авто-заполненных полей (badge «из реестра»). Состояния: **черновик** / **частично авто-заполнено** / **ошибка валидации** / **готово к отправке**. Компоненты SDDS: Input, Select, Dropdown, FileUpload (Board Resolution), Repeater/список подписантов, Stepper.
- **➡ Навигация:** ← **CL-02** → **CL-04**. *Ветка:* при изменении блока данных сущности → DVU-задача (бэк-офис, вне клиентского экрана).

### CL-04 — Бизнес-анкета (Business Nature Questionnaire)
- **Роль · Либа:** Клиент · SDDS · **`source`** `2026-06-01_obo : 004 (BNQ)`
- **🔄 Процесс:** Шаг «Business Nature Assessment». Опросник определяет характер бизнеса и риск-категорию (Low / Medium / High) — влияет на дальнейший сценарий (STP / Hybrid / Offline, нужен ли VCIP с одобрением CRM Head).
- **🗄 Данные:** Вход — индустрия/сегмент, резидентность подписантов, PEP (да/нет), FATCA. Выход — риск-категория, флаг High-Risk Industry. Интеграции: правила риск-категоризации (1D-5, Work In Progress).
- **🖼 Экран:** Пошаговый опросник (radio Yes/No, выбор индустрии). Возможен итоговый экран с присвоенной риск-категорией. Состояния: **в процессе** / **High-Risk → предупреждение об доп. проверке** / **завершено**. Компоненты SDDS: RadioGroup, Select, Stepper, Alert (для High-Risk).
- **➡ Навигация:** ← **CL-03** → **CL-05** (видеоидентификация).

### CL-05 — Видеоидентификация: приглашение (eKyc / VCIP)
- **Роль · Либа:** Клиент · SDDS · **`source`** `2026-06-01_obo : 005`
- **🔄 Процесс:** Шаг 005 «Create logins & send invites + eKyc». После анкет подписанты получают приглашение пройти видеоидентификацию. Действие: ознакомиться, запустить eKyc по **Aadhaar** (QR / номер), выбрать способ — self-VKYC сейчас или запланировать VKYC/F2F.
- **🗄 Данные:** Вход — список подписантов и их статус eKyc, Aadhaar подписанта (QR-аутентификация). Выход — инициированные `VCIPSession[]`, статус eKyc по каждому подписанту. Интеграции: **VKYC, JAZZ** (selfVKYC), **PIF**, UIDAI (Aadhaar).
- **🖼 Экран:** Список подписантов со статусом (ожидает / пройдено), на каждого — «Пройти видеоидентификацию» / «Запланировать». Состояния: **приглашение отправлено** / **eKyc в процессе** / **готово к VCIP**. Компоненты SDDS: List/Cell, Button, Indicator/Badge, Steps, Note.
- **➡ Навигация:** ← **CL-04** → **CL-06**.

### CL-06 — VCIP-сессия
- **Роль · Либа:** Клиент · SDDS · **`source`** `2026-06-01_obo : 005.02 (VKYC)`
- **🔄 Процесс:** Подпроцесс VKYC. Подписант проходит **self-VKYC** видеосессию; решение `eKyc (Aadhaar) passed?` → self-VKYC или назначение VKYC/F2F-сессии. При провале — DVU VKYC.
- **🗄 Данные:** Вход — `VCIPSession` (method `selfVKYC`/`assistedVKYC`/`F2F`), результат eKyc Aadhaar. Выход — статус сессии (`Passed`/`Failed`); при провале → DVU VKYC-задача. Интеграции: **JAZZ, VKYC, DVU**.
- **🖼 Экран:** Экран видеосессии: превью камеры, инструкции, индикатор шагов (лицо → документ → согласие). Состояния: **подготовка** / **идёт сессия** / **успех** / **провал → план VKYC/F2F**. Компоненты SDDS: Button, Steps, Note (Alert), Spinner/Progress, Modal.
- **➡ Навигация:** ← **CL-05** → **CL-07**. *Ветка:* провал → DVU VKYC (бэк-офис, вне клиентского экрана).

### CL-07 — Загрузка документов
- **Роль · Либа:** Клиент · SDDS · **`source`** `2026-06-01_obo : DVU OBO (Document request)`
- **🔄 Процесс:** Когда авто-получение из реестра недоступно (**Hybrid/Offline**) или DVU OBO запросил документы — клиент загружает требуемые документы (Board Resolution, MOA & AOA, Address Proof и т.д.).
- **🗄 Данные:** Вход — список требуемых `Document[]` (handling=`Upload`/`Physical`, status=`Pending`); состав зависит от типа юрлица. Выход — загруженные файлы, status=`Uploaded` → на проверку (`System`/`DVU`). Интеграции: **DATA, DVU OBO, PIF**.
- **🖼 Экран:** Список документов со статусом (требуется / загружен / проверен), зоны загрузки, на каждый — тип и обязательность. Состояния: **ожидает загрузки** / **часть загружена** / **всё загружено** / **на проверке** / **отклонён → повтор**. Компоненты SDDS: Dropzone, File/Attach, List, Badge, Button, Note.
- **➡ Навигация:** ← **CL-06** → **CL-08**. *Ветка:* документ невалиден → DVU OBO (повторный запрос).

### CL-08 — Подтверждение и отправка заявки
- **Роль · Либа:** Клиент · SDDS · **`source`** `2026-06-01_obo : 006`
- **🔄 Процесс:** Перед запуском KYC-проверок (006) и созданием клиента (007) проспект просматривает сводку всех данных (компания, подписанты, бизнес-анкета, документы, VCIP) и подтверждает корректность → отправляет заявку.

- **➡ Навигация:** ← **CL-07** → **CL-09**. *«Изменить»* → возврат к **CL-03** / **CL-04**.

### CL-09 — Результат: открытие счёта
- **Роль · Либа:** Клиент · SDDS · **`source`** `2026-06-01_obo : 009`
- **🔄 Процесс:** Шаг 009 «Account opening trigger». После прохождения KYC (007) и выдачи доступа (008) — триггер открытия счёта (передача в CAM). Клиент видит итог и статус заявки.

### RM-01 — Очередь заявок DVU (заявко-центрично)
- **Роль · Либа:** Менеджер · SDDS (акцент **оранжевый**) · **`source`** `2026-06-01_obo : DVU OBO (Create task)`
- **🔄 Процесс:** Менеджер видит входящие **заявки** на ручной проверке. Ключевое: проверки одной заявки (домены OBO / KYC / VKYC) — это **не разрозненные задачи, а один путь** разбора заявки командой. Очередь **группируется по заявке (компании)**, внутри — статус по каждому домену. Действие: открыть нужную проверку (`Take task in progress`).
- **🗄 Данные:** Вход — `DVUTask[]`, **сгруппированы по компании** (заявке); у каждой — проверки по доменам с их статусами. Выход — открытая проверка. Интеграции: **DVU, OBO, PIF (`PIF.DVU.KAFKA`)**.
- **🖼 Экран:** Список **заявок**: строка-заявка = компания + мини-путь проверки `OBO · KYC · VKYC` со статусом каждого (✓ / ⏳), клик по домену открывает его разбор. Состояния: **есть заявки** / **пусто** / **загрузка**. Компоненты SDDS: Cell/List, Badge/Pill (статусы доменов), Steps (мини-путь).
- **➡ Навигация:** вход — через **карту экранов** (роль Менеджер). По домену → **RM-02** (OBO) / **RM-03** (KYC) / **RM-04** (VKYC-встреча) / **RM-05** (VKYC-сессии).
- **🧭 «Путь заявки»:** во всех карточках разбора (RM-02…05) сверху — единый степпер `OBO → KYC → VKYC` с подсветкой текущего домена и переключением между проверками одной заявки. Так разбор читается как **единый путь одной заявки сквозь команду**, а не 3 несвязанных экрана.

### RM-02 — Карточка задачи: проверка документов и данных
- **Роль · Либа:** Менеджер · SDDS (акцент **оранжевый**) · **`source`** `2026-06-01_obo : DVU OBO (Validate)`
- **🔄 Процесс:** Шаг DVU OBO: `Take in progress → решение «Data and Documents Valid?»`. Менеджер сверяет данные компании/подписантов и загруженные документы. Решение: **пометить валидными** (`Mark valid → Resolve alert`) **или** **запросить документы** (`Document request + notification` → клиенту).
- **🗄 Данные:** Вход — `DVUTask` + связанные `Company`, `AuthorizedSignatory[]`, `Document[]` (status, файлы). Выход — резолюция: `Resolved (valid)` → задача закрыта, поток продолжается; или `Document requested` → нотификация клиенту. Интеграции: **DVU, DATA, PIF, CRM** (нотификация).
- **🖼 Экран:** Двухколоночно: слева данные заявки (компания, подписанты, документы со статусом), справа панель решения (комментарий, кнопки «Подтвердить валидность» / «Запросить документы»). Состояния: **в работе** / **подтверждено** / **запрошены документы**. Компоненты SDDS: Cell/List, Badge, TextArea (комментарий), Button (оранж. accent), Note.
- **➡ Навигация:** ← **RM-01**. *Резолюция:* «валидно» → задача закрыта; «запросить документы» → нотификация клиенту, статус «Запрошены документы» (менеджер остаётся в очереди). Смену роли назад к клиенту несёт карта.

### RM-03 — Карточка: разбор KYC-алерта
- **Роль · Либа:** Менеджер · SDDS (акцент **оранжевый**) · **`source`** `2026-06-01_obo : 006.01.1.2.1 (DVU KYC)`
- **🔄 Процесс:** Ветка **DVU-KYC** «task to solve problem». Авто-проверка KYC дала **Alert** (PAN / OFAC / CRILC / CKYC / Probe42) — менеджер разбирает: `Take in progress → Communicate and update data → Resolve alert`. Может связаться с клиентом и обновить данные.
- **🗄 Данные:** Вход — `DVUTask` (domain=KYC) + `ScreeningResult[]` (тип проверки, статус Pass/Alert, detail), `Company`. Выход — резолюция: данные обновлены → `Resolve` (петля назад к проверкам) или эскалация. Интеграции: **DVU, KYC, PIF, CRM** (коммуникация).
