## О чём встреча

Короткая сессия 1:1 о приоритетах работ. Марго довольна направлением («подход мне нравится, прям моё»), хочет ускориться. Договорились о плане до показа Васе: минимум — добить Sole Proprietor + DVU-сценарий, максимум — верхнеуровнево показать заведение компании. Отдельно: проблема разрыва клиентского пути на VCIP/подписании (внешние ссылки из email) и решение рисовать целевое видение «всё в одном портале». Марго берёт на себя эскалацию конфликта приоритетов Дениса (задачи дизайн-команды Тани + задачи Даниловского, дедлайн 3 недели) через Васю → Таню.

## Ключевые решения

- **Приоритет 1: финализировать Sole Proprietor**, «чтобы не было к нему вообще никаких вопросов», **+ показать обработку кейса DVU** («DVU покажем Sole Proprietor» — проговорено как договорённость дня).
- **Приоритет 2 (по возможности, к вторнику):** верхнеуровнево показать заведение **компании** — даже с недочётами.
- **Сценарий компании зафиксирован:** 3 участника — **2 директора + 1 authorized signatory**; ссылка-приглашение уходит всем троим; показываем на примере одного: перешёл по ссылке → видеосессия → подписание. На экране: Board Resolution выводится клиенту, дропдаун «кто подписывает» (подписанты тянутся из Probe42), поле ввода для уполномоченного.
- **VCIP и подписание — рисуем целевое видение «всё в одном»** (внутри портала, например iframe), хотя текущие подрядчики дают и видео, и подписание отдельными ссылками из email. Клиент не должен уходить с сайта банка. Технические гэпы не прячем — подсвечиваем отдельно и устраняем.
- **Процесс синхронизации:** работаем не по всему массиву BRD-артефактов, а на базе конкретного документа **«Пользовательский путь» (юзер-кейс)** — Марго называет кейс (тип компании, число подписантов), от него и идём.
- **Эскалация приоритетов:** пишем список сценариев на доработку → встреча с Васей (показать, что есть + список) → Марго идёт к Тане с поддержкой Васи. Договорённость с Лапиным уже есть (он в отпуске). Марго в моменте начала эскалировать конфликт задач Дениса.
- Ориентир по срокам: финал Sole Prop — **ко вторнику**, показ Васе — **в среду** «по максимуму» (даты следующей недели — уточнить у Дениса).
- Зависимость: объём подписки Дениса (обсуждает с Ильёй) — если расширят, успевает больше.

## Новые требования

| ID | Требование | Экран / сущность |
|---|---|---|
| **BR-10** | Целевое видение: VCIP **и подписание** проходят внутри портала (embed/iframe), без ухода по внешним email-ссылкам; технические гэпы фиксируются как known gaps, не меняют дизайн | Видеоидентификация, Подписание |
| **BR-11** | Клиентский DVU-сценарий для Sole Proprietor: показать, как заявка обрабатывается, когда что-то ушло на DVU (не только happy path) | Дашборд, Подтверждение данных компании |
| **BR-12** | Сценарий «Компания» (следующая фаза, расширяет BR-03/04/07): Board Resolution выводится на экран; дропдаун подписантов из Probe42; поле ввода authorized signatory; ссылка-приглашение 3 участникам (2 директора + 1 AS); мульти-логин в одну сессию компании; путь участника: ссылка → VCIP → подписание | новые экраны фазы «компания» |

## Затронутые экраны

- **Видеоидентификация + Подписание** — перерисовать как единый бесшовный путь в портале (BR-10), пометить технические гэпы.
- **Дашборд** — DVU-ветка: статусы и запросы документов (BR-11, детализировано на демо-встрече того же дня → BR-15/BR-17).
- **Фаза «Компания»** — пока не строим, сценарий зафиксирован (BR-12).
