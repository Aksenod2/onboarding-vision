## TL;DR (3 строки)

1. **Экран А** перестаёт быть «приложите файл» и становится **актом назначения ОДНОГО Authorised Signatory**: сверху интро-плашка (one single AS + полномочия), затем три секции сверху-вниз — «кто подписывает BR» (Radiobox: ≥2 директора / 1 секретарь, с контактами на каждого), «кто AS» (Radiobox: из директоров / новое лицо — итог один), «governance смены AS» (Select на 2 опции); документ BR — шаблон банка дефолтом, «upload your own» мелкой ссылкой.
2. **Экран Б** перестаёт быть плоским списком людей и становится **двухуровневым кабинетом**: верх — карточки блоков заявки со статус-бейджами (включая «Verify»), drill-down в блок **Personal Identification & Signing** = текущий список участников с прогрессом подписания/VKYC; финал — «waiting for activation» + IFSC + «debit freeze».
3. **Плейсхолдеры от Марго (ОВ):** финальные тексты глоссы/полномочий (А.1–А.2), текст 2-й опции governance (Confluence, у Дениса), полный перечень блоков дашборда (Б.1). Помечены `[PLACEHOLDER-МАРГО]` — Федя ставит мой временный текст, заменяется по приходу.

# ЭКРАН А — `CompanySignatoriesBr.tsx` (Board Resolution / подписанты)

## Задача экрана (одной фразой)
Клиент **назначает одного-единственного Authorised Signatory** на весь будущий онбординг любых продуктов банка и оформляет Board Resolution. Не «загрузка документа», а юридический акт. Ведём по шаблону банка.

## Структура сверху-вниз

```
ScreenV2 (progress = StepProgress, шаг co-signatories-br)
└─ Card
   ├─ CardHeader: Title «Board Resolution» + Subtitle (что здесь происходит)
   └─ CardBody (Stack gap 1.5rem)
      │
      ├─ [СЕКЦИЯ 0] Интро-плашка «One single Authorised Signatory» + полномочия   ← Notification
      │
      ├─ [СЕКЦИЯ 1] Кто подписывает Board Resolution                              ← Radiobox-развилка + контакты
      │     ○ Минимум 2 директора        ○ Один Company Secretary
      │     └─ (в зависимости от выбора) карточки с email/phone на каждого
      │
      ├─ [СЕКЦИЯ 2] Authorised Signatory (распоряжается)                          ← Radiobox-развилка
      │     ○ Из директоров              ○ Добавить новое лицо
      │     └─ выбор директора (один) ИЛИ форма нового лица (ФИО/PAN/email/phone)
      │     итог: РОВНО ОДИН AS
      │
      ├─ [СЕКЦИЯ 3] Governance будущей смены AS                                    ← Select (2 опции)
      │
      ├─ [СЕКЦИЯ 4] Документ Board Resolution                                      ← шаблон-дефолт + мелкая ссылка
      │     (форма BR показана; «upload your own» — link снизу)
      │
      └─ ButtonRow: [Назад]  ……………  [Сформировать BR и продолжить]
```

### СЕКЦИЯ 0 — Интро «One single Authorised Signatory» + полномочия (А.1–А.2)
- **Компонент:** `Notification` (нейтральный/info-вид, сине-серый — НЕ оранжевый, НЕ warning; оранжевый зарезервирован под DVU/ошибку по гайду). Это объяснение, а не успех/тревога.
- **Содержание:** одно вводное сообщение в начале блока:
  - `title`: «One single Authorised Signatory» `[PLACEHOLDER-МАРГО]` (Марго даёт копипейст — ОВ-2).
  - `text`: вы назначаете **одного** Authorised Signatory на весь будущий онбординг **любого продукта и сервиса банка** (не только счёта) `[PLACEHOLDER-МАРГО]`. Временный текст Ульяны до прихода финального: «Вы назначаете одного уполномоченного подписанта (Authorised Signatory). Он сможет от имени компании проходить онбординг и распоряжаться любыми продуктами и сервисами банка — счетами, депозитами, кредитами, торговым финансированием и др.»
- **Состояния:** статичный блок, состояний нет. Не сворачивать в аккордеон — это контекст, который должен прочитаться (по гайду «не отвлекать», но здесь это и есть смысл шага).
- **a11y:** обычный текст, не aria-alert (это не ошибка).

### СЕКЦИЯ 1 — Кто подписывает BR: директора (≥2) ИЛИ 1 секретарь (А.3, А.6)
- **Это отдельная РОЛЬ, не AS.** Подзаголовок секции: «Кто подписывает Board Resolution» / «Who signs the Board Resolution».
- **Развилка типа подписания — `Radiobox` (2 опции, `name="br-signer-mode"`):**
  - `○ Минимум два директора` (`value="directors"`, дефолт)
  - `○ Один Company Secretary (singly)` (`value="secretary"`)
- **Ветка «директора»:** список директоров из реестра — каждый `Checkbox` (множественный выбор, мин. 2). Подпись-хинт: «Выбрано из реестра (Probe42). Минимум двое.» Бейдж «из реестра» рядом с PAN (серый, pre-filled-стиль). PAN — read-only из Probe.
  - **На каждого ОТМЕЧЕННОГО директора раскрываются два `TextField`:** Email, Phone (раньше их не было — ключевое требование А.6). Контакты вручную, т.к. в Probe их нет. Раскрытие — под строкой директора (не модалка): отметил чекбокс → появились поля.
- **Ветка «секретарь»:** одна карточка — секретарь из реестра (если есть в Probe) ИЛИ форма (ФИО/designation read-only + `TextField` email/phone). Один подписант.
- **Состояния:**
  - *empty/дефолт:* выбрана ветка «директора», отмечены все директора из реестра (как сейчас), поля контактов пустые.
  - *error:* выбрано < 2 директоров → инлайн-сообщение под списком (`caption`/`view="negative"` у секции), CTA disabled. Пустой email/phone у отмеченного подписанта → `TextField view="negative"` + caption «Укажите email»/«Укажите телефон».
  - *выбрано:* отмеченный директор — карточка с акцентной рамкой (зелёная), показаны поля контактов.
- **Почему контакты обязательны:** без email/phone «ты никогда в VKYC с ними не проведёшь» (замысел, стр.6) — это вход в персональные сессии подписантов.

### СЕКЦИЯ 2 — Authorised Signatory: из директоров ИЛИ новое лицо, итог ОДИН (А.4, А.6)
- **Убрать плоский безусловный `+ Добавить подписанта` (`AddBtn`/`handleAddAs`)** — это был «произвольный список подписантов», против концепции single. Заменить развилкой.
- **Развилка — `Radiobox` (2 опции, `name="as-mode"`):**
  - `○ Назначить из директоров` (`value="from-directors"`, дефолт) → ниже `Radiobox`-группа ИЛИ `Select` для выбора **одного** директора как AS (PAN autopopulate из Probe; связь «директор→AS» выражена явно). Один выбор, не множественный.
  - `○ Добавить новое лицо` (`value="new-person"`) → форма: `TextField` ФИО, `TextField` PAN (ручной ввод — по Aadhaar PAN не приходит, добираем в VKYC), `TextField` Email, `TextField` Phone.
- **Итог секции — РОВНО ОДИН AS** (концепция single). Не список, не кнопка добавления ещё одного.
- **Состояния:**
  - *empty:* ветка «из директоров», ни один директор не отмечен как AS → CTA disabled + хинт «Выберите Authorised Signatory».
  - *выбрано (из директоров):* подсвечен выбранный директор; показан его PAN (read-only) + контакты (если уже введены в секции 1 — переиспользовать, не спрашивать дважды; если директор не отмечен подписантом в секции 1 — показать поля email/phone здесь).
  - *выбрано (новое лицо):* форма с валидацией (PAN-формат, email, phone).
  - *error:* невалидный PAN → `TextField view="negative"` + caption.
- **Микротекст хинта:** «Authorised Signatory распоряжается продуктами банка от имени компании. Он один.»

### СЕКЦИЯ 3 — Governance будущей смены AS (А.5)
- **Компонент:** `Select` (single-select, `target="textfield-like"`, `label`=«Как будет проходить смена Authorised Signatory»).
- **items (2 опции из BRD 1D-6 #5):**
  - `{ value: 'nominated-official', label: 'Nominated Authorized Official of the Company' }`
  - `{ value: 'decision-pursuant-br', label: 'Decision Pursuant based on Board Resolution' }`
- **`helperText`:** короткое пояснение, что это определяет порядок будущей замены подписанта. Текст 2-й опции — `[PLACEHOLDER-МАРГО/ДЕНИС]` (Confluence-страница недоступна, ОВ-1). Лимит рядом — **не размещать** (только в аудио, не подтверждён, ОВ-1).
- **Состояния:** *empty* — placeholder «Выберите вариант», CTA не блокируется этим (или блокируется — уточнить; по умолчанию делаем обязательным, дефолт не проставляем, чтобы выбор был осознанным). *выбрано* — значение в поле.

### СЕКЦИЯ 4 — Документ Board Resolution: шаблон-дефолт, upload мелко (А.8)
- **Убрать две равноправные `BrOption` radio-карточки.** Шаблон банка — **путь по умолчанию**, не выбор из двух равных.
- **Раскладка:**
  - Сверху — блок «Документ формируется по шаблону банка» с короткой подписью (DSC-подпись участниками в их сессиях). Показываем, что форма BR будет сгенерирована (можно дать `Button view="clear"` «Посмотреть шаблон» → тост/preview, нейтральная ссылка, не зелёная).
  - Снизу — **мелкая неприметная ссылка** `Button view="clear" size="s"` или текст-ссылка: «Загрузить свой Board Resolution» / «Upload your own».
- **При клике на «upload your own»:** раскрывается `Attach` (boevой компонент загрузки вместо `UploadZone`). После выбора файла — `Spinner` + «Распознаём документ…» → состояние «Распознано». Под ним — `Notification`/info-плашка (сине-серая, НЕ warning): «Документ уйдёт на проверку менеджеру банка (вне автоматического маршрута). Это может занять больше времени» — DVU человеческими словами.
- **Состояния:** *default* — шаблон банка, upload скрыт за ссылкой. *upload-idle* — `Attach` готов. *upload-recognizing* — `Spinner`. *upload-done* — «Распознано» + info-плашка DVU.
- **Логика по гайду:** «один акцентный CTA» — зелёный только у «Сформировать BR и продолжить»; «Посмотреть шаблон» и «upload» — нейтральные.

### Нижний ряд действий
- `ButtonRow`: слева `Button view="secondary"` «Назад» (шаг обратим — зона правки↔подтверждения по гайду), справа `Button view="accent"` «Сформировать BR и продолжить».
- CTA disabled пока: < 2 подписантов (или секретарь не задан), не выбран AS, не выбран governance, не введены контакты подписантов.

### Что проверить (Вера) — экран А
- Mobile: секции не ломаются, поля email/phone под директором переносятся, Radiobox-группы читаемы.
- Desktop: ширина 880, ритм 24px между секциями.
- Выбор < 2 директоров блокирует CTA; ветка «секретарь» даёт одного.
- AS всегда ровно один; нельзя добавить второго.
- Контакты собираются и у подписывающих, и у AS; не дублируются, если директор = AS.
- Все radio/checkbox — боевые SDDS (фокус с клавиатуры, Space/Enter), не div-имитация.

# ЭКРАН Б — `CompanyDashboard.tsx` (двухуровневый кабинет)

## Задача экрана
Инициатор (Customer Representative) видит **верхнеуровневую картину «что с моей заявкой»** (блоки заявки + статусы, включая «Verify»), может **провалиться в детализацию** блока **Personal Identification & Signing** (прогресс подписания + VKYC по каждому участнику). Финал — счёт открыт, но «waiting for activation».

## Структура сверху-вниз

```
ScreenV2
├─ PageMeta: AppTitle «Заявка компании» + общий статус-бейдж
├─ AppId: № заявки · юр.имя
│
├─ [ВЕРХНИЙ УРОВЕНЬ] Обзор «Что с моей заявкой»                  ← карточки блоков + статус-бейджи
│     ┌─────────────────────────────────────────────┐
│     │ Company details            [Verify]          │  ← Cell + Indicator/Chip
│     │ Board Resolution           [Done]            │
│     │ Personal Identification & Signing  [In progress] ▸ │ ← кликабельный, drill-down
│     │ Business profile / UBO     [Verify]          │
│     │ …                          [PLACEHOLDER-МАРГО Б.1] │
│     └─────────────────────────────────────────────┘
│
├─ [DRILL-DOWN] блок «Personal Identification & Signing»          ← Accordion / под-экран
│     раскрытие → список участников (текущий recipients.map)
│     каждый участник: имя + роли-чипы + под-статусы Подписание / VKYC + «Войти как»
│
├─ [DVU-панель] обратный запрос банка (как сейчас, #34)            ← оставить
│
└─ [ФИНАЛ] когда всё done: счёт открыт, НО «waiting for activation» + IFSC + debit freeze
```

### ВЕРХНИЙ УРОВЕНЬ — обзор блоков заявки (Б.1, Б.3)
- **Сейчас:** дашборд сразу `recipients.map` (плоский список людей). **Должно стать:** сначала обзор блоков заявки.
- **Компонент строки блока:** `Cell` (`title`=имя блока, `contentRight`=статус-бейдж). Блок-контейнер — Stack карточек.
- **Статус-бейдж:** `Indicator` + подпись, или `Chip view`. Цвета по статусу:
  - `Verify` / `In verification` — **сине-серый** (Verifying≠warning, гайд). Подпись «действий не требуется».
  - `In progress` — синий.
  - `Done` / `Completed` — зелёный.
  - `Action required` — оранжевый (только если реально нужно действие клиента).
- **Перечень блоков (Б.1):** подтверждённые из аудио/замысла — Company details, Board Resolution, Personal Identification & Signing, Business profile / UBO. Полный список и статусы помимо «Verify» — `[PLACEHOLDER-МАРГО]` (ОВ-Б1). Федя: вынести список блоков в массив-конфиг, чтобы дополнить по приходу.
- **«Verify» по умолчанию:** пока банк не подтвердил блок (VKYC, company details) — статус «Verify», не «готово» (замысел Б.3). Не оранжевый — спокойный.
- **Состояния:** *loading* — `Spinner`/skeleton; *in progress* — часть блоков Verify/In progress; *all done* — все Done → показываем финал-блок (см. ниже).

### DRILL-DOWN — блок «Personal Identification & Signing» (Б.2)
- **Поведение:** клик по строке блока «Personal Identification & Signing» в обзоре → раскрытие детализации. Два варианта формы (рекомендую первый):
  - **Рекомендую: `Accordion`/`AccordionItem`** — блок раскрывается на месте (in-place), не уводя с дашборда (по гайду «не отвлекать», минимум подмен экрана). Заголовок аккордеона = «Personal Identification & Signing» + сводный статус.
  - Альтернатива: переход на под-экран `/company/identification-signing` — если контента много. На старте — аккордеон.
- **Содержимое (= текущий `recipients.map`, переезжает СЮДА, перестаёт быть корнем):** список участников. Каждый участник (`Cell` или существующий `PersonCard`):
  - Аватар/инициалы.
  - Имя + роли — `Chip`/`ChipGroup` (весь набор ролей, мультироль — как сейчас).
  - **Два под-статуса вместо одного общего шага:** «Подписание: …» и «VKYC: …» (письмо явно требует видеть и подписание, и VKYC по каждому). Сейчас один общий `currentStep` — развести на два прогресс-индикатора (можно компактно: VKYC ✓/⟳ + Подписание ✓/⟳).
  - Действия: `Button` «Войти как {имя}» (как сейчас), «Напомнить» (`view="clear"`). Скрыты, если участник «Готово» или без роли подписанта.
