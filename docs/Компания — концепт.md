## TL;DR

Компания = тот же скелет, что Sole Proprietor, но с тремя принципиальными надстройками:
1. **Мульти-подписант.** Актёры: 2 директора + 1 authorized signatory (AS). Подписант = один из директоров (кейс 2, happy) ИЛИ отдельное лицо, не директор (кейс 3, DVU).
2. **BR-вопросы в анкете.** В бизнес-анкету добавляется блок «BR Questions»: дропдаун подписантов из Probe + ручной ввод AS, PAN автозаполняется из Probe, email/phone каждого подписанта вводятся вручную (для будущих OTP / ссылок / VKYC).
3. **Разрыв на «персональные сессии».** После того как заполнитель (prospect) подтвердил данные компании, система рассылает **ссылку-приглашение каждому подписанту** («Link to all Signatories»). Дальше у каждого — своя персональная мини-сессия: Consents → Aadhaar QR → VKYC → подписание по **DSC-ссылке** (НЕ OTP, как у Sole Proprietor).

Ключевое отличие в порядке: **Aadhaar QR и согласия у Компании — внутри персональной сессии подписанта, перед его VKYC, а НЕ в начале потока** (у Sole Proprietor — в начале, решение Марго 2026-06-15). Это прямо подтверждено разбором Р-8: для SP заполнитель = единственный подписант и Aadhaar кормит pre-fill; для Компании подписанты ≠ заполнитель, их Aadhaar — чистая идентификация перед видео.

Топ открытых вопросов: лимиты/правила числа AS (В-5); двойное подписание BR в кейсе 3 (В-10); DSC — внутри портала или внешняя ссылка (целевое vs текущее); кто именно проходит VKYC (периметр расходится в BRD, A6).

## Решения по 5 вопросам (Денис, обсуждение 2026-06-16)

Разобрали сами (BRD + кейсы Марго), Марго не понадобилась:

1. **Число подписантов — НЕ ограничиваем.** Sole Proprietor — всегда один. Компания — **один и более** (1, 2, 5, 12…); вести может и один человек, и много. Архитектурно — список произвольной длины, не фикс-число.
2. **Загрузка BR = автозаполнение данных, не подпись.** BRD 6.1: «загрузка BR с автораспознаванием данных; чего нет — клиент дозаполняет». OCR вытягивает подписантов/директоров из BR (как PAN/Aadhaar тянут данные). Двойного подписания нет: загрузка ради данных ≠ финальное подписание пакета по DSC.
3. **DSC-подписание — ВНУТРИ портала** (целевое видение «всё в портале», установка Марго / [[v2-target-vision]]). По образцу OTP-подписания Sole Proprietor; реальная интеграция (eMudhra) — инженерный gap, на прототипе заглушка.
4. **Видео проходят ВСЕ подписанты** (дефолт до сбора комментариев команды). BRD 6.1 говорит «только AS, подписавший BR» — сузим, если команда подтвердит. Ложится на мульти-логин: у каждого своя сессия.
5. **Форма BR-вопросов = BRD §6 пункты 1–4, 7** (свой/шаблон банка → PAN компании авто → мин. 2 директора-подписанта из Probe → назначить AS → подтвердить). Пункты 5–6 (governance, место собрания) ОПУСКАЕМ — юр.обвес, не влияет на путь. Развилка п.1: happy-flow = «шаблон банка» (система генерирует BR); «загрузить свой» = DVU-ветка (трек 3).

## 1. Чем Компания отличается от Sole Proprietor

Сравнение по кейсам Марго (кейс 1 = SP happy, кейс 2 = Company happy, кейс 3 = Company + DVU).

| Аспект | Sole Proprietor | Компания |
|---|---|---|
| **Актёры** | 1 — единственный authorized signatory (= proprietor) | 2 директора + 1 AS. Happy (кейс 2): AS — один из директоров. DVU (кейс 3): AS — НЕ директор |
| **Кто заполняет онбординг** | Он же сам (proprietor) | Prospect-заполнитель (один человек ведёт регистрацию и анкету за компанию) |
| **Анкета** | Business Nature questionnaire | Business Nature questionnaire **+ BR questions** (блок выбора подписантов, см. §3) |
| **Подписанты** | Один, заранее известен из PAN (4-й символ `P`) | Несколько, выбираются из **дропдауна Probe + ручной ввод**; вводятся PAN/email/phone каждого |
| **Согласия (Consents)** | В начале / по шагам потока заполнителя | **В персональной сессии каждого подписанта**, перед его VKYC (НЕ в начале) |
| **Aadhaar QR** | Шаг 2, рано (решение Марго 2026-06-15) — кормит pre-fill личных данных заполнителя | **В персональной сессии каждого подписанта**, перед его VKYC (Р-8: чистая идентификация, не pre-fill) |
| **VKYC** | Один проход (сам proprietor) | **Каждый подписант проходит VKYC** в своей сессии. ⚠️ Решение Дениса 2026-06-16: для прототипа делаем «видео проходят ВСЕ подписанты» (дефолт до сбора комментариев команды; BRD 6.1 говорит «только AS, подписавший BR» — сузим, если команда подтвердит). Естественно ложится на мульти-логин: у каждого своя сессия |
| **Подписание** | Application + Declaration по **mobile OTP** | **Board Resolution + Application + Declaration по DSC-ссылке** (Digital Signature Certificate) |
| **Логин** | Один пользователь, одна сессия | **Мульти-логин:** «Link to all Signatories» — ссылка-приглашение всем троим, каждый входит в свою персональную часть сессии компании |
| **Документ-основание** | Consent & Declaration (декларация) | **Board Resolution** (BR): онлайн-шаблон банка ИЛИ загрузка собственного BR клиента |
| **DVU-ветка (кейс 3)** | Возможна (BR-11), запросы документов | Параллельные DVU-таски: проверка данных+документа, проверка **BR и DSC**; запрос доп. документов |
| **Источник данных компании** | Probe42 / GST по PAN proprietor | Probe42 по PAN компании: Entity Name, CIN, адрес, директора, shareholding и т.д. |

**Главные структурные новизны Компании (списком):**
- Мульти-подписант с ролями (директор / AS / UBO).
- BR Questions внутри анкеты (дропдаун подписантов из Probe + ручной AS).
- Разрыв потока на «фаза заполнителя» → «персональные сессии подписантов».
- Aadhaar QR + Consents переезжают ВНУТРЬ персональной сессии (перед VKYC), а не в начало.
- Подписание по DSC, а не по OTP.
- Мульти-логин через ссылки-приглашения.
- Board Resolution как документ-основание (на экране + опция загрузить свой).

## 2. Сквозной поток Компании (рекомендация)

По образцу нашего потока Sole Proprietor (8 шагов в `app/src/ui/v2/steps.ts`), но с разрывом на две фазы. Порядок собран из кейса 2 Марго (happy) + Р-8 (позиция Aadhaar) + BR-12.

### Фаза A — заполнитель (prospect), одна сессия

| # | Шаг | Общее с SP? | Что меняется для Компании |
|---|---|---|---|
| A1 | **Self-Registration** (email-подтверждение, регистрация) | ✅ переиспользуем | Без изменений — тот же поток регистрации |
| A2 | **Доступ к реестрам и PAN** | ✅ переиспользуем | PAN **компании** → тянет данные юрлица из Probe42 (Entity Name/CIN/адрес/директора) |
| A3 | **Бизнес-анкета + BR Questions** | 🟡 расширяем | К BNQ добавляется блок **BR Questions** (см. §3): дропдаун подписантов из Probe, ручной AS, PAN/email/phone каждого |
| A4 | **Подтверждение данных компании** (pre-filled display, edit, request to upload, confirm) | ✅ переиспользуем (BR-22…24) | Тот же экран review: Legal name, имя директора(ов), корресп-адрес, обязательный proof при правке, серый бейдж Pre-filled. Edit поля → DVU (Verifying) |
| — | **РАЗРЫВ: «Link to all Signatories»** | 🆕 новое | После confirm система рассылает ссылку-приглашение **каждому подписанту** (2 директора + AS). Дальше — фаза B у каждого |

> ⚠️ В кейсе 3 (DVU) между A4 и разрывом добавляются: prospect редактирует данные/грузит документ → **параллельный DVU-таск** на проверку данных+документа; затем **загрузка собственного BR** + ручной ввод доп. полей BR → **параллельный DVU-таск** на проверку BR и DSC. См. кейс 3 дословно в приложении.

### Фаза B — персональная сессия КАЖДОГО подписанта (по ссылке)

Кейс 2, блок «Link to all Signatories for the following» — порядок дословный:

| # | Шаг | Источник позиции | Примечание |
|---|---|---|---|
| B1 | **Consents** (согласия) | кейс 2 | Согласия здесь, в сессии подписанта — НЕ в начале (Р-8). Aadhaar-согласие идёт в связке перед QR/видео (BR-21) |
| B2 | **Aadhaar QR** | кейс 2 + Р-8 | QR на фронте → подписант сканирует приложением Aadhaar → UIDAI-согласие → успех. Для Компании — идентификация перед видео (не pre-fill) |
| B3 | **VKYC** (видеоидентификация) | кейс 2 | Каждый подписант проходит свою видеосессию |
| B4 | **Подписание Board Resolution + Application + Declaration по DSC-ссылке** | кейс 2 | DSC (Digital Signature Certificate), НЕ OTP. Целевое — внутри портала (BR-10), текущее у подрядчиков — внешняя ссылка |
| B5 | **View of Internet Bank** (счёт/продукт + инструкции) | кейс 2 | = финальный экран (BR-16) |
| B6 | **Prospect's dashboard** со статусами всего процесса | кейс 2 | Статусы Done / Verifying / Action Required (кейс 3, = BR-15) |

**Где Aadhaar QR и согласия для Компании** (повтор, потому что это ключевое и подтверждённое):
- НЕ в начале потока (как у SP с 2026-06-15), а **в фазе B — персональной сессии каждого подписанта, перед его VKYC**.
- Подтверждено разбором Р-8 (`docs/Вопросы и расхождения.md`): «для компании подписанты ≠ заполнитель, их данные берутся из BR Questions (PAN из Probe + ручной ввод контактов), Aadhaar там — чистая идентификация перед видео + сверка UIDAI с OCR видеосессии».
- Согласие на Aadhaar — перед QR/видео (BR-21, «идёт в связке»).

### Что общее с SP, что новое/меняется (резюме)

- **Общее (переиспользуем как есть):** регистрация+email, экран PAN+согласие на реестры, экран подтверждения данных (review), VKYC-заглушка, дашборд со статусами, финальный экран Internet Bank.
- **Меняется:** анкета (+BR Questions); подписание (DSC вместо OTP); позиция Aadhaar/Consents (в сессию подписанта, не в начало).
- **Новое:** мульти-логин / ссылки-приглашения; экран подписантов (дропдаун Probe + ручной AS); персональная мини-сессия подписанта; Board Resolution на экране + загрузка своего BR; DVU-ветка с проверкой BR/DSC (кейс 3).

## 3. BR Questions form — состав полей

### Дословно из кейса Марго (кейс 2, «BR Questions form»)
- **Name field of the Authorised Signatory** — Dropdown (from Probe) + Manual Input field.
- **Provide PAN** (auto-populated from Probe), **Email**, **Phone ID** of each Signatory.
- See the BR form here — *1D - 6. Operating Instruction Form (BoardResolution) vs role model in Internet Banking*.
- Declaration — same link see above.

### Дополнения из кейса 3 (Company + DVU), 🟠 = клиентские тексты-подсказки
- Name field of the Authorised Signatory — Dropdown (from Probe) + Manual Input field:
  - 🟠 «Please Nominate the Authorised Signatory as per your BR and provide the details»
  - 🟠 «Please enter details of the UBOs who signed the BR»
- PAN field, E-mail field, phone number field **for all signatories** — for future OTP, Signing and VCIP links.
- Upload the own BR confirming the entered fields — 🟠 «BR should be DSC Signed in advance».

> Разбор Бори (В-10): оранжевые фразы — клиентские тексты-подсказки к полям; «BR should be DSC Signed in advance» — предусловие к загрузке собственного BR (DVU параллельно проверяет «BR and DSC»). Уверенность в смысле высокая, в финальности формулировок — средняя.

### Полная форма BR из BRD 6 (справочно, `6-Operating-Instruction-Form-BR-vs-role-model.md`)

«Basic Questions that lead to BR form generation»:

| # | Вопрос для OBO-пользователя | Атрибут BR |
|---|---|---|
| 1 | Provide the Board Resolution. Use Bank's online template or upload your own. Proceed with online template? Yes/No | Yes → п.2; No → No STP → Alert DVU |
| 2 | Provide PAN of your company | PAN, Company Name, Company Address — auto из Probe42 |
| 3 | Choose **at least two directors or company secretary** who will sign the BR | Name, Designation, Email, Phone — Name & Designation валидируются против Probe42 |
| 4 | **Nominate the Authorised Signatory** | Name, Email, Phone — Manual Input |
| 5 | Choose governance for AS changes | Dropdown: 1) Nominated Authorized Official; 2) Decision Pursuant based on BR |
| 6 | Choose Place of meeting the BR was taken | Dropdown: 1) Virtually; 2) By rotation; 3) Address |
| 7 | Confirm the pre-filled Board Resolution | Date, Confirm — авто-таймстамп |

Структура самого Board Resolution (BRD 6): Company Details (Name/CIN/Address — auto, Date&Place — manual); Resolution (банковские продукты со Сбером); Signatory table (Name, Designation, PAN, Contacts для OTP&VCIP, governance); Details & Declaration of Directors/UBOs/AS (Name, Contacts, Signature, Company Seal); сертификационный чекбокс «true and correct copy…».

> ⚠️ Расхождение для решения: кейс Марго даёт минималистичную форму (AS-дропдаун + PAN/email/phone), BRD 6 — расширенную (7 вопросов + governance + place of meeting). Для прототипа фазы «компания» рекомендую идти по **кейсу Марго** (он новее и приоритетнее), а governance/place — как «детали BR» опционально/в DVU-ветке.

## 4. Что переиспользуем из SP, что строим новое

### Переиспользуем из прототипа Sole Proprietor (`app/src/screens/v2/`)
| SP-экран | Для Компании | Изменение |
|---|---|---|
| `SP01Landing` / `SP02Email` / `SP03Register` (регистрация + email) | A1 Self-Registration | без изменений |
| `SP05Pan` (Доступ к реестрам и PAN) | A2 | PAN компании вместо PAN физлица; pull данных юрлица из Probe |
| `SP07Bnq` (бизнес-анкета) | A3 | добавить блок BR Questions |
| `SP06Company` (подтверждение данных) | A4 | переиспользуем целиком (Legal name, имя директора, proof, Pre-filled бейдж) |
| `SPDataConsents` (согласия) | B1 | переносим логику в сессию подписанта |
| `SPAadhaarQr` (Aadhaar QR) | B2 | переносим в сессию подписанта, перед VKYC |
| `SP08PreVcip` / `SP09Vcip` (согласие + видео) | B3 | переиспользуем заглушку видео |
| `SP10Dashboard` (дашборд + статусы) | B6 | + статус «Action Required», мульти-подписантный прогресс |

### Строим новое
- **Мульти-логин / ссылки-приглашения** — механика «Link to all Signatories»: рассылка ссылки каждому из 3 подписантов, вход в свою персональную часть сессии компании. Нет в SP.
- **Экран выбора подписантов** (BR Questions UI) — дропдаун из Probe + ручной ввод AS + поля PAN/email/phone на каждого. Нет в SP.
- **Board Resolution на экране** — вывод BR клиенту + дропдаун «кто подписывает» + опция «загрузить свой BR» (с подсказкой про DSC). Нет в SP (у SP — декларация, не BR).
- **DSC-подписание** — вместо OTP-подписания у SP. Целевое — внутри портала (iframe, BR-10), текущее — внешняя DSC-ссылка.
- **Персональная мини-сессия подписанта** — обёртка Consents → Aadhaar → VKYC → DSC-sign на каждого участника.
- **DVU-ветка кейса 3** — параллельные таски: проверка данных+документа, проверка BR+DSC; запрос доп. документов в кабинет клиента (мост OBU↔DVU, BR-17).

> Замечание Марго (приоритеты 2026-06-10): фазу «Компания» показываем **верхнеуровнево**, «даже с недочётами» — это приоритет 2 после финализации SP + DVU. Демо строим на примере одного подписанта: перешёл по ссылке → видеосессия → подписание.

## Приложение: дословные источники

### Кейс 2 — Company happy flow (Марго, дословно)
**Actors:** Two directors; Authorised signatory is one of the directors.
**Actions:**
- Self-Registration
- Business Nature questionnaire including BR questions
- Pre-filled data display, edit, request to upload document, confirm
- *Link to all Signatories for the following:*
  - Consents
  - Aadhaar QR
  - VKYC
  - Board Resolution, Application and Declaration to be signed by DSC link
  - View of the Internet Bank with account/product details and following instructions
  - Prospect's dashboard with different statuses of the whole process

### Кейс 3 — Company flow with DVU involvement (Марго, дословно)
**Actors:** Two directors; One Authorised Signatory — not the director.
**Actions:**
- Self-Registration
- Business Nature questionnaire
- Pre-filled data display, edit, request to upload document, confirm
- Prospect edit data and upload the doc
- *In parallel* task to DVU to check the data+doc
- Upload own BR + additional fields for BR details manual input
- *In parallel* — DVU task to check the BR and DSC
- *Link to all Signatories for the following:* Consents · Aadhaar QR · VKYC · Board Resolution, Application and Declaration to be signed by DSC link · View of the Internet Bank · Prospect's dashboard with statuses
- *In parallel* DVU requests Prospect to upload additional document
- Prospect's dashboard with statuses: Done / Verifying / Action Required
- View of the Internet Bank with account/product details and following instructions

### BRD-матрица (кто подписывает BR / кто проходит VCIP), `6-Operating-Instruction-Form`
- **Public/Private Company:** Operating instruction = только один Single AS с Admin role в Internet Bank. Подписывают BR: при 5 директорах — минимум 2 директора / Company Secretary (singly) / Managing Director; право подписи валидируется против Probe42. VCIP проходит: **AS, который подписал BR** (но встречи расширяют — см. A6).
- Probe42 directors: Designation (Director / MD / Company Secretary); Flags=0 (Flag=Deactivated при non-filing DIR-3 KYC).

### Document Execution — Board Resolution, `1.2-Company-Onboarding-BANK-perspective` (step 02)
Два пути: (1) Bank-provided BR — скачать шаблон → wet signature → скан → upload → digital attestation (DSC/eMudhra/eSign OTP); (2) собственный BR клиента — upload → IDP-классификация (это валидный BR?) → OCR извлечение полей → валидация содержания → при успехе system-generated links подписантам на eKYC и VKYC.

### OCR-сервис проверки подписи (DSC), `6.1-Detailed-BR-signing`
Возвращает: is signature valid (True/False), signatory full name, date of signing, timestamp, class of signature, issuer, validity starts/ends. Recipients по типам: Partnership/LLP «1+n» (все партнёры); Public/Private Company «1+n» (AS, подписавший BR); Sole Prop «1».
