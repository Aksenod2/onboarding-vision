# Вижн: единая CRM поверх микросервисов

Статус: **вижн** (стратегическое направление, ради которого Денис в команде). Владелец: Денис. Дата: 2026-06-26.
Назначение: единая опора для обсуждений с командой и банком. Не привязка к реализации — задаёт принцип, под который выстраивается архитектура.

---

## Одной фразой

> **Не пользователь подстраивается под архитектуру системы, а архитектура полностью подчинена единому пользовательскому процессу.** Главный объект проектирования — бизнес-процесс. Все сервисы превращаются в незаметные инструменты, которые подключаются только тогда, когда действительно нужны.

---

## 1. Проблема

Сейчас внутренние банковские системы построены на микросервисной архитектуре. Для **разработки** это хорошо: сервисы независимы, масштабируемы, развиваются отдельными командами.

Для **клиента** всё выглядит цельно: единый процесс онбординга, открытие счёта, проверки, подписание — через единый интерфейс. Архитектура скрыта.

Для **сотрудников банка** — обратная картина. Каждый этап обслуживания клиента — отдельный микросервис: Video KYC, комплаенс, открытие счёта, работа с документами, управление продуктами и десятки других. Менеджер не работает в одной системе — он постоянно переключается между десятками рабочих мест, у каждого свой интерфейс, навигация, логика, UX-паттерны и даже стиль.

**Последствия:**
- растёт время обучения сотрудников;
- при смене роли приходится заново изучать интерфейсы;
- единый UX практически недостижим — над сервисами работают разные команды; даже при наличии дизайн-системы со временем расходятся поведение, терминология и логика взаимодействия.

## 2. Смена модели мышления: зигзаг → линия

Сейчас клиентский путь = **зигзагообразная линия**, проходящая через каждый микросервис. Ощущение, что клиент постоянно перемещается между системами.

Новая модель: клиентский путь — **одна непрерывная горизонтальная линия** (главный бизнес-процесс). Микросервисы перестают быть **этапами** пути — они становятся **сервисами, которые подключаются к пути в момент, когда нужны**, и отключаются, выполнив задачу.

- Нужна идентификация → подключается Video KYC.
- Нужна проверка регулятора → подключается комплаенс.
- Нужно открыть счёт → подключается сервис открытия счетов.

> **Не клиент проходит через микросервисы, а микросервисы обслуживают единый клиентский процесс.**

## 3. Целевая модель

Объединить часть микросервисов в **единую CRM-платформу** — единую точку входа для всех сотрудников банка. Цель — **не отказ от микросервисов**: они продолжают существовать как независимые бизнес-сервисы. Меняется только **способ взаимодействия пользователя с системой**.

- Вместо десятков рабочих мест — одна CRM, в которой работают все сотрудники.
- Интерфейс общий. Меняются только **роли, права доступа и доступные функции** (RBAC).
- Переход между подразделениями не требует заново изучать систему — знакомый интерфейс, меняется функциональность.

Нет необходимости выбирать между микросервисами и единой CRM — совмещаем сильные стороны обоих: микросервисы независимы и масштабируемы; поверх — единая CRM как точка входа; слой оркестрации скрывает техническую сложность интеграции.

## 4. Слой оркестрации и роль AI (честная формулировка)

Раньше единый слой интеграции между CRM и микросервисами был дорогой инженерной задачей. Сегодня значительную часть могут взять на себя агентные/оркестрационные системы: работа с API разных сервисов, преобразование данных, маршрутизация запросов, отслеживание ошибок, контроль последовательности операций, адаптация к изменениям интерфейсов сервисов. Это не отменяет сложность разработки, но снижает стоимость поддержки.

**Граница ответственности (важно для зрелости вижна):**
- **Фундамент — детерминированные контракты** (события/API между процессом и сервисами). Критический путь (открытие счёта, комплаенс-решение, аудит) идёт по контрактам, не «по импровизации».
- **AI — на краях**: маппинг данных, обработка ошибок, адаптация к смене UI сервисов, ассист оператору. Не несущая конструкция критических транзакций.
- Так удешевляем интеграцию, **не жертвуя** воспроизводимостью, аудитом и регуляторной предсказуемостью.

## 5. Индустриальный контекст (это зрелый mainstream, не эксперимент)

Идея имеет имена и живые реализации — словарь для защиты перед командой/банком:
- **Unified Agent Desktop / Agent Workspace** — единое окно сотрудника поверх многих систем.
- **Orchestration layer · Process orchestration (BPM / case management)** — «сервис подключается на шаге процесса».
- **BFF (Backend-for-Frontend) + micro-frontends** — техоснова «единый фронт над N сервисами».
- **Composable enterprise** (Gartner).
- **Доказательство рынком:** Fenergo, nCino, Salesforce Financial Services Cloud — единые платформы поверх сервисов KYC/онбординга/CRM (см. `docs/CRM — research рынка.md`). Лидеры уже так работают.

## 6. Риски и как отвечаем

| Риск | Ответ |
|---|---|
| **AI как «магический клей»** — недетерминизм, аудит, латентность, регуляторика на критическом пути | Контракты — фундамент; AI только на краях (см. §4). Не обещать AI на несущей конструкции |
| **Conway's law** — единый UX держится не дизайн-системой, а орг.моделью | Сильная **платформенная команда + контракты**; иначе расхождения вернутся на уровне CRM |
| **«Линия» vs граф** — процесс не линеен (ветвления/откаты/параллельные проверки) | Линия — в коммуникации; в реализации — **машина состояний/граф**. Не обещать линию там, где состояния |
| **Единая точка входа = единая точка отказа** | Заложить деградацию: лёг сервис — окно живёт (graceful degradation) |
| **Регуляторное разделение обязанностей** (segregation of duties, four-eyes) | RBAC жёстко разделяет права; единый UI ≠ единые полномочия |

## 7. Принципы (включая «не нагородить необратимое»)

1. **Бизнес-процесс — главный объект проектирования.** Архитектура — скрытая внутренняя реализация.
2. **Контракты до клея.** Стандартизованные события/API между процессом и сервисами — раньше, чем оркестрация и AI.
3. **Обратимость старта.** Начинать с решений, которые **можно подвинуть**. Необратимые решения (жёсткие схемы данных, единый стор на всё, выбор оркестратора, vendor-lock) — откладывать до проверки на практике. Цель — не залочиться рано.
4. **Пилот до масштаба.** Доказывать философию на малом (1–2 домена), потом расширять. Работающее малое > слайды.
5. **Единый UX через роли, не через переписывание сервисов.** Сервисы остаются; меняется слой взаимодействия.
6. **Удержание консистентности — процессом, а не надеждой.** Гейты/правила работы команды (`docs/CRM — правила работы команды (гейты).md`).

## 8. Связь с прототипом — это уже пилот вижна

Текущая работа CRM+DVU = **вижн в миниатюре**, проверяемый на практике:
- ядро client-360 = «клиент/процесс как главный объект»;
- CRM- и DVU-домены = «сервисы-инструменты, подключаемые по роли через пресеты»;
- адаптер событий = «контракт», а не зигзаг (сервис обслуживает процесс);
- RBAC + единый UI = «меняются роли/права, не интерфейс»;
- spike «одно окно + консистентность + масштаб через пресеты» = доказательство концепции для N сервисов.

> Если на 2 доменах докажем модель — это рабочий аргумент для большой идеи. Защищаем большое через работающее малое.

## 9. С чего начать — открытый раздел (решаем позже)

Принцип: **не нагородить то, что потом не сможем подвинуть.** Кандидаты на старт (черновик, на обсуждение):
- начать с **контракта процесса** (события: сервис подключился/отработал/вернул результат) — обратимо, фундаментально;
- продолжить **пилотом единого окна** (CRM+DVU spike) — уже в работе;
- **отложить** необратимое: выбор оркестратора/шины, единую модель данных на все сервисы, тяжёлую AI-оркестрацию — до проверки на пилоте.

Конкретную последовательность зафиксируем отдельно, после обсуждения.
