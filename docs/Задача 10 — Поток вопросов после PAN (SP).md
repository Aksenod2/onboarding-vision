# Задача 10 — Поток вопросов после PAN (Sole Proprietor)

> BA-проработка потока сбора данных ПОСЛЕ ввода PAN для роли **Sole Proprietor (ИП)**, рынок Индия.
> Источники: [[BNQ — Business Nature Questionnaire]] · [[BNQ Template A — Sole Proprietorship]] · [[Mock-модель данных]] · [[03 — Атрибуты и документы]] · [[Спецификация экранов v2]] · [[Вопросы и расхождения]] · [[01 — Процесс онбординга]] · `app/src/mock/v2/{seed,types}.ts`.
> Контекст ревью: Марго, 2026-06-08.

## TL;DR
- После PAN банк **синхронно фоном** тянет данные из внутренних источников (PAN-база, Probe42, CKYC/CERSAI, OFAC, Stop-42) — клиенту источники НЕ раскрываются (кроме CKYC).
- Клиенту показываем **два типа вопросов**: (а) предзаполнено из реестра → подтвердить; (б) не подтянулось → ввести вручную.
- Проработано **22 атрибута/вопроса**. Открытых вопросов к аналитику — **9**.
- Рекомендуемый порядок: **скрининг → подтверждение данных компании (SP-06) → BNQ Q1–Q11 (SP-07) → consent перед VCIP (SP-08)**.

## ⚠️ Комплаенс-правило формулировок (для прототайпера)
В текстах, **видимых клиенту**, нельзя называть **Probe42** и **PAN-базу** как источники. Допустимо:
- «Мы определили…», «По нашим данным…», «Мы нашли вашу запись в **CKYC**…» (CKYC раскрывать можно).
- Колонка «Откуда» в таблице ниже — **внутренняя**, в UI не выводится.

## Состав источников (внутреннее понимание)

| Источник | Что даёт для Sole Prop | Раскрываем клиенту? |
|---|---|---|
| **PAN-база** (NSDL/Income Tax) | валидация PAN, тип держателя (4-й символ `P` = физлицо/Proprietor), имя владельца | ❌ нет |
| **Probe42** (MCA + P&L + GST-агрегатор) | tradeName, дата начала деятельности, адрес, индустрия/сегмент, Net Revenue, ВЭД-активность | ❌ нет |
| **CKYC / CERSAI** | CKYC ID, ранее верифицированный KYC-профиль владельца | ✅ да |
| **OFAC / Sanctions** | санкционный скрининг (фон, результат клиенту не показываем детально) | ❌ нет |
| **Stop-42** | проверка владельца на adverse match (фон) | ❌ нет |
| **Клиент** | то, что реестры не дают: tax residency, PEP, кредит-интерес, страны-партнёры, IEC | — |

> Особенность Sole Prop: PAN владельца = PAN бизнеса; **нет** CIN/MOA/Board Resolution; вместо них **Udyam + GST**. Один владелец = один подписант → per-person логика BNQ схлопывается в одно лицо (нет multi-signatory / Board Resolution экрана).

## Открытые вопросы к живому аналитику банка

- **OQ-1 (FATCA).** В таблице BNQ FATCA отсутствует (есть PEP, residency), но в [[03 — Атрибуты и документы]] и на встрече упоминался. **На каком шаге для Sole Prop собирается FATCA и какие поля?** (= В-8)
- **OQ-2 (состав Probe42 для Sole Prop).** Какие именно поля Probe42 реально доступны по **ИП** (а не по компании)? У Sole Prop нет CIN/MCA-записи — подтягиваются ли tradeName, Udyam, GST, P&L (Net Revenue), ВЭД так же надёжно, как для Private Limited? Где fallback на ручной ввод обязателен?
- **OQ-3 (что точно приходит из CKYC).** Какие поля владельца возвращает CKYC и можно ли ими предзаполнять (имя, адрес, дата рождения)? Это единственный раскрываемый клиенту источник — важно для формулировок.
- **OQ-4 (Stop-42).** «Стоп-42» по владельцу — синхронная фоновая проверка или клиенту что-то показываем при match? (= В-6)
- **OQ-5 (Vintage / commencementDate).** Для Sole Prop дата «начала деятельности» — из какого источника приоритетно (Udyam? GST? Probe42 P&L)? Если расходятся — какая выигрывает?
- **OQ-6 (Business Licence trigger).** Для каких именно индустрий/сегментов из списка Q1 требуется загрузка Business Licence у ИП? Нужен точный whitelist индустрий.
- **OQ-7 (Tax Residency опции).** Подтвердить актуальность 3 опций Q4 (включая формулировку про «Foreign national of RF not on employment VISA») для Sole Prop.
- **OQ-8 (Net Revenue источник для ИП).** AOC-4 — форма для компаний; у ИП P&L подаётся иначе. Откуда Net Revenue для Sole Prop в Probe42 и насколько покрыто? (влияет, задаём Q6 или нет)
- **OQ-9 (порядок consents).** На каком шаге фиксируются Aadhaar / Data Principals / KMP Confirmation — внутри SP-07 при сборе данных лица или отдельным экраном? (формат consent-событий — Р-6) ⚠ Частично закрыт демо 2026-06-10: Марго подтвердила принцип «согласия поступательно по шагам, по мере ввода данных» (не пакетом в начале); точный шаг для Aadhaar/KMP не оговаривался. См. [[Разбор встречи — Демо макетов 2026-06-10]].
- **OQ-6 дополнение (2026-06-10, демо):** реплика «business license — у Sole Prop свои документы, скорректируем» (часть контекста потеряна с чанком 2 транскрипта) — уточнить у Дениса, что корректируем.

## Рекомендация по порядку экранов/вопросов после PAN (Sole Proprietor)

```
SP-05  Ввод PAN + авто-скрининг (фон)
        Клиент видит прогресс: PAN ✓ → проверки ✓ → «нашли запись в CKYC» → данные подтянуты
        (источники Probe42/PAN-база НЕ называем)
   │
SP-06  Подтверждение данных компании
        Сначала ПП-поля (B1, B3–B6) с бейджем «из реестра» → подтвердить
        Затем РВ-поля (B7 корресп.адрес если нужно, B8 cheque book)
        Edit-режим: правка реестрового поля → upload proof → предупреждение про DVU
   │
SP-07  BNQ — линейный поток Q1→Q11
        Block 1 (риск): Q1 индустрия(ПП/РВ + licence) → Q2 vintage(скрыт если есть) →
                        Q3 residency(скрыт если есть) → Q4 tax residency(РВ) → Q5 PEP(РВ)
        Consents по ходу при сборе данных лица: Aadhaar, Data Principals, KMP
        Block 2 (доп): Q6 net revenue(скрыт если есть) → Q7 кредит(РВ) →[Yes]→ Q8 сумма →
                        Q9 ВЭД(ПП/РВ) →[No: skip]→ Q10 страны → Q11 IEC(disclaimer + upload)
        Риск-скор НЕ показываем нигде.
   │
SP-08  Consent перед VCIP (Data Accuracy)
   │
SP-09  VCIP + подписание (один владелец → один сеанс)
```

**Принципы порядка:**
1. **Сначала всё, что подтянулось (ПП), потом ручной ввод (РВ)** — снижает когнитивную нагрузку: клиент видит, что система «уже многое знает».
2. **Подтверждение данных компании (SP-06) — до BNQ.** BNQ Q1–Q3/Q6/Q9 опираются на уже подтверждённые реестровые поля; если клиент их подтвердил/поправил, эти BNQ-вопросы либо скрываются (vintage, residency, net revenue), либо предзаполняются (индустрия, ВЭД).
3. **Скрытие вопросов при наличии данных** (Q2, Q3, Q6) — ключевой UX BNQ: не задаём то, что реестр уже дал.
4. **Ветвление только в Block 2** (Q7 кредит, Q9 ВЭД) — Block 1 линеен Q1→Q5.
5. **Фоновые триггеры** (Alert DVU при нерезиденте Q3 / PEP Q5 / IEC Q11; Alert CRM при кредите Q7) — клиенту не показываем, идут в бэк.

## Результаты для прототипа (мэппинг на текущую модель)
- Таблица согласуется с `businessFieldSources` (SP-06): `tradeName/gstin/udyam/commencementDate/registeredAddress/companyResidency = registry` (ПП), `pan/industry/segment = manual` (РВ/правка).
- BNQ-массив в `seed.ts` уже размечен `source: available|not_available` — это и есть переключатель ПП vs РВ для каждого Qn. Маркеры `triggered: 'DVU'|'CRM'` = фоновые алерты (не показывать клиенту).
- `risk.shownToClient: false` — соблюдать на всех экранах.
