# UX-концепт «процессного окна» — ось процесса (process spine) над объединённым окном

G3-концепт ДО кода. Денис смотрит КАРТУ/СХЕМЫ, не финальный визуал. Эстетика не финалится.
Стек: MUI v5, тема `app/src/rm/theme.ts` (оранжевый акцент RM), semantic-палитра MUI. Десктоп-first, внутренний инструмент.
Автор: Ульяна. Дата: 2026-06-26.
Вход: `docs/Вижн — единая CRM поверх микросервисов.md`, `docs/ds/crm-dvu-объединённое-окно.md` (РАЗВИВАЕМ его, не противоречим), `docs/Карта стыковок онбординг × CRM × DVU.md`, `docs/CRM — research рынка.md`.

## TL;DR

Добавляю к объединённому окну **третью ось — линию процесса (spine)**: горизонтальную полосу узлов под контекстом клиента, которая показывает «где клиент в пути и какой сервис активен сейчас». Spine — это **визуализация машины состояний** заявки (этапы 1–11 из карты стыковок), а не выдуманная прямая. Сервис (VKYC / комплаенс / открытие счёта) **разворачивается прямо в рабочей области окна inline-панелью**, отработал — свернулся в узел, линия идёт дальше; пользователь не уходит в другую систему и не теряет контекст клиента (полоса B). Две оси работают вместе: **процесс (где клиент) × роль (чьими глазами)** — на пересечении узел×роль определяет, что показать в рабочей области и какие действия активны. Ядро объединённого окна (A-shell, B-client-360, C-workspace, D-actions, пресет-паттерн, RBAC, four-eyes) — **не трогаю**, spine встаёт новой полосой между B и C и управляет тем, ЧТО монтируется в C.

Три тезиса в новой оптике: **одно окно** = сервис разворачивается inline, не уводит → §2; **консистентность** = spine + контекст B переживают и смену узла, и смену роли → §3; **масштаб** = новый сервис = новый тип узла + inline-рендерер на тех же UI-примитивах → §5.

Жёсткое NB к вижну: «линия» — для коммуникации и навигации; под ней **граф состояний** (ветвления/откаты/параллельные DVU). Spine честно показывает это ветками и бейджами, не врёт прямой (риск из вижна §6).

## 0. Что нового по сравнению с объединённым окном

| | Объединённое окно (текущее) | + Процессное окно (это расширение) |
|---|---|---|
| Главный навигатор | role-switcher в шапке (смена оптики) | **+ spine** под контекстом (смена ШАГА процесса) |
| Ось UX | одна: **роль** | две: **процесс × роль** |
| Где живёт сервис | абстрактно «в workspace» | **узел spine → inline-панель в рабочей области** |
| Что в зоне C | пресет роли целиком | **активный шаг/сервис**, выбранный на spine, в рамке пресета роли |
| Связь с картой стыковок | неявная | spine = прямая визуализация 11 этапов + точек подключения CRM/DVU |

Ничего из объединённого окна не отменяется. Spine — **дополнительный слой навигации внутри зоны workspace**, ядро (A/B/пресеты/RBAC/four-eyes) сохраняется как есть.

## 1. Анатомия экрана = ядро client-360 + spine + рабочая область + действия

Зоны объединённого окна остаются (A шапка [S], B контекст клиента [S], C рабочий стол [D], D действия [D], E навигация роли [D, опц.]). **Новое:** между B и C появляется зона **P — линия процесса (spine) [S по структуре, D по содержимому]**, а зона C переосмысляется как **сцена активного шага**.

| Зона | Тег | Что внутри | MUI-каркас |
|---|---|---|---|
| **A. Шапка** | **[S]** | продукт · поиск по PAN · role-switcher · язык · аккаунт | `AppBar`+`Toolbar` (из объед. окна) |
| **B. Контекст клиента (client-360)** | **[S]** | кто открыт: компания, тип (chip), PAN/CIN, статус Application, мини-сводка | `Paper` sticky (из объед. окна) |
| **P. ЛИНИЯ ПРОЦЕССА (spine)** ⭐ | **[S]-структура** | горизонтальная цепочка узлов-этапов заявки: Вход → PAN/Probe → Опросник → Данные → Подписанты/BR → Идентификация(VKYC) → Подписание → Открытие счёта → Готово. Текущий узел подсвечен. Узлы = состояния (см. §4) | `Stepper`(нелинейный, `nonLinear`) ИЛИ кастом-полоса на `Box`+`Chip`+коннекторы (см. развилку) |
| **C. Рабочая область (сцена шага)** ⭐ | **[D]** | разворачивается **выбранный узел spine** глазами текущей роли: либо сводка шага, либо **inline-сервис** (VKYC/комплаенс/счёт), либо карточка задачи DVU. Единые UI-примитивы (§5) | контейнер-слот; рендерер = f(узел, роль) |
| **D. Панель действий** | **[D]** | действия для (узел × роль): напр. узел VKYC × DVU = «Назначить F2F / Одобрить / Эскалировать» с four-eyes-гардом | `Paper` правая колонка (из объед. окна) |
| **E. Навигация роли** | **[D, опц.]** | как было — «свои» разделы роли. Может схлопываться в табы | `Drawer`/`Tabs` (из объед. окна) |

**Ключевая механика:** клик по узлу spine → в зоне C монтируется рендерер этого узла; **контекст B и сама spine не перерисовываются** (как смена роли в объед. окне не трогает A/B). Переход по узлам = навигация ВНУТРИ окна, не переход в другую систему.

## 2. Паттерн «сервис в окне» — единый для всех сервисов

**Решение по умолчанию: INLINE-ПАНЕЛЬ в зоне C (не drawer, не модалка).**

Сервис (VKYC / комплаенс-скрининг / открытие счёта / DVU-проверка) при активации узла **разворачивается на месте рабочей области** — как вложенная карта-панель внутри C, в рамке пресета роли. Действия сервиса уходят в зону D (общая панель действий), не плодят свой набор кнопок.

### Жизненный цикл сервиса на узле

```
   узел NEXT          узел ACTIVE                 узел DONE
   (свёрнут)   ──▶   разворачивается inline  ──▶  сворачивается обратно в узел
   ○ серый          ◉ панель в зоне C            ● зелёный + сводка-строка
                    действия → зона D            «отработал, результат N»
                    статус → бейдж на узле
```

1. **Свёрнут (next/not-started):** узел на spine = маленький кружок-чип. Зона C его не показывает (или показывает заглушку «шаг ещё не начат»).
2. **Развёрнут (active):** клик/авто-фокус на активном узле → сервис рендерится inline-панелью в C на единых примитивах (§5). Это **не iframe чужой системы** — это наш рендерер над контрактом события сервиса (`crm/types/events.ts`).
3. **Свёрнут после (done):** сервис вернул результат → панель сворачивается, узел красится в «готов» (●), в C при повторном клике — **read-only-сводка** результата. Линия идёт к следующему узлу.

### Почему inline-панель, а не drawer/модалка (обоснование)

| Вариант | За | Против | Вердикт |
|---|---|---|---|
| **Inline-панель в C** ✅ | контекст B + spine + действия D **всегда видны рядом**; сервис «внутри пути», не поверх него; десктоп-плотность ок | при очень сложном сервисе C может быть тесной | **ПО УМОЛЧАНИЮ.** Прямо отвечает вижну «сервис разворачивается В окне, не уводит» |
| Drawer (правый) | для глубокого фокуса; не ломает C | **перекрывает зону D-действия** и часть spine → теряется «где клиент»; ощущение «ушёл вбок» | только для длинного вспом-контента (история/лог), не для основного сервиса |
| Модалка | максимальный фокус | **разрывает контекст** (B и spine спрятаны под backdrop) — прямо противоречит вижну «не уводить» | **запрет** для шага процесса. Допустима лишь для атомарного подтверждения (four-eyes confirm) |

**Единство:** все сервисы рендерятся одним контейнером `ServiceStage` с одинаковой шапкой (название сервиса + статус-chip + кнопка «свернуть в сводку»), телом из примитивов §5 и действиями в D. Команда VKYC и команда комплаенса **не рисуют свои окна** — они дают данные/контракт, оболочку даёт ядро. Это ответ на Conway's law из вижна §6: UX держится оболочкой `ServiceStage`, а не дисциплиной каждой команды.

## 3. Две оси UX: процесс × роль

Spine = ось **«где клиент»** (горизонталь, этапы). Role-switcher = ось **«чьими глазами»** (из объед. окна). На пересечении **узел × роль** ядро решает, что показать в C и что в D.

### Матрица узел × роль (что монтируется в рабочую область)

Строки — узлы spine (этапы карты стыковок), колонки — роли. Ячейка = что видит роль на этом узле.

| Узел spine \ Роль | Sales/Менеджер | Оператор КЦ | DVU-офицер | Compliance (задел) |
|---|---|---|---|---|
| **Вход / self-reg** (эт.1) | лид-карточка, источник | «брошена?» → возобновить | — (нет задачи) | — |
| **PAN → Probe42** (эт.2) | обогащённый профиль | поиск по PAN, история | алерт если Probe/CRILC/OFAC fail | обзор скрининга |
| **Опросник** (эт.3) | сигнал кредита → оффер | read-only | DVU-задача если IEC/High-Risk | риск-категория |
| **Данные компании** (эт.4) | read-only | read-only | **DVU OBO** если ручная правка reg-поля | — |
| **Подписанты + BR** (эт.5) | состав (read) | read-only | **DVU** если own-BR / правка состава | — |
| **Идентификация VKYC** (эт.8) | статус (read) | мониторинг | **selfVKYC fail → F2F/meeting** (four-eyes) | High-Risk апрув |
| **Подписание DSC** (эт.8) | read | read | mismatch PAN → алерт | — |
| **Открытие счёта** (эт.9) | оффер/сделка | read | KYC auto-fail → алерт | финальный комплаенс |
| **Готово / счёт** (эт.9) | сделка closed-won | — | — | реестр решений |
| **Догрузка документа** (эт.11) | — | подтолкнуть клиента | **DVU resolve-петля** | — |

Читать матрицу так: **spine один и тот же для всех ролей** (этапы заявки объективны), но **рендерер ячейки и набор действий — функция роли**. Менеджер на узле VKYC видит «статус, read-only»; DVU на том же узле видит «задачу с four-eyes и действиями назначения». Это прямое продолжение «один клиент — разные проекции» из объед. окна §3, только теперь проекция = **f(клиент, узел, роль)**, а не f(клиент, роль).

### Как spine + RBAC-пресет работают вместе

```
   role-switcher (ось РОЛЬ) ──┐
                              ├──▶  ядро выбирает: presets[role]  (workspace-оболочка, действия, nav)
   spine-узел (ось ПРОЦЕСС) ──┘                          +
                                    nodeRenderer = presets[role].render(activeNode)
                                                          │
                                                          ▼
                              зона C = оболочка пресета роли, внутри = рендерер активного узла
                              зона D = действия пресета, отфильтрованные под (узел × RBAC × four-eyes)
```

- **RBAC решает доступность**, не наличие узла. Узел VKYC видят все роли (он есть в пути), но действия «Одобрить/F2F» активны только у DVU; у Sales тот же узел — read-only-сводка. Совпадает с вижном §6 «единый UI ≠ единые полномочия».
- **Если у роли нет задачи на узле** — не пустой экран-ошибка, а приглашение: «У роли Sales по узлу VKYC нет действий — статус только для просмотра» (как empty-паттерн объед. окна §6).
- **Пресет получает +1 функцию `render(node)`** — это единственное расширение пресет-контракта из объед. окна §5. Реестр пресетов не ломается.

## 4. Состояния узла линии (semantic-палитра MUI)

Шесть состояний узла — визуальный язык spine. Цвета берём из **semantic-палитры MUI** (`success/warning/error/info/grey` + `primary` для активного), не хардкод. Форма + цвет + иконка — чтобы не опираться только на цвет (a11y).

| Состояние | Что значит | Глиф/форма | Цвет (MUI semantic) | Бейдж/чип |
|---|---|---|---|---|
| **не начат** (not-started) | шаг впереди, недоступен | ○ пустой кружок, тонкий контур | `grey.400` / `text.disabled` | — |
| **активен** (active) | клиент здесь сейчас, сервис может разворачиваться | ◉ залитый, акцент-кольцо | `primary.main` (оранжевый RM) | пульс-кольцо (reduced-motion: статичное) |
| **ждёт** (waiting) | отдан сервису/клиенту, ждём ответ (selfVKYC идёт, клиент грузит док) | ◐ полу-залитый + часики | `info.main` (сине-серый) | chip «ждём клиента/сервис» |
| **готов** (done) | шаг пройден, результат есть | ● залитый + ✓ | `success.main` | — (свёрнут в линию) |
| **требует действия** (action-required) | нужна рука оператора/DVU (DVU-задача, догрузка, four-eyes-апрув) | ◉ + «!» бейдж | `warning.main` (оранжево-янтарный) | chip «требует действия» + счётчик |
| **ошибка** (error/blocked) | авто-проверка провалена / сервис лёг / откат | ◉ + «×» | `error.main` (красный) | chip «ошибка: Probe недоступен» + «Повторить» |

NB к гайду объед. окна: оранжевый RM — это `primary` (активный узел), а `warning` (action-required) — янтарный, отличимый оттенок. На демо проверить, что активный (primary-оранжевый) и action-required (warning-янтарь) не сливаются — при близости развести формой (◉ vs ◉+!) и подписью, не только цветом. (Перекликается с клиентским правилом «оранжевый зарезервирован», но это RM-сторона, своя палитра.)

### Коннекторы между узлами (тоже несут состояние)

```
● ━━━━ ●        пройденный участок: сплошной success
◉ ━ ━ ○        от активного к next: пунктир grey (ещё не пройдено)
●     ╲
       ◉       ВЕТКА (DVU/exception): узел уходит вбок — параллельная DVU-проверка,
       │       не на главной линии. Возврат на линию = resolve.
       ●━━▶
```

Ветка решает риск вижна §6 «линия vs граф»: exception-проверки DVU (этапы 2/4/5/8/11) рисуются **ответвлением от главной линии**, не ломая её. Откат (вернули на доработку) = стрелка назад по линии + узел красится в action-required.

## 5. Единые UI-примитивы поверх разных сервисов

Чтобы UX не расползался между командами/сервисами (Conway's law, вижн §6) — **5 примитивов**, из которых собран любой inline-сервис в зоне C. VKYC, комплаенс, счёт — все рендерятся ими. Команда сервиса даёт данные, не рисует свой UI.

| Примитив | Что это | MUI-основа | Где применяется |
|---|---|---|---|
| **`TaskCard`** карточка задачи | заголовок + SLA + назначенный + статус-chip + приоритет | `Card`/`Paper` + `Chip` + `LinearProgress`(SLA) | DVU-задача, догрузка, эскалация |
| **`ServiceForm`** форма | поля ввода/правки (reg-поле, причина отклонения, выбор F2F-слота) | `TextField`/`Select`/`RadioGroup` (size small, из rmTheme) | правка данных, причина решения, опросник-фрагмент |
| **`StatusBlock`** статус | состояние шага: chip + сводка результата (selfVKYC: ✗, Probe: ✓) | `Chip` + `Typography` + `Alert`(info сине-серый) | результат любого сервиса, заголовок панели |
| **`Timeline`** таймлайн | хронология попыток/решений/событий узла (audit trail, обязателен для DVU по BRD) | `@mui/lab Timeline` или `List`+коннектор | лог решений DVU, попытки VKYC, история догрузок |
| **`ActionBar`** действия | контекстные кнопки → **всегда уходят в зону D**, не внутрь панели | `Stack` из `Button` + `Tooltip`(four-eyes-гард) | единый набор действий узла×роли |

**Контейнер-оболочка `ServiceStage`** (обёртка любого сервиса в C):
```
┌─ ServiceStage ──────────────────────────────────────┐
│  [StatusBlock]  Personal Identification (VKYC)  [chip]│  ← шапка: имя сервиса + статус
│  ─────────────────────────────────────────────────── │
│  [TaskCard]   задача · SLA · назначен                 │  ← тело: из примитивов
│  [StatusBlock] selfVKYC ✗ · PAN ✓                     │
│  [Timeline]   попытки / эскалация / решения           │
│  ─────────────────────────────────────────────────── │
│  (действия [ActionBar] вынесены в зону D справа)      │
└──────────────────────────────────────────────────────┘
```

Любой новый сервис = новый набор данных под эти 5 примитивов + регистрация типа узла. Оболочка, шапка, состояния, микротексты — общие. **Это и есть масштаб без расползания UX** (вижн §3, §5).

## 6. Состояния и удобство (контекст не теряется)

Правило объед. окна: loading / empty / error / disabled / success — не оставлять дыр. Теперь на **двух уровнях**: spine (полоса) и узел (содержимое C).

| Уровень | loading | empty | error |
|---|---|---|---|
| **Spine (полоса P)** | skeleton-цепочка узлов (серые кружки + коннекторы-плейсхолдеры) | «Заявка не начата» — один узел «Вход» активен, остальные not-started | если не пришла машина состояний: полоса в `error`-состоянии + «Не удалось загрузить процесс. Повторить» — но **B остаётся** |
| **Узел / зона C** | skeleton inline-панели (как skeleton workspace в объед. окне) | узел без задачи для роли = приглашение «нет действий, только просмотр» (не ошибка) | сервис лёг = `StatusBlock` error + «Повторить» **внутри C**; spine-узел красится error; **B и spine живут** (graceful degradation, вижн §6) |

**Контекст клиента и процесса не теряются — гарантии:**
- Смена узла spine → перерисовывается **только зона C** (+ D). B и сама spine не мигают. Лёгкий fade C ≤150 мс (как смена роли в объед. окне §4).
- Смена роли → spine остаётся (этапы объективны), меняется рендерер ячейки и действия. **Клиент + позиция в процессе сохранены.**
- Переход по узлам **не уводит из окна** — нет навигации в чужую систему, нет потери B. Это буквальная реализация вижна «не прыгать между системами».
- **Deep-state:** активный узел и роль — в общем сторе (`ProcessContext` рядом с `SelectedClientContext`). Перезагрузка/возврат → восстанавливается и клиент, и узел, и оптика.
- **Graceful degradation:** лёг один сервис → его узел error, остальная линия и контекст работают (вижн риск «единая точка отказа»).

## 7. Развилки для Дениса (форма — за тобой решение)

| # | Развилка | По умолчанию (моя рекомендация) | Альтернатива |
|---|---|---|---|
| **1** | **Spine горизонтальный сверху vs вертикальный сбоку** | **Горизонтальный** под контекстом B (схема A). Путь читается слева-направо как «линия» вижна; экономит вертикаль для C; совпадает с метафорой «непрерывная горизонтальная линия» (вижн §2) | Вертикальный spine в зоне E (слева): вмещает больше узлов/подэтапов, но «линия» теряет горизонтальную метафору и ест ширину C. Беру про запас для очень длинных процессов |
| **2** | **Сервис: inline-панель vs drawer vs модалка** | **Inline-панель в C** (§2) — «в окне, не уводит» | Drawer — только для вспом-контента (длинный лог/история). Модалка — запрет для шага, лишь для атомарного four-eyes-confirm |
| **3** | **Spine как `Stepper`(MUI) vs кастом-полоса** | **Кастом-полоса** на `Box`+`Chip`+коннекторы: MUI `Stepper` не умеет 6 семантических состояний узла, бейджи, ветки DVU из коробки | `Stepper nonLinear` — быстрее собрать spike, но придётся переопределять `StepIcon` под состояния §4. Возможен гибрид: Stepper-каркас + кастомный StepIcon |
| **4** | **Ветки DVU: на spine (ответвления) vs только в C** | **На spine ответвлением** (§4) — видно, что подключилась exception-проверка, честно к «граф vs линия» | Прятать ветку в C (линия всегда прямая) — чище визуально, но врёт про параллельность DVU. Не рекомендую |
| **5** | **Гранулярность узлов: 9 крупных vs 11 этапов карты vs под-узлы** | **9 крупных узлов** (как схема A), под-этапы (consents→aadhaar→vkyc→dsc) раскрываются ВНУТРИ C, не на spine | Все 11/14 на spine — перегруз полосы. Под-spine внутри узла — задел |
| **6** | **Активный узел: авто-фокус vs ручной клик** | **Авто-фокус на активный** при открытии клиента (сразу видно «где он»), клик по другим узлам — навигация | Всегда ручной выбор — лишний клик к тому, что и так «сейчас» |
| **7** | **Spine: общий для ролей vs роль-специфичный** | **Общий** (этапы заявки объективны) — это и есть консистентность; роль меняет рендерер ячейки, не набор узлов | Роль-специфичный spine (DVU видит свою цепочку проверок) — ломает «один процесс», разводит оптику. Не рекомендую; DVU-специфику даю веткой (разв.4) |

## 8. Что проверить (концепт → spike)

Desktop (основной, ≥1280):
- [ ] Spine виден между B и C; активный узел подсвечен; состояния узлов различимы формой+цветом+подписью (не только цвет).
- [ ] Клик по узлу меняет ТОЛЬКО зону C (+D); B и spine не мигают; контекст клиента не теряется.
- [ ] Смена роли НЕ меняет набор узлов spine; меняет рендерер ячейки и действия D; позиция в процессе сохранена.
- [ ] Сервис на узле разворачивается inline в C (не drawer/модалка); сворачивается в сводку после результата.
- [ ] Все сервисы используют одну оболочку `ServiceStage` + 5 примитивов (нет «своих окон» сервисов).
- [ ] DVU four-eyes на узле VKYC: действия disabled + tooltip-причина при своей сессии (из объед. окна §6).
- [ ] loading/empty/error на ДВУХ уровнях (spine-полоса и узел-C) — нет белого экрана.
- [ ] Лёг один сервис → его узел error, остальная линия и B живут (graceful degradation).
- [ ] Ветка DVU (exception) рисуется ответвлением, не ломает главную линию; откат = стрелка назад + action-required.

Узкий вьюпорт (вторично, инструмент десктопный):
- [ ] Горизонтальный spine скроллится/схлопывается в «текущий + соседи»; не ломает B.
- [ ] Зона C|D в одну колонку, D вниз (как объед. окно).

A11y-флор:
- [ ] Узлы spine — фокусируемы с клавиатуры (`Tab`/стрелки), у активного видимый фокус.
- [ ] Состояние узла доступно не только цветом (форма+иконка+`aria-label` «Идентификация — требует действия»).
- [ ] Контраст semantic-цветов узлов на фоне полосы; reduced-motion гасит пульс активного узла.

## Ограничение (зафиксировать)

Внутренний MUI v5-инструмент, не SDDS-витрина. Из frontend-design взят только КРАФТ (анатомия, состояния узлов, единые примитивы, микротексты, a11y, паттерн подключения). Эстетика НЕ финалится здесь. Компоненты — боевой MUI v5 + `rmTheme` (оранжевый primary) + semantic-палитра MUI. Никакого кастома вне MUI. Spine-полоса — кастом-композиция из MUI-атомов (`Box`/`Chip`/`Stepper`), не сторонний компонент.
