# Дизайн-бриф CRM «одного окна» — 3 экрана (RM / MUI v5)

Автор: Ульяна (UX/DS). Дата: 2026-06-26. По нему кодит Федя (`app/src/crm/screens/`).
Стек: MUI v5, тема `app/src/rm/theme.ts` (оранжевый `#F5811F`, radius 8, плотный корпоративный, TextField `size="small"` по умолчанию, кнопки без капса/тени), шрифт SB Sans.

Источники: `docs/CRM — спецификация и фичи.md`, `docs/Разбор встречи — CRM одного окна 2026-06-25.md`, `docs/CRM — план реализации (рабочий).md`.

## Принципы для всех 3 экранов (DS-инвариант)

1. **Плотность оператора, не маркетинг.** Density умеренно-высокая: `TextField size="small"`, `Table size="small"`, паддинги секций 16–24px, не воздушный лендинг. Оператор колл-центра смотрит на это весь день — читаемость и скорость важнее «дыхания».
2. **Ритм.** Вертикальный шаг между секциями карточки профиля — 24px (`Stack spacing={3}`). Внутри секции — 16px (`spacing={2}`). Один ритм на всех 3 экранах.
3. **Оранжевый — только акцент действия.** `primary` (CTA, активные табы, фокус-ринг, ссылки). НЕ красить им статусы/чипы воронки — иначе теряется акцент действия. Статусы — semantic-палитра MUI (success/warning/error/info/default).
4. **Никакого хардкода цвета.** Только `theme.palette.*` и `sx`-токены. Чип-цвета — через `color="success|warning|error|info|default"`, не hex.
5. **Overlay-правила едины.** Создание оффера/звонка/встречи — `Dialog` (модалка) с заголовком + `DialogActions` (CTA справа, «Отмена» слева/text). Не drawer для коротких форм. Drawer зарезервируем под «историю/детали» позже, в v1 не вводим.
6. **A11y-floor.** Видимый фокус (MUI даёт по умолчанию — не отключать `outline`), размер тапа ≥ 40px (иконочные кнопки `size="medium"`, не `small` для primary-действий), контраст текста ≥ 4.5:1. Поиск и таблица — навигабельны с клавиатуры.
7. **Микротексты двуязычны.** RU — основной (рынок-команда русскоязычная), EN — в скобках/как `data-*` или просто зафиксирован в брифе для будущей i18n. В прототипе пишем RU видимым, EN — комментарием рядом в коде. Тон: активный глагол, sentence case, без капса, errors не извиняются.
8. **Хлебные крошки / возврат.** На профиле и создании — кнопка «Назад к поиску» слева сверху (стрелка + текст), `Button startIcon={<ArrowBack/>} variant="text"`. CTA-действия — справа. (правило проекта: Назад слева, CTA справа).

## Экран 1 — CrmSearch (`/rm/crm`)

### Задача экрана
Точка входа «одного окна». Оператор на входящем звонке вводит PAN и за 1 шаг понимает: клиент есть или нет. Если есть — список с уже видимым Application-статусом (чтобы решить «возобновить/оффер» ещё до открытия профиля). Если нет — прямой CTA «Завести профиль». Заменяет прыжки по Caldera (Customers / Online Onboarding / Sales).

### Секции / иерархия
```
┌─────────────────────────────────────────────────────────┐
│  CRM · Одно окно                          [Завести профиль] │  ← AppBar/заголовок страницы
├─────────────────────────────────────────────────────────┤
│   ┌───────────────────────────────────────────────┐      │
│   │ 🔍  Поиск по PAN…                       [Найти] │      │  ← крупный поиск, центр-верх
│   └───────────────────────────────────────────────┘      │
│   PAN — ключ. CIN/название — позже в фильтрах (v2)        │  ← helper микротекст
├─────────────────────────────────────────────────────────┤
│  Клиенты (All Clients)                    N результатов   │
│  ┌──────┬───────────┬───────┬──────────┬──────────────┐  │
│  │ PAN  │ Название  │ Тип   │ Адрес    │ Application   │  │  ← Table
│  ├──────┼───────────┼───────┼──────────┼──────────────┤  │
│  │ ABC… │ Acme LLP  │ LLP   │ Mumbai   │ ● Active      │  │
│  └──────┴───────────┴───────┴──────────┴──────────────┘  │
└─────────────────────────────────────────────────────────┘
```

Иерархия внимания: **поиск-бар (доминанта)** → таблица результатов → вторичный CTA «Завести профиль» (в шапке, всегда доступен — оператор может завести профиль и без поиска, но штатно приходит сюда из «не найдено»).

### Ключевые MUI-компоненты
- Поиск: `TextField` (size `medium` — это доминанта, исключение из дефолтного small; full-width в контейнере `maxWidth ~640px`) + `InputAdornment position="start"` с `<SearchIcon/>` + кнопка «Найти» либо как `InputAdornment position="end"` (`Button`), либо submit по Enter. **Дефолт: поиск по Enter + видимая кнопка «Найти»** (оператору с клавиатуры быстрее Enter, кнопка — для мыши/ясности).
- Таблица: **обычный `Table` (`TableContainer`+`Table`+`TableHead`+`TableBody`+`TableRow`+`TableCell`), size `small`**, обёрнут в `Paper variant="outlined"`. **НЕ DataGrid** — у MUI X DataGrid платный (Pro/Premium) для части фич (фильтры/группировка), а у нас простой список с одним статус-столбцом; Table закрывает потребность v1 без лицензии и без лишнего веса. Сортировку (если нужна) — `TableSortLabel`, опционально.
- Статус Application в строке — `Chip size="small"` (см. цвет-маппинг ниже) + точка-индикатор.
- Строка кликабельна целиком → переход на `/rm/crm/profile/:id` (`onClick` на `TableRow` + `hover` + `cursor: pointer`; и/или ссылка на PAN).
- CTA «Завести профиль» — `Button variant="contained" color="primary"` в шапке (оранжевый).

### Состояния

- Плейсхолдер поиска: **«Поиск по PAN»** / "Search by PAN"
- Helper под поиском: **«PAN — обязательный ключ поиска»** / "PAN is the required search key"

### A11y
- Поиск — `<form>` с submit по Enter; кнопка «Найти» `type="submit"`; `aria-label` на иконке-поиске.
- Таблица — `<caption>` (visually hidden) «Список клиентов»; столбец-статус с текстом внутри чипа (не только цвет).
- Кликабельная строка — добавить `role`/доступность с клавиатуры: либо ссылка-обёртка ячейки PAN (`<a>`/`Link`), либо `tabIndex` + `onKeyDown` Enter. Дефолт: ячейка PAN — `MuiLink` (нативно фокусируемая), строка — доп. удобство мышью.
- Чип статуса несёт текст, цвет — вторичен (дальтоник-safe).

### Развилки (нужен выбор Дениса)
- **Р1.** Empty-состояние: «чистое приглашение» (дефолт) vs «Недавно открытые профили» списком. → дефолт: приглашение; «недавние» — если останется время.
- **Р2.** Кнопка «Найти» отдельная vs только Enter/живой поиск. → дефолт: Enter + видимая кнопка. Живой поиск (debounce) не делаем в v1 (mock, лишняя сложность).

## Экран 2 — CompanyProfile (`/rm/crm/profile/:id`)

### Задача экрана
Материнская сущность «одного окна». Оператор/менеджер видит ВСЮ картину по клиенту без прыжков: кто это (primary), что советует AI, где клиент в воронке, что было в коммуникациях, и что можно сделать прямо сейчас. Это центр концепции Сергея/Марго.

### Секции / иерархия (сверху вниз — порядок = приоритет внимания)
```
┌──────────────────────────────────────────────────────────────┐
│ ← Назад к поиску                                               │
│ ┌────────────────────────────────────────────────────────────┐│
│ │ Acme Trading LLP            [LLP]   Источник: [Звонок ▾]    ││ ← ШАПКА (Paper)
│ │ PAN ABCDE1234F · CIN U72…  · Mumbai, MH                     ││
│ └────────────────────────────────────────────────────────────┘│
│ ┌────────────────────────────────────────────────────────────┐│
│ │ ✦ AI-сводка                                    рекомендация ││ ← AI SUMMARY (Paper, акцент-бордюр)
│ │ Импортно-экспортная деятельность. Задолженность перед …     ││
│ │ Рекомендуем предложить current account + …                  ││
│ └────────────────────────────────────────────────────────────┘│
│  Воронка                                                       │
│ ┌──────────┐→┌──────────┐→┌──────────┐→┌────────────────────┐ │ ← FUNNEL STRIP
│ │ Lead     │ │ Offer    │ │ Deal     │ │ Application         │ │
│ │ 2 · по…  │ │ 1 актив. │ │ —        │ │ ● Abandoned        │ │
│ └──────────┘ └──────────┘ └──────────┘ └────────────────────┘ │
│ ┌──────────────────────────┐ ┌──────────────────────────────┐ │
│ │ История коммуникаций      │ │ Действия                     │ │ ← 2 колонки (desktop)
│ │ • 24.06 Звонок входящий   │ │ [Создать оффер]              │ │
│ │ • 20.06 Заявка брошена    │ │ [Запланировать звонок]       │ │
│ │ • 18.06 Self-reg начата   │ │ [Назначить встречу]          │ │
│ │   (цифровой след)         │ │ [Сгенерировать ссылку]       │ │
│ └──────────────────────────┘ └──────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

Иерархия: **шапка (кто) → AI-сводка (что советуем — высоко, т.к. оператору на звонке это первое после «кто») → воронка (где в цикле) → история + действия (что было / что делать)**. Порядок AI выше воронки — сознательно: на входящем звонке оператору сначала нужна подсказка «как говорить», потом детали.

### Подача воронки — РЕШЕНИЕ: горизонтальный **FunnelStrip** (4 стадии в ряд), НЕ сгруппированные списки
Обоснование:
- Воронка Lead→Offer→Deal→Application — это **последовательность с направлением** (стрелки между стадиями несут смысл «продвижение»). Горизонтальная лента кодирует это направление визуально; вертикальные списки — нет.
- Оператору нужен **мгновенный «где клиент сейчас»** взгляд — стрип читается за полсекунды (4 карточки, счётчик + статус в каждой). Списки требуют чтения.
- На каждой стадии — счётчик/последний статус, не полный список сущностей. Клик по стадии → раскрытие деталей (в v1 — `Accordion`/`Collapse` под стрипом или просто tooltip-список; дефолт: клик стадии скроллит/раскрывает список под лентой).
- Контраргумент (много сущностей на стадии) для v1 неактуален — у клиента единицы лидов/офферов. Если станет много — деталь уходит в раскрытие, лента остаётся обзором.
- **Не использовать MUI `Stepper`** буквально как «шаги визарда» — это не процесс заполнения, а статусная воронка; берём паттерн-вид Stepper'а (горизонтальные узлы + коннекторы), но через кастомный ряд `Card`/`Paper` + `Box` коннекторы, чтобы класть в узел счётчик и чип статуса. (Stepper можно как основу `connector`, но контент узла кастомный.)

### Ключевые MUI-компоненты
- Шапка: `Paper variant="outlined"` + `Stack`. Название — `Typography variant="h5"`. Тип юрлица — `Chip` (нейтральный, `variant="outlined"`). Источник — `Select size="small"` (дропдаун «откуда пришёл»: Звонок / Potential / Self-registration). Primary-строка (PAN·CIN·Адрес) — `Typography variant="body2" color="text.secondary"`, разделители «·». CIN — условный: если нет (Sole Proprietor) — не рендерим, не пишем «—» в primary-строке.
- AI summary: `Paper variant="outlined"` с левым акцент-бордюром (`borderLeft: 3px solid primary.main`) ИЛИ `Alert`-подобный блок с иконкой ✦/`AutoAwesomeIcon`. Заголовок «AI-сводка» + бейдж «рекомендация» (`Chip size="small" variant="outlined"`). Текст — `body2`. Рекомендательный тон, не директивный. Заглушка — статичный текст из seed.
- Воронка: ряд из 4 `Paper`/`Card variant="outlined"` + коннекторы (стрелки/линии `Box`). В каждом узле: заголовок стадии (`overline`/`subtitle2`), счётчик (`h6`), статус-строка (`Chip` где применимо). Адаптив: на mobile — вертикальный стек карточек (коннекторы — вниз).
- История коммуникаций / цифровой след: `List` + `ListItem` с `ListItemIcon` (иконка типа события: звонок/заявка/онбординг/отказ) + `ListItemText` (primary = действие, secondary = дата/канал). Отказы (D3) — в этом же списке с `error`-акцентом иконки. Авто-оффер из онбординга (fixture) — появляется здесь как событие.
- Панель действий: `Stack` кнопок. Главное — «Создать оффер» (`contained primary`). Остальные — `outlined`: «Запланировать звонок», «Назначить встречу», «Сгенерировать ссылку». Клик «Создать оффер» → `Dialog` (форма оффера — заглушка v1: продукт обяз. + источник + теги + описание). «Сгенерировать ссылку» — disabled/tooltip «ссылка рождается из оффера» если оффера нет (правило Хафизовой: ссылка привязана к офферу, не к профилю).
- Layout desktop: 2 колонки (история 2fr / действия 1fr) через `Grid container` или `Stack direction="row"`. Mobile: одна колонка, действия — над/под историей (действия выше — оператору важнее CTA).

### Чипы статусов Application — цвет-маппинг (MUI semantic, без hex)
| Статус | MUI `color` | Смысл | Микротекст RU / EN |
|---|---|---|---|
| **active** | `success` | заявка в работе, живая | «Активна» / "Active" |
| **abandoned** | `warning` | брошена, не дозаполнена → возобновить | «Брошена» / "Abandoned" |
| **rejected** | `error` | отклонена | «Отклонена» / "Rejected" |
| (нет заявки) | `default` | — | «Нет заявки» / "No application" |

Тот же маппинг — на CrmSearch (столбец Application). **Один словарь статус→цвет на оба экрана** (вынести в `crm/components/statusChip` или константу). Чип всегда несёт текст (не только цвет) — a11y.
> ВНИМАНИЕ Феде: abandoned = `warning` (оранжево-янтарный MUI), это НЕ конфликт с брендовым оранжевым `#F5811F`, т.к. брендовый живёт только в `primary` (кнопки/акценты), а `warning` — отдельный токен палитры. Но визуально проверить, что они различимы; при близости — `warning` чип сделать `variant="outlined"`, а primary-кнопки `contained`.

### A11y
- Шапка — `<h1>` = название компании (один h1 на странице). Секции — `<h2>` (AI-сводка, Воронка, История, Действия).
- Воронка — `<ol>`/`role="list"` со стадиями по порядку; коннекторы-стрелки `aria-hidden`; статус каждой стадии — текстом.
- AI-блок — пометить как рекомендацию текстом, не только иконкой; иконка ✦ `aria-hidden`.
- Dialog оффера — `aria-labelledby` на заголовок, фокус-трап (MUI Dialog даёт), Esc закрывает, фокус возвращается на кнопку-триггер.
- Disabled-кнопка «ссылка» — tooltip доступен с клавиатуры (`Tooltip` оборачивает `span`, т.к. disabled-кнопка не ловит события).

### Развилки (нужен выбор Дениса)
- **Р3.** Позиция AI-сводки: выше воронки (дефолт, «как говорить» важнее на звонке) vs ниже шапки в правой колонке. → дефолт: широкий блок выше воронки.
- **Р4.** Воронка: FunnelStrip (дефолт, обосновано) vs сгруппированные списки. → дефолт FunnelStrip.
- **Р5.** Раскрытие деталей стадии: `Collapse`/`Accordion` под лентой (дефолт) vs переход на под-страницу. → дефолт: раскрытие на месте (одно окно — не уводим).
- **Р6.** «Сгенерировать ссылку» без оффера: disabled+tooltip (дефолт, по правилу Хафизовой) vs скрыть. → дефолт: disabled+tooltip (виднее логика).

## Экран 3 — CreateProfile (`/rm/crm/new`)

### Задача экрана
Ветка «не найдено» (UC-2б). Оператор заводит новый Company Profile минимальным набором primary-атрибутов (достаточным для ЕПК), с возможностью автозаполнить из Probe42 по PAN — чтобы не печатать руками. Фиксирует источник обращения.

### Секции / иерархия
```
┌──────────────────────────────────────────────────────────┐
│ ← Назад к поиску                                           │
│  Новый профиль компании                                    │
│ ┌────────────────────────────────────────────────────────┐│
│ │ PAN *  [ ABCDE1234F        ]   [Подтянуть из Probe42]  ││ ← PAN + авто-кнопка
│ │ (helper: введите PAN и подтяните данные из реестра)    ││
│ ├────────────────────────────────────────────────────────┤│
│ │ Тип юрлица *      [ LLP ▾ ]                             ││
│ │ Название *        [ ___________________ ]              ││
│ │ CIN (если есть)   [ ___________________ ]  ← условно   ││
│ │ Адрес *           [ ___________________ ]              ││
│ │ Источник *        [ Звонок ▾ ]                          ││
│ ├────────────────────────────────────────────────────────┤│
│ │                          [Отмена]  [Создать профиль]   ││ ← CTA справа
│ └────────────────────────────────────────────────────────┘│
└──────────────────────────────────────────────────────────┘
```

Иерархия: PAN — первый и выделенный (это ключ + триггер автозаполнения). «Подтянуть из Probe42» — рядом с PAN, не внизу (логически: ввёл PAN → подтянул → проверил/дозаполнил). Остальные поля — заполняются либо вручную, либо приходят из Probe.

### Ключевые MUI-компоненты
- Контейнер формы: `Paper variant="outlined"`, `maxWidth ~560px`, центр. `Stack spacing={2}` между полями.
- PAN: `TextField required` + кнопка «Подтянуть из Probe42» рядом (`Button variant="outlined" startIcon={<CloudDownloadIcon/>}`, disabled пока PAN не валиден). Можно `InputAdornment end` с кнопкой — но дефолт: отдельная кнопка справа от поля (`Stack direction="row"`), виднее.
- Тип юрлица: `Select` / `TextField select` (опции: Private Limited / LLP / Sole Proprietor / Partnership / …). `required`.
- Название, Адрес: `TextField required`. Адрес — `multiline` (2–3 строки) т.к. registered address длинный.
- CIN: `TextField` (НЕ required) + helper «только для зарегистрированных компаний». Если тип = Sole Proprietor — поле можно дизейблить/прятать (у ИП нет CIN). Дефолт: показываем всегда, но опционально + helper; при выборе Sole Proprietor — `disabled` с подсказкой.
- Источник: `Select required` (Звонок / Потенциальный / Самостоятельная регистрация). Если пришли с `?pan=` из not-found звонка — предзаполнить «Звонок».
- CTA: `DialogActions`-паттерн в конце — «Отмена» (`text`, слева) + «Создать профиль» (`contained primary`, справа). Прокидываем `?pan=` из CrmSearch → PAN предзаполнен.

### Состояния
| Состояние | Что |
|---|---|
| **Пустая форма** | Все поля пустые (кроме PAN, если пришёл из `?pan=`). «Создать профиль» disabled пока обязательные не заполнены. «Подтянуть из Probe42» disabled пока PAN невалиден. |
| **Probe loading** | После «Подтянуть» — кнопка → спиннер + disabled; поля формы → можно показать `Skeleton`/`LinearProgress` сверху формы. Микротекст «Запрашиваем данные из Probe42…». |
| **Заполнено из Probe (success)** | Поля заполнились значениями из mock. Пометить визуально, что данные пришли из реестра: helper-текст под полем «Из Probe42» (серым) ИЛИ `Alert severity="success" variant="outlined"` сверху «Данные подтянуты из Probe42. Проверьте и дополните.». Поля остаются редактируемыми (оператор может поправить). |

### A11y
- Каждое поле — `<label>` (MUI TextField даёт через `label` prop, связь `htmlFor`). Required — атрибут + визуальная *.

- Кнопка «Подтянуть» — `aria-busy` при загрузке; результат загрузки объявить (`Alert` с `role="status"`/`alert`).
