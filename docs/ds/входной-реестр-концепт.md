# UX-концепт «Рабочее место» — входной реестр (вход в процессное окно)

G3-концепт ДО кода. Денис смотрит КАРТУ/СХЕМЫ + дефолты, не финальный визуал. Эстетика не финалится.
Стек: MUI v5, тема `app/src/rm/theme.ts` (оранжевый primary), semantic-палитра MUI. Десктоп-first, внутренний инструмент.
Автор: Ульяна. Дата: 2026-06-26.
Вход: `docs/ds/процессное-окно-концепт.md` (вход ВЕДЁТ сюда — не противоречить), `docs/ds/crm-dvu-объединённое-окно.md` (роли/RBAC/пресеты — те же), `docs/Карта стыковок онбординг × CRM × DVU.md` (этапы = узлы мини-spine), `docs/CRM — research рынка.md` (save-and-resume, столбец Application).

---

## TL;DR

«Рабочее место» — **первый экран менеджера при входе**, ДО открытия профиля компании. Показывает **все заявки + все профили клиентов** списком с фильтрами; **дефолт-вид зависит от роли** (менеджер — портфель/воронка, оператор — входящие/поиск, DVU — очередь алертов со SLA) — один экран, разный дефолт, как рабочий стол объед. окна. Ключевое связующее звено с процессным окном: в строке реестра есть **колонка-этап (мини-spine)** — где клиент на линии процесса; **клик по строке → процессное окно ровно на этом узле**. Единый flow: **реестр → процессное окно → профиль**, контекст клиента и позиция в процессе передаются без потери. Переключатель **«Заявки / Профили клиентов»** (заявка = единица работы с этапом и SLA; профиль = клиент-360 без этапа). Сегменты-фильтры **Требует действия · Мои · Новые · Все** (+ SLA-горит для DVU). Глобальный поиск по PAN — из шапки (та же, что в объед. окне).

Дефолт формы — **список с колонкой-этапом**. Второй режим (переключаемо) — **канбан, где колонки = узлы spine** (этапы процесса): карточка-заявка едет по доске слева-направо ровно как точка по линии. Оба описаны, дефолт — список. Роли/RBAC/пресеты — **те же**, что в объед. окне: реестр — это ещё один пресетный вид над тем же ядром (A-shell + поиск), не новая система.

Сверка с макетом аналитиков (§6): беру их сильное — **«воронка в строке»** (Applications/Offers/Deals/Leads/Tasks по клиенту) → это и есть наш режим «Список», обогатил колонки; **время в статусе «(N дн)»** → сигнал возраста/SLA; **Prospect/Customer**, поиск по PAN/LLPIN/имени, **Add New+**, меню разделов. Поверх кладу наше: этап = **мини-spine** (мост в процессное окно, не текст-статус), переключатель Заявки/Профили, роль-дефолты, канбан. Расхожусь только где их термин смешивает уровни (их «Company Profile» как имя реестра ≠ наш экран одного клиента).

## 0. Где реестр в общем flow

```
   ВХОД менеджера
        │
        ▼
   ┌─────────────────────────────────────────────────────────────┐
   │  РАБОЧЕЕ МЕСТО (входной реестр)   ← ЭТОТ концепт              │
   │  список/канбан заявок+профилей · фильтры · мини-spine         │
   └─────────────────────────────────────────────────────────────┘
        │ клик по строке/карточке (несёт clientId + node)
        ▼
   ┌─────────────────────────────────────────────────────────────┐
   │  ПРОЦЕССНОЕ ОКНО   (docs/ds/процессное-окно-концепт.md)       │
   │  открыто на узле spine = колонка-этап из реестра             │
   └─────────────────────────────────────────────────────────────┘
        │ узел "Готово" / переключатель на профиль
        ▼
   ┌─────────────────────────────────────────────────────────────┐
   │  ПРОФИЛЬ КЛИЕНТА (client-360)   — контекст B объед. окна       │
   └─────────────────────────────────────────────────────────────┘
        ◀── «Назад к рабочему месту» (хлебная крошка, не теряя фильтр)
```

Реестр и процессное окно делят **ту же шапку A** (поиск по PAN, role-switcher, язык) и **тот же `SelectedClientContext`/`ProcessContext`**. Реестр — это состояние «клиент не выбран» (показываем всех); клик выбирает клиента+узел → монтируется процессное окно. Возврат восстанавливает фильтр/сегмент/режим реестра.

| **R4. Реестр** ⭐ | **[D-роль+режим]** | главная область: **список** (дефолт) ИЛИ **канбан** (узлы spine = колонки). Наполнение и колонки — по роли и по сущности (заявка/профиль) | `Table`/`DataGrid` (список) · `Box`-доска (канбан) |
| **R5. Right-rail (опц.)** | **[D, опц.]** | сводка-виджеты роли: счётчики SLA, мой портфель, «брошенные» (save-and-resume) → быстрый переход. Можно убрать в v1 | `Paper` правая колонка |

Замечания:
- **A — та же шапка**, что в процессном окне: ни поиск, ни role-switcher не дублируются, единый shell.
- **R2 сегменты — ядро удобства роли**: дефолтный сегмент = «что роль открывает первым» (DVU → «Требует действия», менеджер → «Мои»).
- **R5 опционально**: в spike можно не делать, сегменты+колонка-этап несут основную нагрузку.

## 2. ASCII — РЕЖИМ КАНБАН (второй, переключаемо): колонки = узлы spine

Канбан есть **только для сущности «Заявки»** (у профилей нет этапа → нечего раскладывать по доске; в режиме «Профили» переключатель канбана disabled + tooltip «Канбан доступен для заявок»). Колонки доски = **узлы линии процесса** (те же 9 крупных узлов, что на spine, разв.5 процессного концепта). Карточка-заявка лежит в колонке своего текущего узла; drag не обязателен (этап двигает процесс, не рука — read-mostly доска).

Канбан и список — **две проекции одного набора заявок** под текущим сегментом/фильтром. Сегменты R2 фильтруют обе формы одинаково (напр. «Требует действия» в канбане прячет колонки без action-required-карточек или красит их бейджем). Карточка канбана несёт мини-состояние узла (§4 процессного концепта: ⚠/⏳/✗/✓), как чип строки в списке.

**Почему канбан полезен именно DVU/менеджеру:** видно «затор» на этапе (где скопились заявки — VKYC-очередь, догрузка). Это операционная картина потока, которой список не даёт. Для save-and-resume (research #6) колонка «Вход/брошенные» сразу показывает зависшие.

## 3. Матрица «дефолт-вид × роль»

Один экран, дефолты по роли (как рабочий стол объед. окна §2). Меняется **дефолтная сущность · дефолтный сегмент · набор колонок/сортировка · дефолт-форма**.

| | Менеджер / Sales | Оператор КЦ | DVU-офицер | Compliance (задел) |
|---|---|---|---|---|
| **Дефолт-сущность** | Профили (портфель) | Заявки (входящие) | Заявки (очередь) | Заявки (на проверке) |
| **Дефолт-сегмент R2** | **Мои** | **Новые** / последние контакты | **Требует действия** | На подтверждение |
| **Дефолт-форма** | Список (портфель) | Список (поиск-центрично) | **Список** (очередь со SLA); канбан полезен (затор) | Список |
| **Ключевые колонки** | компания · продукты · этап · последняя активность · оффер | компания · PAN · этап · последний контакт · «брошена?» | компания · этап · **SLA** · назначен · приоритет · тип проверки | компания · решение DVU · этап · эскалация |
| **Доп-сегмент** | Воронка (Lead→Offer→Deal) | — | **SLA горит** | Эскалации |
| **Действие в строке** | Открыть профиль · Создать оффер | Возобновить заявку · Позвонить | Назначить на себя · Открыть проверку | Подтвердить · Эскалировать |
| **Что несёт мини-spine** | где клиент (для апсейла/контекста) | где брошен (возобновить с точки) | где затык (взять в работу) | где решение нужно |

Читать так: **реестр — это пресет роли** (тот же реестр пресетов `presets[role]` из объед. окна §5, расширенный полем `registryDefaults: { entity, segment, form, columns }`). Сам экран, шапка, сетка зон — общие; роль переставляет дефолты и колонки. Добавить роль = дописать `registryDefaults` в её пресет, ядро реестра не трогаем.

Связь с RBAC: роль видит в реестре **только доступные ей записи/действия**. DVU видит заявки на проверке, не весь портфель Sales; действия в строке отфильтрованы RBAC + four-eyes (напр. «Назначить на себя» disabled, если вёл сессию — как §6 объед. окна).

## 4. Колонка-этап (мини-spine) — связь с процессным окном

Это главный мост между концептами. Один источник истины — позиция клиента на линии процесса.

| Свойство | Реестр (мини-spine) | Процессное окно (полный spine) |
|---|---|---|
| Узлы | те же 9 крупных (разв.5 проц.концепта) | те же |
| Состояния узла | те же 6 (§4 проц.концепта): not-started/active/waiting/done/action-required/error | те же |
| Размер | компактный, ~7–9 точек + подпись текущего | полноширинный, с ветками DVU |
| Ветки DVU | **не рисуем** (тесно) — отражаем состоянием `action-required`/`error` на узле | рисуем ответвлением |
| Клик | по строке целиком → процессное окно **на текущем узле** | по любому узлу → монтирует его в C |

**Передача контекста (реестр → процессное окно):** клик по строке кладёт в `SelectedClientContext` = clientId и в `ProcessContext` = activeNode (узел из колонки). Процессное окно открывается сразу на нужном шаге (его дефолт «авто-фокус на активный узел», разв.6 — совпадает). **Возврат** «Назад к рабочему месту» восстанавливает реестр в том же сегменте/форме/скролле (хлебная крошка, не перезагрузка).

Микро-консистентность: состояние узла в строке и в полном spine выглядит одинаково (цвет+форма+иконка из semantic-палитры MUI) — пользователь учит язык один раз.

## 5. Состояния и удобство (loading / empty / error)

Правило: не оставлять дыр — на уровне реестра И на уровне сегмента.

| Состояние | Список | Канбан |
|---|---|---|
| **loading** | `Skeleton` строк × N (как CompanyProfile); мини-spine = серая цепочка-плейсхолдер | skeleton-карточки в колонках; заголовки колонок-узлов видны сразу |
| **empty (сегмент пуст)** | приглашение, не настроение: «В сегменте «Требует действия» сейчас пусто. Новые проверки появятся здесь.» / для Мои: «Нет назначенных. Возьмите из «Новые».» | «Нет заявок на этом этапе» в пустой колонке (бледный плейсхолдер), доска-каркас остаётся |
| **empty (поиск/фильтр ничего)** | «Ничего не найдено по фильтрам. Сбросить фильтры.» (кнопка) | то же + «сбросить» |
| **empty (нет PAN-результата из шапки)** | «Клиент не найден по PAN. Завести профиль или направить в self-reg.» (как объед. окно §4) | — |
| **error** | `Alert severity=error` над таблицей + «Повторить»; **шапка и сегменты живут** (graceful degradation) | то же над доской; колонки-каркас остаются |
| **частичный сбой** | одна заявка не отдала этап → её мини-spine в состоянии `error` + «Повторить» в строке, остальной реестр работает | карточка с error-бейджем, доска работает |

**Удобство:**
- Дефолт-сегмент роли открыт сразу — менеджер не настраивает фильтры при каждом входе.
- Сегменты со счётчиками («Требует действия (7)») — видно нагрузку до клика.
- Сортировка по SLA для DVU из коробки (горящие сверху).
- Переключение Список↔Канбан и Заявки↔Профили — **без потери сегмента/фильтра** (та же выборка, другая проекция; лёгкий fade ≤150 мс).
- Поиск по PAN из шапки **работает поверх любого вида** — мгновенный переход к клиенту, минуя фильтры.

## 6. Сверка с видением аналитиков

Аналитики показали свой макет этого экрана (заголовок «Company Profile», единая таблица «воронка по клиенту в строке»). Денис: пожелание, не обязательно — беру разумное, не копирую слепо. Ниже: что берём, что добавляем поверх, где расходимся.

### 6.1 Их экран (как у аналитиков)

- Заголовок **«Company Profile»**. Левое меню: **My Dashboard · CRM (Company Profile / Applications / Sales) · Calendar**.
- Поиск **«Search by PAN / LLPIN / Company Name»** + кнопка **«Add New +»**.
- **Единая таблица, строка = компания**, колонки: **Company Name** (+ под именем статус **Prospect/Customer**) · **PAN Number** (PAN + CIN) · **Applications** (со временем в статусе: «CAF · New (1 day)», «KYC (2 days)», «In progress (5 days)») · **Offers** (продукт, дата, приоритет High/Medium) · **Deals** (продукт, дата, приоритет) · **Leads** (источник Incoming Call/Website + Approved/Rejected) · **Tasks** (нумерованный список).
- Суть: **вся воронка по клиенту в одной строке**.

### 6.2 Что берём (разумное — сильные паттерны)

| Берём | Куда легло у нас |
|---|---|
| **«Воронка в строке»** (Applications/Offers/Deals/Leads/Tasks по клиенту) | это и есть наш **режим «Список»** — обогатил колонки строки этими сущностями (см. ASCII §1: Offer/Deal · Lead/источник · Tasks). Сильный паттерн, совпадает с research #3 (HubSpot/FSC) |
| **Время в статусе «(N дней)»** | добавил в колонку «Этап» под статусом узла: `⚠ требует действ. (2дн)` — сигнал возраста/SLA. Сильно усиливает «затор/брошенные» (research #6) |
| **Prospect / Customer** | подпись под названием компании в строке (`·Prospect` / `·Customer`) — статус отношения, ортогонален этапу заявки |
| **Поиск по PAN / LLPIN / Company Name** | расширил охват поиска шапки A (был только PAN) — для India LLPIN/имя нужны (Sole Prop без CIN) |
| **Add New +** | кнопка в R1 — завести профиль/заявку вручную (оператор, эт.1 карты стыковок → CreateProfile) |
| **Меню разделов** (Company Profile / Applications / Sales) | зона E — левое меню верхнего уровня. Их «Company Profile/Applications» ≈ наш переключатель **Профили/Заявки**; «Sales» ≈ воронка-сегмент Sales |
| **PAN + CIN в одной ячейке** | PAN-ячейка несёт и CIN (как у них) |

### 6.3 Что добавляем ПОВЕРХ их видения (наше)

| Добавляем | Зачем |
|---|---|
| **Колонка-этап = мини-spine** (а не текст-статус внутри Applications) | у них этап — просто текст «In progress (5 days)» в ячейке Applications. У нас та же информация — но как **мини-линия процесса** с 6 состояниями узла, **визуально единая с процессным окном**. Текст остаётся подписью под линией → их «(N days)» сохранён, но получает наглядный «адрес на линии» |
| **Клик по строке → процессное окно на узле** | у них клик ведёт в плоский профиль. У нас — в **процессное окно на нужном шаге** (несём clientId + node). Их статус-текст становится **точкой входа в работу**, а не просто меткой |
| **Роль-зависимый дефолт-вид** | их таблица — одна для всех. У нас тот же реестр, но **дефолт-сущность/сегмент/колонки по роли** (§3). DVU открывает «Требует действия/SLA», менеджер — «Мои/портфель». Один экран, разный вход |
| **Канбан-режим** (колонки = узлы spine) | у них только таблица. Наш канбан = та же выборка второй проекцией: **видно затор по этапам** потока (где скопились VKYC/догрузки). Их таблица ≈ наш «Список» (дефолт), канбан — сверх |
| **Сегменты-фильтры со счётчиками** (Требует действия · Мои · Новые · Все · SLA) | у них фильтрация не показана как сегменты — мы делаем нагрузку видимой до клика |

### 6.4 Где расходимся и почему

| Расхождение | Их | Наше | Почему |
|---|---|---|---|
| **Заголовок экрана** | «Company Profile» | «Рабочее место» | «Company Profile» у нас = экран ОДНОГО клиента (client-360, объед. окно зона B). Входной реестр — список ВСЕХ → отдельное имя, чтобы не путать уровень (реестр ≠ профиль). Их «Company Profile» как имя реестра — коллизия терминов |
| **Сущность строки** | всегда «компания» (профиль), воронка в ячейках | **переключатель Заявки/Профили**: строка-заявка (единица работы, есть этап+SLA) ИЛИ строка-профиль (клиент-360, без этапа) | заявка и клиент — разные жизненные циклы (research #3/#4, развилка Марка). У клиента может быть 0..N заявок. Их «одна строка = компания» теряет это; мы даём оба среза |
| **Этап как текст vs мини-spine** | текст «In progress (5 days)» | мини-spine + текст | см. §6.3 — связь с процессным окном требует единого визуального языка линии, не свободного текста |
| **Меню «Company Profile / Applications / Sales»** | пункты меню (3 раздела) | у нас Профили/Заявки = переключатель в R1, «Sales» = сегмент-воронка | избегаем дубля навигации: один реестр с переключателем сущности вместо трёх почти одинаковых таблиц-разделов. (Развилка §7.3: можно и пунктами меню) |

**Итог сверки:** ядро их видения — «воронка по клиенту в одной строке» + «время в статусе» — **берём целиком** в режим «Список». Поверх кладём наше: этап = мини-spine (мост в процессное окно), переключатель Заявки/Профили, роль-дефолты, канбан. Расходимся только там, где их термин/структура смешивает уровни (реестр vs профиль, компания vs заявка) — развожу явно, чтобы flow реестр→процессное окно→профиль был чистым.

## 7. Развилки для Дениса (форма — за тобой)

| # | Развилка | По умолчанию (рекомендация) | Альтернатива |
|---|---|---|---|

| **6** | **Right-rail R5 (сводка-виджеты)** | **убрать в v1** — сегменты+счётчики несут нагрузку | оставить для DVU (SLA-дашборд) / Sales (портфель-цифры) |

| **11** | **Этап в строке: мини-spine (наше) vs текст-статус «(N дн)» (как у аналитиков)** | **мини-spine + текст** — связь с процессным окном | только текст-статус как у них — проще верстать, но теряется визуальный мост к spine |

## 8. Что проверить (концепт → spike)

Desktop (основной, ≥1280):
- [ ] Дефолт-вид зависит от роли (DVU → Заявки/Требует действия/Список; менеджер → Профили/Мои; оператор → Заявки/Новые).
- [ ] Переключатель Заявки/Профили меняет сущность и колонки; сегмент/фильтр сохраняется где применимо.
- [ ] Колонка-этап (мини-spine) показывает узел + состояние теми же визуалами, что полный spine (§4 проц.концепта).
- [ ] Клик по строке → процессное окно открывается **на узле из колонки «Этап»** (контекст clientId + node переданы, не теряются).
- [ ] «Назад к рабочему месту» восстанавливает сегмент/форму/скролл (не перезагрузка фильтров).
- [ ] Переключатель Список/Канбан — одна выборка, две проекции; сегмент R2 фильтрует обе одинаково.
- [ ] Канбан-колонки = узлы spine; карточка лежит в колонке своего узла; счётчики колонок верны.
- [ ] Канбан disabled в режиме «Профили» + tooltip-причина.
- [ ] Поиск по PAN из шапки работает поверх любого вида; нет результата → приглашение завести/направить.
- [ ] loading/empty/error на уровне реестра И сегмента (нет белого экрана); шапка+сегменты живут при error.
- [ ] RBAC: роль видит только доступные записи/действия; four-eyes-гард на действиях строки.

Узкий вьюпорт (вторично):
- [ ] Колонки списка приоритизируются (компания · этап · действие остаются), второстепенные прячутся.
- [ ] Канбан горизонтально скроллится; заголовки колонок-узлов sticky.

A11y-флор:
- [ ] Строки/карточки фокусируемы с клавиатуры; Enter = открыть процессное окно.
- [ ] Состояние мини-spine доступно не только цветом (иконка + `aria-label` «Acme — этап Идентификация, требует действия»).
- [ ] Контраст semantic-цветов статусов; видимый фокус на сегментах и переключателях.

## Связь с уже сделанным (чтобы не разъехалось)

- **Вход в процессное окно:** клик по строке = открыть `процессное-окно-концепт.md` на узле из колонки-этапа. Мини-spine реестра и полный spine окна — один визуальный язык, одни 6 состояний узла.
- **Те же роли/RBAC/пресеты**, что в `crm-dvu-объединённое-окно.md`: реестр — пресетный вид над тем же ядром (A-shell + поиск + `SelectedClientContext`). Пресет роли получает поле `registryDefaults` (сущность/сегмент/форма/колонки) — единственное расширение контракта. «Добавить роль = дописать пресет» сохраняется.
- **Та же шапка A** (поиск по PAN, role-switcher, язык) — не дублируется между реестром и процессным окном.
- **Контракт событий:** статус заявки/этап в реестре читается из той же машины состояний (`application.stage-changed` двигает узел и в строке, и на spine; `dvu.result` красит action-required; `onboarding.completed` → узел «Готово» / переключение Заявка→Профиль). Save-and-resume (research #6): «брошенные» видны сегментом + узлом «Вход» в action-required.
- Не финалю эстетику. Денис смотрит анатомию + 2 ASCII + матрицу дефолтов + развилки §6, правит форму.

## Ограничение (зафиксировать)

Внутренний MUI v5-инструмент, не SDDS-витрина. Из frontend-design взят только КРАФТ (анатомия, дефолты по роли, состояния, микротексты empty/error, a11y, мост к процессному окну). Эстетика НЕ финалится. Компоненты — боевой MUI v5 + `rmTheme` (оранжевый primary) + semantic-палитра MUI. Мини-spine — кастом-композиция из MUI-атомов (как полный spine), не сторонний компонент. Никакого кастома вне MUI.
