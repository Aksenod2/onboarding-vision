# UX-анатомия объединённого окна «отдел работы с клиентами» (CRM + DVU)

G3-дизайн ДО кода. По нему Федя собирает spike. Денис правит карту зон, не финальный визуал.
Стек: MUI v5, тема `app/src/rm/theme.ts` (оранжевый акцент RM). Десктоп-first, внутренний инструмент.
Автор: Ульяна. Дата: 2026-06-26. Вход: `docs/Решение — объединение CRM и DVU 2026-06-26.md`, `docs/CRM — спецификация и фичи.md`, research Марка.

---

## TL;DR

Окно = **неизменное ядро (client-360) + сменный рабочий стол (workspace-пресет по роли)**. Шапка, контекст клиента и каркас сетки — СТАТИЧНЫЕ для всех ролей; меняется только центральная рабочая область и панель действий. Один клиент — разные проекции: при смене роли контекст выбранной компании НЕ теряется, профиль тот же, переключается «оптика». Добавить роль/домен = добавить пресет, ядро не трогаем. Four-eyes показываем как disabled-действие с tooltip-причиной, не прячем.

Три тезиса Дениса закрываются так: **одно окно** = всё в одной сетке, переключение ролью внутри, без выхода → ниже зона A; **консистентность** = ядро B (контекст+профиль) переживает смену роли, источник один → разделы 3-4; **масштабируемость** = пресет-паттерн → раздел 5.

## 1. Анатомия экрана — зоны

Пять зон. Помечены: **[S] СТАТИЧНАЯ** (всегда, одинакова для всех ролей) / **[D] ДИНАМИЧЕСКАЯ** (меняется по роли).

| Зона | Тег | Что внутри | MUI-каркас |
|---|---|---|---|
| **A. Глобальная шапка** | **[S]** | логотип/продукт · глобальный поиск по PAN · **role-switcher** · переключатель языка RU/EN · аккаунт-меню | `AppBar` + `Toolbar`, `Autocomplete` (поиск), `Select`/`Menu` (роль), `ToggleButtonGroup` (язык) |
| **B. Контекст клиента (client-360)** | **[S]** | «кто открыт сейчас»: название компании, тип юрлица (chip), PAN/CIN/адрес, статус Application, мини-сводка подписанты/продукты. Один и тот же для всех ролей | `Paper` (sticky-полоса под шапкой) + `Typography` + `Chip` |
| **C. Рабочий стол (workspace)** | **[D]** | главная сменная область. Sales: воронка Lead→Offer→Deal→Application. DVU: очередь алертов + карточка проверки. Оператор: входящие/быстрый поиск + история | контейнер-слот, наполнение из пресета (см. §2) |
| **D. Панель действий** | **[D]** | контекстные действия роли. Sales: создать оффер/звонок/встречу/ссылку. DVU: одобрить/отклонить/запросить документ (с four-eyes-гардом). Оператор: позвонить/возобновить заявку/направить в self-reg | `Paper` правая колонка (desktop) / нижний бар (узкий вьюпорт) |
| **E. Боковая навигация доменов** | **[D, опц.]** | переключение между «своими» разделами роли (у DVU — фильтры очереди; у Sales — Мои клиенты / Воронка / Таски). Может быть свёрнута в v1 | `Drawer` (permanent, узкий) или `Tabs` в зоне C |

Замечания по зонам:
- **B — ключ консистентности.** Она читается из контекста выбранного клиента, а не из роли. Поэтому при смене роли B не перерисовывается — только C и D.
- **E опциональна для spike.** Чтобы не раздувать v1, навигацию роли можно дать табами вверху зоны C. Drawer — задел.
- **A.role-switcher — демо-RBAC.** В проде это не «выпадашка ролей», а текущая роль из RBAC. В прототипе — переключалка, чтобы Денис показывал проекции на одном клиенте.

## 2. Матрица «зона × роль»

Колонки — роли. Задел под Compliance (5-я колонка) и Concurrent Audit (read-only) показан, но в v1 spike хватит трёх.

| Зона | Менеджер / Sales | Оператор колл-центра | DVU-офицер | Compliance (задел) |
|---|---|---|---|---|
| **A. Шапка** [S] | видно, идентична | видно, идентична | видно, идентична | видно, идентична |
| **A. role-switcher** | показывает «Sales» | «Оператор» | «DVU» | «Compliance» |
| **B. Контекст клиента** [S] | видно, **идентично** | видно, **идентично** | видно, **идентично** | видно, **идентично** |
| **C. Рабочий стол** [D] | **воронка** Lead→Offer→Deal→Application (FunnelStrip + стадии, портфель/Мои клиенты) | **входящие + быстрый поиск**: история коммуникаций, AI-summary, брошенная заявка → «возобновить» | **очередь алертов** (таблица задач со SLA) + **карточка проверки** выбранной заявки (документы, KYC-результат, лог решений) | **обзор решений DVU** (read-mostly) + контур эскалаций |
| **D. Панель действий** [D] | Создать оффер · Звонок · Встреча · Ссылка-из-оффера (disabled без оффера) | Позвонить · Возобновить заявку · Направить в self-reg · Завести профиль (Probe42) | Одобрить · Отклонить · Запросить документ · Назначить на себя — **с four-eyes-гардом** | Подтвердить · Эскалировать · Вернуть на доработку |
| **E. Навигация роли** [D] | Мои клиенты · Воронка · Таски/календарь | Очередь входящих · Поиск | Очередь (фильтры: новые/мои/SLA-горит) · История проверок | Реестр решений · Аудит-след |

Расшифровка «что именно меняется в рабочем столе» (зона C):
- **Sales** = pull-модель, воронка-центрично. Главный объект — продвижение по стадиям. Виджет — FunnelStrip (уже есть) + карточки Lead/Offer/Deal.
- **Оператор** = reactive, входящие-центрично. Главный объект — текущий контакт: история + AI-summary + действие «возобновить/оффер». Самый «лёгкий» стол.
- **DVU** = push-модель, task-центрично. Главный объект — задача из очереди. Стол двухчастный: список-очередь (`Table`/`DataGrid`) сверху → карточка выбранной проверки снизу/справа. Лог решений (audit trail) обязателен (BRD).
- **Sales и Оператор близки** (оба вокруг продажи/контакта) → можно держать один Sales-workspace с под-видом, но в матрице разведены, т.к. дефолтный стол и набор действий разные. Рекомендация Сони — объединять; в spike можно как 2 пресета над общим компонентом.

## 3. Что сохраняется, что меняется при смене роли

Это и есть **консистентность для пользователя** (тезис Дениса №2 + №1).

| | СОХРАНЯЕТСЯ при смене роли | МЕНЯЕТСЯ при смене роли |
|---|---|---|
| **Зона** | A (шапка-каркас), B (контекст клиента) | C (рабочий стол), D (действия), E (навигация роли) |
| **Данные** | выбранный клиент (id), его профиль, PAN/тип/адрес, статус Application, подписанты/продукты | проекция этих данных: воронка vs очередь vs история |
| **Состояние** | кто открыт, язык, позиция в client-360 | активный виджет, набор кнопок, доступность four-eyes |

**Главная идея для Дениса: «один клиент — разные проекции».**
Открыта `Acme Trading Pvt Ltd`. Менеджер видит её как точку воронки (где оффер, дошла ли до сделки). Оператор — как историю звонков + брошенную заявку. DVU — как задачу проверки с документами. **Компания одна, данные одни, оптика разная.** Переключил роль — клиент не «потерялся», контекст-полоса B осталась, сменилась только середина. Это прямой ответ на «не прыгать между системами»: смена угла зрения происходит внутри окна, без перезагрузки контекста.

Технически: контекст выбранного клиента живёт в общем сторе/контексте (`SelectedClientContext`), не внутри workspace-компонента. Workspace — чистая функция от (роль, clientId). Смена роли меняет только какой workspace смонтирован.

## 4. Состояния и удобство

Все состояния зоны C — не оставлять дыр (правило: loading / empty / error / disabled / success).

| Состояние | Sales workspace | DVU workspace | Оператор |
|---|---|---|---|
| **loading** | Skeleton воронки (как в текущем CompanyProfile) | Skeleton строк очереди (`Skeleton` × N) | Skeleton истории |
| **empty** | «Пока нет лидов. Создайте оффер, чтобы начать воронку.» | «Очередь пуста. Новые проверки появятся здесь.» (invitation, не mood) | «Клиент не найден по PAN. Завести профиль или направить в self-reg.» |
| **error** | `Alert severity=error` + «Повторить» | то же | то же |
| **disabled** | «Ссылка-из-оффера» disabled без оффера + tooltip-причина (правило Хафизовой) | **four-eyes**: «Одобрить»/«Отклонить» disabled если офицер вёл эту сессию (см. §6) | «Возобновить» disabled если нет брошенной заявки |
| **success** | Snackbar «Оффер создан» | Snackbar «Заявка одобрена» / «Отправлено на доработку» | Snackbar «Заявка возобновлена» |

**Без потери контекста при переключении:** смена роли НЕ сбрасывает выбранного клиента и НЕ делает повторный fetch профиля (B живёт в общем сторе). Перезагружается только содержимое C — со своим loading-скелетоном. Микро-переход: лёгкий fade зоны C (≤150 мс), B и A не мигают.

## 5. Паттерн масштабирования: ядро + сменные workspace-пресеты

Тезис Дениса №3 (масштабируемость): добавить роль/домен без переделки.

```
        ┌──────────────────────────────────────────┐
        │  ЯДРО (не трогаем при добавлении роли)     │
        │  A. Shell (AppBar + role-switcher + поиск) │
        │  B. ClientContext (client-360, sticky)     │
        │  Grid-каркас C|D                           │
        └──────────────────────────────────────────┘
                          ▲ монтирует по роли
        ┌─────────────┬─────────────┬─────────────┬─── + новый
        │ SalesPreset │  CcPreset   │  DvuPreset  │  ХPreset
        │ workspace+  │ workspace+  │ workspace+  │  (новая роль/
        │ actions     │ actions     │ actions     │   домен)
        └─────────────┴─────────────┴─────────────┴───
```

**Пресет = декларативный объект:** `{ role, workspaceComponent, actions[], navItems[], rbac }`. Реестр пресетов — `presets[role]`. Ядро читает текущую роль → рендерит `presets[role].workspaceComponent` в слот C и `presets[role].actions` в зону D.

Добавление новой роли (напр. Compliance) или домена = **дописать один пресет** в реестр + добавить роль в role-switcher. Зоны A/B/сетка не меняются — отсюда «без переделки». Это ровно паттерн Salesforce FSC: Sales Cloud vs Service Console над общим Account 360 (research Марка, паттерн #1/#4).

Преимущество для кода Феди: workspace-компоненты независимы, тестируются отдельно, DVU-пресет в v1 — fixture-заглушка (решение команды: «DVU-очередь — заглушка в v1»), позже наполняется без касания ядра.

## 6. Role-switcher и four-eyes

### Role-switcher (демо-RBAC)
- **Где:** зона A (шапка), справа, перед языком и аккаунтом. `Select` или `Button`+`Menu` с текущей ролью и иконкой.
- **Как выглядит переключение:** клик → меню ролей (Sales · Оператор · DVU · [Compliance]). Выбор → меняется C и D, B и A остаются. Лёгкий fade зоны C.
- **Что на экране:** контекст клиента сохраняется (см. §3). Если у новой роли нет доступа к открытому клиенту по RBAC — B остаётся, C показывает empty-состояние «У роли DVU нет активной проверки по этому клиенту» (не ошибка, а пустое приглашение).
- **Демо-маркер:** рядом со свитчером — маленький chip «демо-роль», чтобы Денис на показе не путал прототип с реальным RBAC.

### Four-eyes (segregation of duties) — обязательное правило команды
Регуляторика: «система исключает офицера, проводившего VKYC, из её валидации» (BRD `3-DVU #8`). Показываем, не прячем:

- DVU-действия «Одобрить»/«Отклонить» по заявке, где `session.performedBy === currentOfficer`, рендерятся **disabled** + `Tooltip`: «Нельзя проверять заявку, которую вели вы. Назначьте другого офицера.» (паттерн disabled+tooltip как у «Ссылки-из-оффера» в текущем коде — `Tooltip` оборачивает `<span>`, т.к. disabled-кнопка не ловит события).
- В карточке проверки — плашка-подсказка кто вёл сессию (`Alert severity=info`, сине-серая — оранжевый зарезервирован), чтобы причина disabled читалась без наведения.
- Кнопка «Назначить на себя» тоже disabled, если это твоя сессия.
- Compliance/Audit как отдельный контур: read-mostly, действия «Подтвердить/Эскалировать» — третья рука, не пересекается с DVU.

Принцип микротекста (frontend-design): запрет объясняет причину и путь решения, не извиняется, не абстрактен.

## 7. Развилки для Дениса (решения формы — за тобой)

1. **Sales и Оператор — один пресет или два?** Рекомендация Сони/research — близки, можно один Sales-workspace с под-видом. Я по умолчанию развожу на 2 пресета (чище для демо проекций), но над общим компонентом воронки/истории. **Развилка:** объединить в один пресет с табом «Воронка | Входящие» — экономит зону E.
2. **Зона E (навигация роли): Drawer или Tabs?** По умолчанию для spike — `Tabs` вверху зоны C (меньше кода, десктоп-плотность ок). Drawer — задел под рост числа разделов. **Развилка:** нужен ли постоянный Drawer уже в v1.
3. **DVU-очередь: `Table` или `DataGrid` (@mui/x)?** `DataGrid` даёт сортировку/SLA-колонки из коробки, но это доп-зависимость. Для заглушки v1 хватит `Table`. **Развилка:** тянем ли `@mui/x-data-grid` сразу.
4. **role-switcher: `Select` в шапке vs отдельный экран-«рабочий стол выбора роли»?** По умолчанию — `Select` в шапке (быстрое переключение проекций, что и нужно для демо тезиса). Экран-лендинг ролей — лишний клик. **Развилка:** оставляем в шапке.
5. **Контекст-полоса B: всегда видна (sticky) или сворачивается?** По умолчанию sticky-полоса (консистентность = всегда видно, кто открыт). **Развилка:** при узком вьюпорте схлопывать в одну строку.
6. **Граф связей Company↔Signatory↔UBO в B (research #2, ARC)?** По умолчанию — плоские chip-списки в v1 (как сейчас), граф позже. **Развилка:** мини-граф уже в spike.

## 8. Что проверить (spike)

Desktop (основной, ≥1280):
- [ ] Смена роли в шапке меняет ТОЛЬКО C и D; A и B не мигают, клиент не теряется.
- [ ] Контекст-полоса B одинакова во всех трёх ролях для одного клиента.
- [ ] Каждый workspace имеет loading/empty/error (не белый экран).
- [ ] DVU four-eyes: disabled-кнопки + tooltip-причина при своей сессии.
- [ ] Добавление 4-го пресета (Compliance-заглушка) не требует правок зон A/B/сетки.
- [ ] Глобальный поиск по PAN из шапки доступен в любой роли.

Узкий вьюпорт (вторично, инструмент десктопный, но не ломать):
- [ ] Сетка C|D схлопывается в одну колонку, D уходит вниз.
- [ ] Контекст-полоса B сворачивается в одну строку, остаётся читаемой.

A11y-флор (внутренний, но не ниже пола):
- [ ] Видимый фокус на role-switcher и действиях; tooltip-причина доступна с клавиатуры.
- [ ] Контраст оранжевого primary на тексте действий — проверить на disabled-состоянии.

## Ограничение (зафиксировать)

Это внутренний MUI-инструмент, не клиентская SDDS-витрина. Из frontend-design взят только КРАФТ (зоны, иерархия, состояния, микротексты, four-eyes-доступность). Эстетика НЕ финалится здесь — Денис смотрит карту зон и правит. Компоненты — боевой MUI v5 + тема `rmTheme` (оранжевый). Никакого кастома вне MUI-палитры.
