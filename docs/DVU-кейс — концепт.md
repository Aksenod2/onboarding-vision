# DVU-кейс — концепт (сквозной сценарий DVU ↔ клиент, Sole Proprietor)

> Автор: Боря (BA) · 2026-06-10 · Приоритет 1 от Марго (BR-11, BR-17).
> Источники: BRD `_исходники/2026-06-09_BRD/` (главный — `3-DVU-Data-Verification-Unit-Process.md`), разборы встреч 2026-06-08/10, код прототипа (`app/src/screens/RM01…RM05`, дашборд клиента, mock-модели).
> Статус: **концепт для обсуждения** — в BRD по взаимодействию DVU↔клиент много дыр, все перечислены в §4.

**TL;DR.** DVU (= AOD, Account Opening Division) — единая ручная точка обработки всех исключений онбординга: на каждый сбой автоматики или правку данных генерится алерт, офицер DVU разбирает, при необходимости запрашивает у клиента документы, итог — approve / запрос документов / эскалация в Compliance. Для прототипа рекомендую один сквозной сценарий: **клиент правит реестровое поле на «Подтверждении данных компании» → шаг уходит в Verifying → задача в очереди DVU → офицер запрашивает документ → у клиента на дашборде «Требует действия: загрузите документ» → клиент грузит → DVU approve → Done, счёт активен.** Обе стороны уже частично свёрстаны; главный технический гэп — клиентский (v2) и DVU-шный (v1) mock живут в разных базах и не связаны.

## §1. Что такое DVU-кейс per BRD

### 1.1. Роль DVU (verbatim-смысл)

DVU = Data Verification Unit, часть AOD (Account Opening Division); в BRD термины DVU и AOD взаимозаменяемы — `1.1-SOP… → Appendix 1 (Abbreviations)`. Назначение — «сконцентрировать все ручные проверки в одних руках» — `3-DVU… → Business Overview`:

- на **каждое исключение или сбой интеграции** генерится алерт в workspace AOD;
- AOD разбирает алерты вручную, чтобы онбординг продолжился;
- AOD **коммуницирует с проспектом для запроса доп. документов**;
- AOD отвечает за верификацию данных и документов в течение всего онбординга;
- AOD ассистирует при офлайн-онбординге в отделении.

«DVU team shall be responsible for handling **all exception scenarios** … based on system triggers and defined controls» — `1.1-SOP… → Exception Handling (DVU SubProcess)`.

Дополнительные триггеры из STP-критериев (`1.1-SOP… → STP criteria`): провал любого критерия выводит из STP → DVU и/или Compliance. Для Sole Prop важные: Registry Validation fail → **DVU Verification**; PEP → **Compliance**; **нерезидент / FATCA** → **Compliance Review**; eKYC/VKYC fail → **DVU + CRM**; не все согласия перед VKYC приняты → алерт DVU, офицер звонит клиенту (`1.1-SOP… → Table A, шаг 03/04A`).

В процессе Sole Prop DVU включается на шагах 04/05a (ручное заполнение/валидация правок), 07 (BGV-алерт), 08 (KYC fail), 10 (DVU+BGV+Screening), 11 (эскалация в Compliance) — `1.4-Sole-Prop… → Process Description`.

### 1.3. Что DVU делает с задачей (по сценарию #5 — наш кейс)

«Prospect uploads docs; AOD either (a) marks data **valid**; (b) **corrects data** and pushes to OBO for prospect confirmation; (c) **requests new documents**. Chunks may contain each part of an application (UDIN, CRILC, LEI, IEC)» — `3-DVU… → #5`.

### 1.4. Чем кончается

| Исход | Что происходит | Источник |
|---|---|---|
| **Approve** | Данные валидны → флоу продолжается; счёт (открытый в **debit freeze**) размораживается при выполнении ВСЕХ условий: initial deposit + BGV clear (если требуется) + DVU checks clear | `1.1-SOP… → Product Creation`; `1.4 → step 09/12` |
| **Запрос документов** | AOD коммуницирует с клиентом, клиент догружает → новый цикл проверки | `3-DVU… → Business Overview, #5(c)` |
| **Исправление данных банком** | DVU правит данные → пушит в OBO **на подтверждение клиенту** | `3-DVU… → #5(b)` |
| **Эскалация в Compliance** | Фрод / forged PAN / VCIP-подозрение / high-risk → Compliance, при reject — счёт не открывается; о решении клиенту сообщает **CRM** «in a professional manner» | `1.4 → Compliance Escalation Criteria`; `1.1-SOP… → Sanctions or PEP Screening (CRM)` |
| **Fail на валидации правок** | `1.4 → step 05a`: «if fail **stop**» — деталей, что видит клиент, в BRD нет | `1.4 → Process Description` |

⚠️ Клиентская терминология: клиенту показываем статус **«Verifying»**, слово «DVU» наружу не выносим (решение Марго 2026-06-10, BR-15 — `Разбор встречи — Демо макетов 2026-06-10.md`).

## §2. Сквозной сценарий для прототипа (рекомендация)

Выбран сценарий **AOD #5 (Probe42 Data Validation)** — он единственный, где есть полный цикл «клиент ↔ DVU ↔ клиент» с обеими сторонами UI, и именно его Марго назвала приоритетом (BR-11 + BR-17: «запрос документов из рабочего места DVU прилетает в кабинет клиента»).

**Демо-цепочка (9 тактов, две роли):**

| Такт | Сторона | Что происходит | Экран |
|---|---|---|---|
| 1 | Клиент | На «Подтверждении данных компании» правит реестровое поле (напр., юр. адрес), грузит обязательный proof (BR-23), жмёт Continue. Нотификация: «данные направлены на проверку» | Подтверждение данных компании |
| 2 | Клиент | На дашборде шаг «Подтверждение данных компании» = **Verifying** (не Done); остальные шаги доступны — клиент идёт дальше (согласие перед видео, видео) | Дашборд клиента |
| 3 | Банк | В очереди DVU — заявка Aarav Sharma, проверка «OBO · данные», причина «изменено поле: registered address, приложен proof» | Очередь DVU (RM-01) |
| 4 | Банк | Офицер берёт задачу (лок для остальных — BR-18), открывает карточку: старое значение (Probe42) ↔ новое (клиент) + загруженный proof | Карточка задачи (RM-02) |
| 5 | Банк | Proof не подходит (напр., utility bill старше 3 мес — `1.4 → Address Proof 2`) → «Запросить документы»: выбор типа документа из справочника + комментарий клиенту | Карточка задачи (RM-02) |
| 6 | Клиент | На дашборде шаг → **«Требует действия»**: карточка в блоке «Что требуется от вас» — «Загрузите документ: Utility Bill (не старше 3 мес)» + комментарий офицера | Дашборд клиента |
| 7 | Клиент | Открывает запрос, грузит документ → шаг снова **Verifying** | Дашборд / экран запроса документов |
| 8 | Банк | В карточке появился догруженный документ → офицер жмёт «Подтвердить валидность» | Карточка задачи (RM-02) |
| 9 | Клиент | Шаг → **Done**; когда все шаги done — блок «Счёт открыт» на дашборде | Дашборд клиента |

Почему адрес, а не PEP/CRILC: правка реестрового поля — единственный триггер, который клиент создаёт сам, на уже свёрстанном экране, и который закрывается документом (полный цикл). PEP уходит в Compliance (другая роль), CRILC — эпик виджета счетов, VKYC-fail уже показан в RM-04/05 как отдельные виньетки.

**Опциональное расширение демо (если успеем):** такт 5' — вместо запроса документа офицер сам **исправляет значение** и пушит клиенту на подтверждение (`3-DVU… → #5(b)`) — красиво показывает вторую ветку, но требует ещё один клиентский экран «Банк предлагает исправление».

### U1. Принцип, на котором всё держится

**«Verifying» и «Требует действия» — два разных эмоциональных режима, и их нельзя красить одинаково.**

| Режим | Чей ход | Семантика | Цвет/тон |
|---|---|---|---|
| **Verifying** | Банка | «Всё в порядке, мы работаем, от вас ничего не нужно» | Спокойный сине-серый, **без** warning-семантики, без пульсаций |
| **Requires action** | Клиента | «Без вас не продолжится» | Тёплый оранжевый (уже есть на дашборде) |

Оранжевый/warning зарезервирован **только** за действиями клиента. Если Verifying покрасить в жёлтое/оранжевое — клиент будет дёргаться и звонить в банк («что со мной не так?»), хотя сценарий штатный. Это правило предложу в UX-гайд после одобрения Дениса.

### U2. Клиентская сторона (SDDS, зелёный акцент)

#### U2.1. Статус «Verifying» на дашборде

Новое 5-е значение `StepStatus = 'verifying'` (гэп §3-1). В карточке шага (`SP10Dashboard` → `StepCard`):

- **Кружок-бейдж номера:** сине-серый фон (в палитре дашборда: `rgba(100,116,139,…)`-семейство), иконка-оверлей **⟳** — единственный новый глиф в ряду ✓ ● ○ ⚠.
- **Лейбл статуса:** «Проверяем» / “Verifying” — тем же сине-серым, **не** bold (bold у нас сигнал requires_action).
- **Note под лейблом** (показываем для verifying так же, как сейчас для requires_action): одна строка «что проверяем + от вас ничего не нужно» (текст — U2.4).
- Карточка шага **остаётся кликабельной**: внутри шаг открывается в read-only со спокойной плашкой «Данные на проверке» — клиент может посмотреть, что он отправил, но не редактировать поле повторно, пока идёт проверка (повторная правка = новая задача, в демо не нужна).
- **В блок «Что требуется от вас» verifying НЕ попадает** — этот блок только для requires_action. Никаких баннеров сверху: проверка — фоновый процесс, его место в списке этапов.
- Прогресс заявки в шапке: статус заявки остаётся «In Progress», ничего глобально не перекрашиваем (Д-2: считаем Verifying статусом **уровня шага**; уровень заявки не трогаем).

#### U2.2. Карточка «Требует действия: загрузите документ» (такт 6)

Живёт там, где уже живёт requires_action: **блок «Что требуется от вас» вверху дашборда** (`ActionBlock`) + оранжевая подсветка шага в списке. Внимание привлекает позицией (первый контентный блок) и единственным оранжевым пятном на экране — этого достаточно, доп. анимации не нужны.

Состав карточки (`ActionItem`, расширенный):

1. **Заголовок — что сделать, конкретно:** «Загрузите документ: Utility Bill» (имя документа — термином, не переводим).
2. **Требование к документу** отдельной строкой: «не старше 3 месяцев» — то, из-за чего чаще всего заворачивают, нельзя прятать в комментарий.
3. **Зачем** (одна строка, без слова DVU): «Нужен, чтобы подтвердить изменённый адрес регистрации».
4. **Комментарий банка** — цитатой, визуально отделён (фон/кавычки): текст офицера дословно `(Д-1: допускаем — видит дословно; офицеру в форме это подсвечиваем, см. U3.2)`.
5. **CTA «Загрузить документ»** (accent, справа — по гайду).

CTA ведёт **не на экран шага, а на отдельный лёгкий экран «Запрос документа»** (гэп §3-3): заголовок = что загрузить, требование, комментарий банка, **одна** зона загрузки (переиспользуем proof-upload с «Подтверждения данных компании»), кнопка «Отправить». Один экран — одно действие, без скролла по всей форме компании в поисках нужного поля.

Канал-дублёр `(Д-5)`: для демо достаточно упоминания «Мы отправили письмо и SMS» в карточке; отдельный экран письма — опция, как было с приглашением на email.

#### U2.3. После загрузки (такт 7)

- Сразу после «Отправить»: **подтверждение на месте** (positive-Note на экране запроса): «Документ отправлен — снова проверяем», кнопка «Вернуться к заявке».
- На дашборде шаг возвращается в **verifying**, но note меняется: «Документ получен, проверяем» — клиент видит, что его действие засчитано, а не «опять то же самое».
- Блок «Что требуется от вас» исчезает (requires_action больше нет) — дашборд снова «спокойный».
- Такт 9: шаг → done, и если всё done — существующий блок «Счёт открыт» (про условность разморозки — Д-9, known gap).

### U3. Сторона DVU-офицера (оранжевый акцент, RM-экраны)

Контент RM — **English-first** (гэп §3-5); ниже даю EN, в скобках RU для обсуждения.

#### U3.1. Карточка задачи (RM-02): блок «Data change» для сценария правки

Сейчас RM-02 показывает статичные «Компания/Подписанты/Документы» (и это Private Limited — гэп §3-8: для SP заменить на PAN/GSTIN/Udyam, один владелец). Для кейса правки добавляю **первым блоком левой колонки** — суть задачи, а не справку:

- **Блок “Data change” (Изменение данных):** одна строка на изменённое поле, две колонки значений:
  - “Registry (Probe42)” — старое значение, серым/зачёркнутым;
  - “Submitted by client” — новое значение, bold;
  - бейдж “changed” на строке. Если полей несколько — список строк, неизменённые поля не показываем (офицеру нужен diff, не дамп).
- **Proof-превью** внутри того же блока: чип файла (имя, «uploaded by client · дата»), клик → превью (для прототипа — заглушка-картинка документа). Proof стоит **рядом со своим полем**, не в общем списке документов — офицер сверяет пару «новое значение ↔ документ» одним взглядом.
- Существующие документы заявки остаются ниже в блоке “Documents” как контекст.
- Правая колонка — без изменений по структуре: “Decision” с «Confirm validity» (accent) / «Request documents» (secondary). Кнопки уже есть.

#### U3.2. Форма запроса документа (раскрывается по «Request documents»)

**Inline-раскрытие в правой колонке** (не модалка — офицер должен видеть diff и proof, пока пишет комментарий):

1. **Select “Document type”** — справочник адрес-пруфов Sole Prop `(Д-10: TABLE A/B из 1.4)`: Utility Bill (≤3 months), Bank Statement, Registered Rent Agreement, Property Tax Receipt…
2. **Авто-подсказка требования** под селектом (из справочника): “Must be issued within the last 3 months”.
3. **TextArea “Comment to client”** с хелпером-предупреждением: **“The client will see this text verbatim”** («Клиент увидит этот текст дословно») `(Д-1)` — дисциплинирует тон.
4. Кнопка **“Send request”** (accent) + “Cancel”.

После отправки — существующий warning-Note «Documents requested», задача → `DocumentsRequested` (статус уже в модели).

#### U3.3. Состояние «ждём клиента»

- **В карточке (RM-02):** вместо панели Decision — панель **“Waiting for client”**: какой документ запрошен, требование, отправленный комментарий, timestamp “Requested 10 Jun, 14:32 · Email + SMS sent”, ниже — цепочка follow-up как пассивный таймлайн: “T+1 — email + SMS reminder · T+2 — call from officer” `(Д-3)` и кнопка “Contact client” (плейсхолдер, гэп §3-6). Кнопки решения скрыты — решать пока нечего.
- **В очереди (RM-01):** чип проверки уже умеет «запрос данных» (`DocumentsRequested`) — оставляем, тон neutral: для офицера это «не мой ход».
- **Когда клиент догрузил (такт 8):** в очереди чип получает **accent-тон и лейбл “Client replied” («ответ клиента»)** — единственный сигнал «снова твой ход», виден без открытия карточки. В карточке: в блоке Data change под полем появляется второй файл-чип с бейджем **“New”** + timestamp, панель Decision возвращается. Офицер жмёт “Confirm validity” → existing positive-Note, задача Resolved → у клиента done.

### U4. Демо-сценарий для показа (как прокликивает Денис)

Подготовка: **два окна браузера рядом** — слева «Клиент» (`/v2/dashboard`, зелёный), справа «Банк» (`/rm/queue`, оранжевый). Контраст акцентов сам объясняет смену роли, проговаривать «теперь я банк» почти не нужно.

| # | Окно | Действие | Что видит аудитория |
|---|---|---|---|
| 1 | Клиент | Открыть «Подтверждение данных компании» → Edit → поправить адрес → приложить proof → Continue | Нотификация «данные отправлены на проверку» |
| 2 | Клиент | Вернуться на дашборд | Шаг «Подтверждение данных компании» = **Проверяем ⟳**, спокойный; остальные шаги доступны (Д-7) |
| 3 | Банк | Очередь: заявка Aarav Sharma, чип «OBO · данные — изменено поле: registered address» | Правка клиента долетела в банк |
| 4 | Банк | Открыть задачу | Diff «Registry ↔ Client» + proof-превью |
| 5 | Банк | “Request documents” → выбрать Utility Bill (≤3 mo) → комментарий → “Send request” | Форма запроса; предупреждение «клиент увидит дословно» |
| 6 | Клиент | Обновить дашборд | Оранжевый блок «Что требуется от вас»: Utility Bill + комментарий банка |
| 7 | Клиент | «Загрузить документ» → экран запроса → загрузить → «Отправить» | «Документ отправлен — снова проверяем», шаг снова Verifying |
| 8 | Банк | Очередь: чип “Client replied” → карточка → файл “New” → “Confirm validity” | Цикл закрыт со стороны банка |
| 9 | Клиент | Обновить дашборд | Шаг **Done**, «Адрес подтверждён»; если все done — «Счёт открыт» |

Кульминация показа — такты 5→6: «офицер нажал — у клиента появилось». Это и есть BR-17, на нём стоит задержаться.

### U6. Открытые UX-вопросы (поверх §4 Бори)

- **U-а.** Read-only шага при verifying (U2.1) — моё допущение; альтернатива — разрешить повторную правку (порождает вторую задачу). Для демо read-only проще и честнее. → решить с Денисом.
- **U-б.** Появление запроса у клиента в демо — по refresh окна. «Живой» push/тост без перезагрузки эффектнее, но требует общий state (гэп §3-2) — решит инженерия.
- **U-в.** После одобрения концепта добавлю в UX-гайд правило U1 (Verifying ≠ warning) и правило «один запрос = один экран-действие».
