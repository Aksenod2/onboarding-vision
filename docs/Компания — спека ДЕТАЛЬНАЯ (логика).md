## 0. Модель данных и роли — общий контракт (читать первым)

### 0.1 Мультироль
Роль участника = **набор** (`roles: Role[]`), не одно значение. Одно лицо несёт несколько ролей одновременно.

```
Role = 'CustomerRepresentative' | 'Director' | 'AuthorizedSignatory' | 'UBO'
```

| Роль | Что даёт в потоке |
|---|---|
| `CustomerRepresentative` | Заполняет фазу A, рассылает ссылки, владеет дашбордом-монитором. Сам по себе НЕ обязывает проходить фазу B. |
| `Director` | Подписант BR. Проходит фазу B (Aadhaar+видео+DSC). Для Company минимум 1 (BRD §6: «≥2 директора», но решение Дениса — число не ограничиваем, см. 0.4). |
| `AuthorizedSignatory` (AS) | Распоряжается счётом; по BRD = один из директоров (happy). Проходит фазу B. |
| `UBO` | Бенефициар >10%. В happy-скоупе отдельной сессии не проходит, если не совмещён с Director/AS (см. открытый вопрос UBO). |

**Правило «кто проходит фазу B»:** лицо проходит персональную сессию ⟺ в его `roles` есть `Director` ИЛИ `AuthorizedSignatory`. `CustomerRepresentative` без этих ролей фазу B не проходит. Если у Customer Representative в наборе есть Director/AS — он и заполняет фазу A, и проходит свою фазу B.

### 0.4 Решения Дениса 2026-06-16 (фиксированные, не пересматривать без обсуждения)
1. **Число подписантов не ограничиваем.** Company = 1 и более (произвольный список). Sole Proprietor — всегда один.
2. **Загрузка BR = автозаполнение, не подпись.** В happy-скоупе загрузки нет — генерируем шаблон банка.
3. **DSC-подписание — ВНУТРИ портала** (целевое видение). По образцу OTP-подписания SP; eMudhra — заглушка.
4. **Видео проходят ВСЕ подписанты** (дефолт до комментариев команды). BRD §6 говорит «только AS, подписавший BR» — сузим, если команда подтвердит.
5. **Форма BR-вопросов = BRD §6 пп.1-4,7.** Пункты 5-6 (governance, место собрания) опускаем.

## A1 — Доступ к реестрам + PAN компании

| | |
|---|---|
| **Цель** | Получить PAN юрлица, дать согласие на доступ к реестрам, подтянуть золотую запись компании из Probe42. |
| **Входные данные** | Сессия заполнителя (email, phone из регистрации). Ничего о компании ещё не известно. |
| **Что собираем/показываем** | Ввод: **PAN компании** (ручной). Согласие `Registry Access` (чекбокс). После ввода система тянет из Probe42: `legalName`, `cin`, `registeredAddress`, `incorporationDate`, `companyStatus`, `directors[]` (список с Designation), shareholding. |
| **Логика и правила** | • Валидация PAN: 10 знаков, **4-й символ = `C`** (Company), в отличие от SP где 4-й = `P`. ⚠️ см. открытый вопрос ПАН-1.<br>• Согласие `Registry Access` обязательно до запроса в реестр; фиксируется `timestamp` (IST).<br>• Probe-ответ кладётся в `company.*` (source=`registry`) + наполняет внутренний список директоров — он станет источником дропдауна в A2.<br>• Фоновый скрининг (PAN/CKYC/Probe42/Sanctions) — happy = всё Pass, клиенту не показываем результаты (как у SP). |
| **Переход** | PAN валиден + Probe вернул данные + согласие дано → A2 (анкета+BR). `company.currentScreen = 'CO-PAN'`. |
| **Данные mock** | `company.pan/legalName/cin/registeredAddress/incorporationDate/companyStatus`; `consents[Registry Access].status=given, timestamp`; внутренний `probeDirectors[]` для дропдауна A2. |
| **Открытые вопросы** | **ПАН-1:** подтвердить, что для прототипа валидируем 4-й символ PAN = `C`. BRD §1.2 даёт пример PAN компании `RKRCD8099B` (4-й = `C`) — согласуется, но явного правила «4-й символ» для company в BRD нет. |

## A2 — Бизнес-анкета + блок BR-вопросов

Это переиспользованная BNQ (`SP07Bnq`) **плюс** новый блок BR-вопросов. Логика BNQ — как у SP (риск-категоризация, источники Probe/ручной), здесь не дублируем; описываем новый блок.

| | |
|---|---|
| **Цель** | Собрать состав подписантов BR (директора + AS), их контакты, и сгенерировать Board Resolution по шаблону банка. |
| **Входные данные** | `company.*` из A1; внутренний `probeDirectors[]` (имя + Designation из Probe). |
| **Что собираем/показываем** | По BRD §6 пп.1-4,7 (решение 5):<br>**п.1** «Свой BR или шаблон банка?» → happy = **шаблон банка** (No-ветка = DVU, out of scope).<br>**п.2** PAN компании → уже введён в A1, имя/адрес авто (только показ).<br>**п.3** «Выбрать ≥2 директоров-подписантов» → **дропдаун из Probe** (`probeDirectors[]`: имя + Designation). На каждого выбранного — ручной ввод `email`, `phone`.<br>**п.4** «Назначить Authorized Signatory» → ручной ввод (`fullName`, `email`, `phone`; PAN — ручной, если AS не из Probe).<br>**п.7** «Подтвердить сгенерированный BR» → кнопка, авто-таймстамп. |
| **Логика и правила** | • **Источник директоров (п.3):** дропдаун заполнен из Probe-списка. Имя + Designation приходят из реестра (source=`registry`, не редактируются), `email`/`phone` — ручной ввод (`manual`). PAN директора — авто из Probe.<br>• **Назначение AS (п.4):** ручной ввод. Если AS = один из уже выбранных директоров (happy по кейсу 2) — к его `roles` добавляется `AuthorizedSignatory` (не создаём дубль-лицо). Если AS — отдельное лицо (как Amit в демо) — создаётся `Signatory` с `roles:[AuthorizedSignatory]`, `panSource='manual'`.<br>• Каждое выбранное/введённое лицо → запись в `signatories[]` с накопленным `roles`.<br>• **Генерация BR (п.7):** система собирает `BoardResolution` из `company.*` (Name/CIN/Address авто) + `signatoryTable = signatories`, ставит `confirmed=true`, `confirmedAt` (IST-таймстамп). Двойного подписания нет: это подтверждение состава, финальная подпись — DSC в фазе B.<br>• **Валидация:** минимум подписантов — по решению «не ограничиваем», но для соответствия BRD §6 happy-демо держит 2 директора + AS. Имя+Designation директоров «валидируются против Probe42» (в прототипе — заглушка Pass).<br>• Пункты 5-6 BRD (governance, place of meeting) — НЕ показываем. |
| **Переход** | Все обязательные поля заполнены + BR подтверждён → A3. `company.currentScreen='CO-BNQ-BR'`; `br.confirmed=true`. |
| **Данные mock** | `signatories[]` создаются здесь: `{id, fullName, roles, pan, panSource, email, phone}`; `br.{template:'bank', signatoryTable, confirmed, confirmedAt}`; BNQ-ответы — как у SP в `bnq[]`. |
| **Открытые вопросы** | **В-6:** Probe-проверка директоров (Designation, Flag=Deactivated) — фоновая или показываем клиенту флаги? Happy = фон, Pass.<br>**UBO:** Amit несёт `UBO` — в кейсе 3 есть поле «details of UBOs who signed the BR». В happy-скоупе UBO как отдельную сущность не собираем; роль фиксируем на лице, но отдельного UBO-ввода нет. Подтвердить, что для прототипа этого достаточно. |

# ДАШБОРД КОМПАНИИ — статус-монитор подписантов

| | |
|---|---|
| **Цель** | Customer Representative видит прогресс по каждому подписанту и понимает, когда откроется счёт. Точка входа в персональные сессии (демо: «Войти как»). |
| **Входные данные** | `signatories[]` с их `currentStep`/`status`; `company.status`. |
| **Что показываем** | Заголовок заявки + статус компании. По КАЖДОМУ подписанту: имя, роли (чипами — ВСЕ роли лица), его текущий шаг, кнопка «Войти как [имя]» (для лиц со статусом ≠ done). Поясняющая строка «Счёт откроется, когда все подписанты пройдут идентификацию и подпишут». |
| **Логика и правила** | • **Статус подписанта = его `currentStep`** мини-сессии: `waiting` → `consents` → `aadhaar` → `vkyc` → `dsc-sign` → `done`. Маппинг на отображаемый ярлык (Ожидает / Согласия / Aadhaar / Видео / Подписание / Готово).<br>• Роли показываем все (чипы): напр. Rajesh = `Director · AS · Заполнитель`.<br>• **«Войти как [имя]»** — имитация перехода по ссылке из письма → открывает персональную сессию (фаза B) этого лица на его `currentStep` (resume, не с нуля).<br>• **Условие открытия счёта:** счёт открыт ⟺ ВСЕ подписанты (все с ролью Director/AS) имеют `status='done'` (прошли Aadhaar+видео+DSC). Тогда `company.status='Completed'`, показываем блок «Счёт открыт» (как у SP).<br>• Дашборд — живой агрегат, не статичный список: пересчитывается по `signatories[].status`. |
| **Переход** | • «Войти как [имя]» → фаза B этого подписанта (его текущий шаг).<br>• Возврат из фазы B → дашборд с обновлённым `currentStep`/`status` этого лица.<br>• Все `done` → блок «Счёт открыт» / финальный экран Internet Bank. |
| **Данные mock** | Чтение `signatories[].{fullName, roles, currentStep, status}`; запись `company.status='Completed'` при всех done. |
| **Открытые вопросы** | **Дашборд-роль:** дашборд-монитор технически принадлежит Customer Representative. Если у компании несколько Customer Representative — не в скоупе (демо = один). |

## B4 — Подписание по DSC

| | |
|---|---|
| **Цель** | Подписант ставит цифровую подпись (DSC) под Board Resolution + Application + Declaration. |
| **Входные данные** | `signatory` на шаге `dsc-sign`; сгенерированный `br`; пройденные Aadhaar+видео. |
| **Что показываем** | Документы к подписанию (BR + Application + Declaration) + DSC-подписание. Переиспуем `SPSign`, но **DSC вместо OTP**. |
| **Логика и правила** | • Подписание по **DSC** (Digital Signature Certificate), НЕ mobile OTP как у SP (различие SP↔Company).<br>• DSC — **внутри портала** (решение 3, целевое видение); eMudhra/реальная интеграция — заглушка.<br>• Фиксируется `signature.{signed:true, method:'DSC', timestamp}`.<br>• Подписывает каждый подписант свою часть; BR считается полностью подписанным, когда все требуемые подписанты подписали. |
| **Переход** | Подписано → `signatory.currentStep='done'`, `status='done'` → возврат на дашборд компании, статус этого лица обновлён.<br>Если это последний из всех подписантов → пересчёт `company.status='Completed'` → блок «Счёт открыт». |
| **Данные mock** | `signatory.signature={signed:true, method:'DSC', timestamp}`; `currentStep='done'`, `status='done'`. |
| **Открытые вопросы** | **DSC-1:** DSC внутри портала (iframe) vs внешняя ссылка подрядчика — где проходит и что показываем как known gap. Прототип = внутри портала (заглушка).<br>**В-10:** в кейсе 3 собственный BR грузится уже DSC-подписанным, но подписанты снова подписывают DSC — двойное подписание? (касается только DVU-ветки, в happy не возникает). |

# Сводная логика статусов и условие открытия счёта

| Уровень | Поле | Значения | Кто меняет |
|---|---|---|---|
| Компания | `company.status` | Draft → InProgress (после A4) → Completed (все подписанты done) | система |
| Компания | `company.currentScreen` | CO-PAN → CO-BNQ-BR → CO-CONFIRM → CO-DASHBOARD | фаза A |
| Подписант | `signatory.currentStep` | waiting → consents → aadhaar → vkyc → dsc-sign → done | фаза B (его шаги) |
| Подписант | `signatory.status` | waiting → in_progress → done | фаза B |

**Условие открытия счёта (главное правило):**
`company.status = 'Completed'` ⟺ для всех `s ∈ signatories` где `s.roles ∩ {Director, AuthorizedSignatory} ≠ ∅` выполнено `s.status === 'done'`.
В демо: счёт открыт, когда Rajesh, Priya и Amit все `done`.
