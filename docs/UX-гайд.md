# UX-гайд — онбординг юрлиц (клиентская сторона, SDDS Serv)

> Живой документ. Растёт по одному правилу за раз — по мере UX-решений, а не пишется заранее.
> Ведёт субагент `ux-designer`. Каждое правило: что → почему → когда зафиксировано.
> Рамка: боевой прототип на палитре SDDS Serv, зелёный акцент клиента. Без кастомной эстетики вне whitelist.

## Согласия (consent)

- **Согласие — всегда внизу экрана, перед основной кнопкой.** Сначала пользователь видит суть (поля, данные), согласие даёт последним осознанным действием. Не размещать галочку-согласие над контентом. — 2026-06-09
- **Смежные «согласие + ввод» объединяем в один экран.** Если согласие нужно непосредственно для следующего ввода (напр. доступ к реестрам → ввод PAN), это один шаг, а не два: галочка внизу, поле сверху. — 2026-06-09
- **Два смежных «подтверждающих» экрана должны визуально и текстово отличаться по роли, иначе читаются как «подтверди дважды».** Экран-РЕДАКТИРОВАНИЕ (можно менять данные) и экран-ПРИСЯГА (юридическое подтверждение достоверности перед точкой невозврата) — это разные моменты. На экране-присяге НЕ повторяем полный дамп тех же атрибутов: данные сворачиваем в напоминание (свёрнутая сводка / ссылка «посмотреть»), а главным визуальным объектом делаем само согласие + предупреждение о необратимости. Почему: одинаковый паттерн (заголовок про проверку + список тех же полей + кнопка подтверждения) подряд воспринимается как дублирование, пользователь перестаёт читать и кликает на автомате — теряется осознанность согласия. — 2026-06-09

## Сетка / ширина

- **Единая ширина контентной области для ВСЕХ шагов — 880px (решение Дениса 2026-06-09).** Раньше ширина гуляла 520–720 по экранам и «прыгала» на переходах; затем пробовали два токена (640/880), но Денис выбрал одну ширину везде = ширина бизнес-анкеты (880). Механизм: одна константа `CONTENT_MAX_WIDTH` в `ScreenV2`, без пропов ширины в экранах. Почему: одинаковая ширина на всех шагах линейного потока = ноль «прыжков», на десктопе не узко. — 2026-06-09 (заменяет прежнее правило про две ширины)

## Кнопки

- **Основная кнопка («Продолжить» и аналоги) — всегда у правого края (решение Дениса 2026-06-09).** Обёртка кнопки — `display:flex; justify-content:flex-end`; в парах с «Назад» — «Назад» слева, основная справа. Применять на всех шагах потока единообразно. Почему: правый край — устойчивое место «движения вперёд», единообразие убирает разнобой расположения CTA между экранами. — 2026-06-09

## Вертикальный ритм / отступы

- **Между крупными блоками экрана (карточка ↔ секция ↔ карточка) — единый зазор 1.5rem (24px), внутри карточки — 1.25rem.** Многоблочные экраны (VCIP, дашборд) собираем через один контейнер-`Stack` с `gap`, а не через `margin-bottom` на каждом блоке — так зазор не «плавает» от экрана к экрану. Почему: контейнер `ScreenV2.Content` без своего gap, и без обёртки блоки слипаются; одно значение в одном месте = ритм предсказуем и правится централизованно. Замеченный дрейф (дашборд: 1.75rem, VCIP: 1.5rem) свести к 24px. — 2026-06-09
- **Подсказку-аффорданс под прогресс-баром показываем только на 1–2 шаге онбординга, дальше скрываем.** Постоянная подпись на каждом из 7 шагов = визуальный шум: одноразовое сообщение «как тут навигировать» прочитывается один раз, после чего лишь спорит за внимание с заголовком шага и кнопкой. Сами сегменты прогресса остаются кликабельными (ховер + курсор) — аффорданс никуда не девается, исчезает только текстовое дублирование. — 2026-06-09
