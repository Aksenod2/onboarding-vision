# CRM — архитектура объединённого окна «отдел работы с клиентами» (spike)

Автор: Лев (solution-архитектор CRM-трека). Дата: 2026-06-26.
Назначение: опорная архитектура под единое окно CRM+DVU (advisory, для UX-проекта Ульяны и spike Феди). Код не пишу — даю доменную логику и структуру модулей Косте/Феде на вход.
Опирается на: `Решение — объединение CRM и DVU (мнения команды) 2026-06-26.md`, `Карта стыковок онбординг × CRM × DVU.md`, `CRM — спецификация и фичи.md`, текущий код `app/src/crm/*`.

---

## TL;DR

**Один стор-ядро (client-360), два домена-надстройки, RBAC переключает рабочий стол.** CompanyProfile и его воронка живут в едином синглтон-сторе (он уже есть — `crmApi.ts`). DVU не заводит свой стор: DVU-Task ссылается на профиль по `profileId` и хранится в том же ядре. Действие DVU мутирует ядро → CRM-домен читает то же место → обновление видно мгновенно, дублей primary-полей нет. Внешний `dvu.result` из events.ts превращается во **внутренний переход** (DVU-домен пишет в ядро напрямую, без сериализации через шину).

**Структурная развилка — закрыта: для spike DVU = подпапка `crm/dvu/` (позиция Кости), НЕ параллельный модуль `app/src/dvu/`.** Обоснование ниже (раздел 6). Меняю свою прежнюю позицию из карты стыковок — для spike она создаёт лишний барьер вокруг общего стора.

3 тезиса Дениса обеспечены:
1. **Консистентность** — один стор, FK по `profileId`, DVU-результат = внутренняя мутация (раздел 4).
2. **RBAC** — роль → права → workspace-пресет зон; four-eyes заложен полем `assignedTo`/`approvedBy` даже при заглушке (раздел 5).
3. **Масштабируемость** — добавить Compliance = +пресет +права, ядро не трогаем (раздел 7).

**Вне scope spike:** прод-готовность DVU (16 сценариев BRD), реальная очередь/SLA, аудит-трейл регулятора, настоящий four-eyes-воркфлоу. Spike доказывает 3 тезиса на 1-2 fixture-сценариях.

## 2. Слои (ядро + домены + пресеты)

| Слой | Что | Где в коде (spike) | Кто трогает при росте |
|---|---|---|---|
| **Ядро (client-360)** | стор: `profiles`, `leads/offers/deals/applications/history`, **+`dvuTasks`**; репо-функции | `crm/mock/crmApi.ts`, `crm/mock/seed.ts`, `crm/types/domain.ts` | НЕ трогаем при добавлении ролей/доменов |
| **Адаптер** | единственный вход внешних событий онбординга | `crm/adapter/ingest.ts`, `crm/types/events.ts` | трогаем только при новом внешнем контракте |

## 3. Доменная модель: где живёт что

### 3.1 CompanyProfile живёт в ядре (единственный экземпляр)

CompanyProfile — материнская сущность (Д1). Один экземпляр на PAN, в `state.profiles[]`. Оба домена читают его **отсюда же**, копий нет. PAN — ключ find-or-create (Д2).

### 3.2 DVU-Task — новая сущность ядра, ссылается на профиль по profileId

Сейчас DVU «нигде не живёт целиком» (тех-долг, отметил Костя). Вводим сущность в `types/domain.ts`:

```
DvuTask {
  id: string
  profileId: string          // FK на CompanyProfile (НЕ дубль профиля — ссылка)
  onboardingCaseId?: string  // трассировка к заявке онбординга
  kind: 'probe-fail' | 'crilc' | 'ofac' | 'ckyc' | 'manual-edit'
       | 'doc-request' | 'vkyc-fail' | 'abandoned'   // источники из карты стыковок
  status: 'open' | 'in-review' | 'pending-approval' | 'resolved' | 'rejected'
  outcome?: 'approved' | 'rejected' | 'more-info'
  assignedTo?: string        // DVU-офицер (исполнитель)  ── four-eyes
  approvedBy?: string        // финальный аппрувер ≠ assignedTo ── four-eyes
  createdAt / updatedAt: IsoTimestamp
  log: Array<{ at, by, action, comment? }>   // лог решений (Костя: «лог решений»)
}
```

Хранение в сторе — по тому же паттерну, что воронка: `dvuTasks: Record<string, DvuTask[]>` (profileId → задачи). Никакого второго стора.

### 3.3 Связь DVU-Task ↔ CompanyProfile ↔ Application

```
CompanyProfile (1) ──< (N) DvuTask        связь по profileId
       │                      │
       └──< (N) Application ──┘ (опц.)     DvuTask.onboardingCaseId → Application.onboardingCaseId
```

DVU-офицер в очереди видит `DvuTask`; чтобы открыть контекст компании — переходит на тот же `CompanyProfile` по `profileId`. Менеджер на профиле видит связанные задачи **через тот же FK**. Один источник, две точки входа.

## 4. Механизм консистентности (тезис 1 — главный)

### Решение: ОДИН стор, НЕ два стора + контракт

**Почему один.** Два стора + контракт между ними = ровно тот сценарий, который команда отвергла (микросервисы внутри окна): дубль primary-полей, две «БД», хореография, и баг «DVU подтвердил — а CRM ещё не знает». Денис хочет «мгновенно видит обновление» — это даёт только общая память. Стор `crmApi.ts` уже устроен как client-360 (профиль + коллекции по profileId); DVU подключается дописыванием `dvuTasks` в тот же синглтон.

### Как мутация DVU отражается в CRM — пошагово

```
1. DVU-офицер в Queue-WS жмёт «Approve» по DvuTask T (profileId=P).
2. DVU-домен зовёт репо ядра:  resolveDvuTask(P, T.id, 'approved', by=officer)
       └─ мутирует state.dvuTasks[P]  (status: resolved, outcome: approved, approvedBy)
       └─ пишет в state.history[P]    (внутренний переход, см. ниже)
       └─ опц. патчит Application     (patchApplicationByCase — функция УЖЕ есть)
3. CRM-домен (Sales-WS) при следующем чтении getFunnel(P)/getDvuTasks(P)
   читает ТО ЖЕ state.* → видит resolved.  Дубля нет, рассинхрона нет.
```

В spike «следующее чтение» = перезапрос после действия или общий re-render (Феде хватит простого refetch/инвалидации; реактивный pub-sub — прод-задача, не spike).

### dvu.result: из внешнего события → во внутренний переход

Сейчас `dvu.result` (events.ts) приходит **снаружи** через `ingest` — это был артефакт «DVU вне CRM». После объединения DVU **внутри** окна, значит:

| Было (DVU снаружи) | Стало (DVU внутри окна) |
|---|---|
| DVU-сервис → шина → `dvu.result` DTO → `ingest()` → история | DVU-домен → `resolveDvuTask()` репо ядра → мутация `dvuTasks` + `history` напрямую |
| сериализация, риск рассинхрона | один вызов в общий стор, атомарно |

**Закрывает OQ-9** (как и предсказано в решении команды): `dvu.result` перестаёт быть внешним контрактом, становится внутренним переходом. `ingest()` для `dvu.result` оставляем как back-compat shim на время spike (если onboarding ещё шлёт), но первичный путь — внутренний.

⚠️ **Развилка для Кости/Дениса.** Остаётся пограничный кейс: `dvuRequest` (обратный запрос документа DVU → онбординг, этап 7/11 карты). Это DVU пишет **в онбординг** (чужой домен за границей изоляции). Тут контракт через шину сохраняется (DVU-домен → событие → онбординг). То есть: DVU↔CRM внутренний переход; DVU→онбординг остаётся событием. Граница изоляции цела.

### Инвариант консистентности (что защищаем)

> Для любого PAN существует ровно один CompanyProfile, и любая DVU-задача/заявка/оффер ссылается на него по id, а не копирует его primary-поля. Мутация любого домена идёт через репо-функции ядра, читающие/пишущие один синглтон.

Это уже соблюдено в коде для воронки (`seed.ts`: «привязка по profileId — как FK»). DVU встаёт по тому же правилу.

## 5. RBAC-модель (тезис 2)

### Модель: роль → права → workspace-пресет

```
Role ──имеет──▶ Permission[] ──группируются в──▶ WorkspacePreset (набор зон+действий)
```

### Минимальная матрица для spike

| Роль | Workspace | Зоны (включено) | Ключевые действия | four-eyes |
|---|---|---|---|---|
| **Менеджер (RM)** | Sales-WS | профиль, воронка, история, AI, таски | create offer/deal/lead, link onboarding | — |
| **Оператор колл-центра** | Sales-WS (урезанный) | поиск, профиль (read), история, возобновление | search PAN, resume application, create profile | — |
| **DVU-офицер** | Queue-WS | очередь задач, карточка задачи, профиль (read) | take task, review, **resolve** (исполнитель) | **assignedTo** |
| **DVU-аппрувер / Compliance** *(задел)* | Queue-WS | очередь, карточка задачи | **approve/reject** (финал) | **approvedBy ≠ assignedTo** |

Менеджер + оператор делят Sales-WS (их job-ы близки — pull/reactive, Соня), различаются урезанием прав. DVU — отдельный рабочий стол того же продукта.

### Как выразить в коде (Феде)

```
crm/rbac/roles.ts      — type Role = 'manager' | 'operator' | 'dvu-officer' | 'dvu-approver'
crm/rbac/presets.ts    — Record<Role, WorkspacePreset>
                         WorkspacePreset = { workspace: 'sales'|'queue', zones: Zone[], can: Permission[] }
crm/rbac/RoleContext   — React-контекст: текущая роль (в spike — переключатель в DemoNav)
```

- **Роль в контексте**, не в пропсах: `useRole()` / `useCan('resolve-dvu')`.
- Гейтинг зон: `CrmRoutes` по `preset.workspace` монтирует Sales-WS или Queue-WS.
- Гейтинг действий: кнопка обёрнута `can('resolve-dvu')` → рендерится/дизейблится.

### Four-eyes — заложить даже при заглушке

Минимально, чтобы прод-воркфлоу встал без переделки ядра:
1. **Поля уже есть** в DvuTask: `assignedTo` (исполнитель) + `approvedBy` (аппрувер).
2. **Инвариант** (в spike — простая проверка, не полноценный воркфлоу):
   > `resolveDvuTask` не даёт одному пользователю быть и assignedTo, и approvedBy по одной задаче.
3. В spike это **одна строка-гард** (`if (assignedTo === approver) throw/deny`) + статус `pending-approval` между review и resolved. Полный воркфлоу (роутинг на второго офицера, SLA) — прод (вне scope).

Это закрывает «единственное серьёзное возражение» команды (segregation of duties) на уровне модели, не откладывая на потом.

## 6. Структурная развилка — РЕКОМЕНДАЦИЯ для spike

### Вердикт: DVU = подпапка `crm/dvu/` (НЕ параллельный `app/src/dvu/`)

Меняю свою прежнюю позицию (параллельный модуль) **для фазы spike**. Обоснование — приземлённо:

| Критерий | `crm/dvu/` (подпапка) ✅ для spike | `app/src/dvu/` (параллельный) |
|---|---|---|
| Доступ к ядру client-360 | прямой, оба домена внутри одного барьера изоляции | нужен второй барьер или общий extracted-core → лишняя работа на spike |
| Граница изоляции | **одна** (онбординг \| crm+dvu), `check-crm-isolation.mjs` почти не меняется | две границы, гард усложняется |
| Консистентность (тезис 1) | тривиальна — один стор в одном модуле | требует расшаренного стора между модулями = риск дубля |
| Совпадение с консенсусом | да (Костя: зонтичный модуль с подмодулями `profile/`+`dvu/`) | расходится |
| Скорость для Феди | минимальная: +`crm/dvu/`, +`crm/rbac/`, +`dvuTasks` в стор | выше: вынос ядра, два theme-mount, два гарда |

**Когда параллельный `dvu/` оправдан (НЕ сейчас):** если DVU дорастёт до отдельного деплоя/команды/БД (прод-фаза, 16 сценариев BRD). Тогда `crm/dvu/` рефакторится в отдельный модуль с extracted client-360 как shared-core. Для spike это преждевременно.

**Нейминг (развилка Кости).** Зонтичную папку можно переименовать `crm/` → `workspace/`, раз внутри уже два домена. **Для spike — НЕ переименовываю** (лишний diff, ломает импорты/гард/доки). Отдельным шагом после spike, если приживётся. Помечаю для Кости.

### Граница изоляции к онбордингу — БЕЗ изменений

`crm/**` (включая `crm/dvu/`, `crm/rbac/`) НЕ импортит онбординг (`screens/v2*`, `mock/v2*`, `company*`). Вход извне — только `crm/index.ts` barrel + `adapter/ingest`. `check-crm-isolation.mjs` стережёт ту же одну границу — Косте править минимально (добавить `crm/dvu`, `crm/rbac` в перечень охраняемых, если он по подпапкам).

## 7. Масштабируемость (тезис 3): добавить Compliance

Сценарий: появляется роль **Compliance** (отдельный контур, Боря: «Compliance — отдельный контур; Concurrent Audit — независимая третья рука»).

```
ЧТО ДОБАВЛЯЕМ:
  + crm/rbac/roles.ts        : Role | 'compliance'
  + crm/rbac/presets.ts      : compliance → { workspace:'queue', zones:[...], can:['review-aml', ...] }
  (опц.) + DvuTask.kind      : 'aml-escalation'   — если нужен новый тип задачи

ЧТО НЕ ТРОГАЕМ:
  ✗ ядро client-360 (стор, profiles, FK profileId)
  ✗ адаптер / контракт событий
  ✗ другие пресеты (manager/operator/dvu)
  ✗ граница изоляции
```

То же для нового **домена** (напр. отдельный case-management): добавляем `crm/<newdomain>/`, его сущность ссылается на `profileId`, +пресет. Ядро стабильно.

**Это и есть «ядро + домены + пресеты» как расширяемая ось:** рост идёт по краям (пресеты, подпапки доменов, опц. новые kind/события), центр (client-360) не пересматривается. Сравнение с рынком (Марк): Salesforce FSC — разные консоли (Sales/Service) над общим Account 360; новые консоли добавляются, Account 360 — стабильное ядро. Наша схема — тот же приём.
